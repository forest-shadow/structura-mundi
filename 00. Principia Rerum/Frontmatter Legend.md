# Frontmatter Legend

## Назначение

Этот документ объясняет, что означает каждое поле `frontmatter`, как его заполнять и каких ошибок избегать.

Этот документ не является:

- полным словарём допустимых значений;
- политикой тегов;
- каталогом шаблонов.

Для этого нужно использовать:

- [[Controlled Vocabulary]]
- [[Tag Policy]]
- [[Templates/Article Template|Agent Article Template]]
- [[Templates/Overview Template|Agent Overview Template]]
- [[00. Obsidian Praxis/__00. Obsidian Praxis|00. Obsidian Praxis]]

## Минимальный стандарт frontmatter

```yaml
---
aliases: []
note_type:
area:
domain:
section:
parent:
status:
related: []
tags: []
---
```

## Обязательность полей

- обязательно: `note_type`, `area`, `status`
- обязательно в обычном случае: `domain`, `section`
- условно обязательно: `parent`
- обычно необязательно: `aliases`, `related`, `tags`

`domain` может быть пустым только у `area-level root overview`, которая описывает всю `area` целиком.

`section` может быть пустым только в двух случаях:

- у `area-level root overview`;
- у `domain-root overview`, которая описывает весь `domain`, а не локальный кластер внутри него.

`parent` заполняется, если заметка логически подчинена обзорной теме. Для корневой `overview`-заметки поле может быть пустым.

## `aliases`

Назначение:

- хранит альтернативные названия той же самой заметки.

Тип значения:

- список ссылкоподобных строк или обычных строковых названий.

Обязательность:

- необязательное поле.

Пример:

```yaml
aliases:
  - Online Transactional Processing
  - OLTP
```

Типичные ошибки:

- выносить alias в отдельную заметку вместо этого поля;
- добавлять в `aliases` не синонимы, а соседние понятия;
- использовать `aliases` для хранения редакторских пометок.

## `note_type`

Назначение:

- задаёт роль заметки в системе.

Тип значения:

- одиночная строка.

Обязательность:

- обязательное поле.

Пример:

```yaml
note_type: article
```

Типичные ошибки:

- пытаться кодировать через `note_type` зрелость заметки;
- использовать неутверждённые значения;
- дублировать тип заметки в имени файла.

Допустимые значения см. в [[Controlled Vocabulary]].

## `area`

Назначение:

- обозначает самую крупную область знания.

Тип значения:

- одиночная строка.

Обязательность:

- обязательное поле.

Пример:

```yaml
area: computer-science
```

Типичные ошибки:

- использовать здесь локальную тему вместо большой области;
- вводить новое `area` без необходимости;
- дублировать `area` в тегах.

Допустимые значения см. в [[Controlled Vocabulary]].

## `domain`

Назначение:

- задаёт главную дисциплинарную или профессиональную оптику заметки.

Тип значения:

- одиночная строка или пустое значение у `area-level root overview`.

Обязательность:

- обязательное поле, кроме `area-level root overview`.

Пример:

```yaml
domain: databases
```

Типичные ошибки:

- делать поле множественным;
- пытаться перечислить в нём все возможные перспективы темы;
- оставлять `domain` пустым вне `area-level root overview`;
- использовать `domain` как замену `section`.

Допустимые значения и правила расширения см. в [[Controlled Vocabulary]].

## `section`

Назначение:

- связывает заметки в локальный тематический кластер внутри `domain`.

Тип значения:

- одиночная строка или пустое значение в канонических исключениях.

Обязательность:

- обязательное поле, кроме `area-level root overview` и `domain-root overview`.

Пример:

```yaml
section: oltp
```

Типичные ошибки:

- дублировать `domain` в значении `section`;
- создавать новое `section` ради одной заметки;
- оставлять `section` пустым у вложенной заметки, которая уже принадлежит локальному кластеру;
- вводить служебные значения вроде `*-core` только ради заполнения поля;
- использовать `section` как скрытую замену `parent`.

Допустимые значения и правила расширения см. в [[Controlled Vocabulary]].

## `parent`

Назначение:

- указывает родительскую обзорную заметку, если такая есть.

Тип значения:

- одиночная wikilink-ссылка или пустое значение.

Обязательность:

- условно обязательное поле.

Пример:

```yaml
parent: "[[OLTP]]"
```

Типичные ошибки:

- оставлять поле пустым у заметки, которая явно должна входить в ветку;
- использовать `parent` для поперечных связей;
- указывать в `parent` не обзорную, а случайную соседнюю статью.

## `status`

* <b>Назначение:</b> фиксирует степень зрелости заметки.
* <b>Тип значения:</b> одиночная строка.
* <b>Обязательность:</b> обязательное поле.

Пример:

```yaml
status: draft
```

Типичные ошибки:

- использовать произвольные статусы;
- путать зрелость заметки с её типом;
- не обновлять статус после заметного изменения качества заметки.

Допустимые значения см. в [[Controlled Vocabulary]], а операционные критерии — в [[Note Lifecycle and Merge Policy]].

## `related`

Назначение:

- хранит поперечные содержательные связи между заметками.

Тип значения:

- список wikilink-ссылок.

Обязательность:

- необязательное, но очень полезное поле.

Пример:

```yaml
related:
  - "[[MVCC]]"
  - "[[WAL]]"
  - "[[Sharding]]"
```

Типичные ошибки:

- использовать `related` вместо `parent`;
- превращать `related` в список всех упоминаний подряд;
- оставлять только иерархические связи, забывая про поперечные.

## `tags`

Назначение:

- задаёт дополнительные вторичные аспекты, но не основную классификацию.

Тип значения:

- список строк.

Обязательность:

- необязательное поле.

Пример:

```yaml
tags:
  - performance
  - consistency
```

Типичные ошибки:

- дублировать через теги `area`, `domain`, `section` или `parent`;
- использовать теги как основную онтологию;
- добавлять теги, которые полезны только одной заметке.

Полные правила и допустимые рабочие теги см. в [[Tag Policy]].

## Как читать frontmatter целиком

Минимальный пример для `article`:

```yaml
---
aliases: []
note_type: article
area: computer-science
domain: databases
section: oltp
parent: "[[OLTP]]"
status: draft
related:
  - "[[MVCC]]"
  - "[[WAL]]"
tags:
  - transactions
  - consistency
---
```

Это означает:

- это обычная статья;
- она относится к `computer-science`;
- написана из `databases`-оптики;
- входит в `oltp`;
- подчинена обзорной статье `OLTP`;
- пока находится в статусе `draft`;
- имеет поперечные связи и вторичные аспекты.

Два канонических исключения:

`area-level root overview`:

```yaml
---
aliases: []
note_type: overview
area: computer-science
domain:
section:
parent:
status: draft
related: []
tags: []
---
```

Это означает:

- это корневая обзорная заметка всей `area`;
- у неё нет отдельного `domain`, потому что она описывает поле целиком;
- у неё нет `section`, потому что над ней ещё нет локального кластера.

`domain-root overview`:

```yaml
---
aliases: []
note_type: overview
area: computer-science
domain: algorithms
section:
parent: "[[Computer Science]]"
status: draft
related: []
tags: []
---
```

Это означает:

- это корневая обзорная заметка целого `domain`;
- `domain` уже задаёт нужную оптику;
- `section` пустое, потому что заметка описывает весь домен, а не его внутренний подраздел.

## Шаблоны

Готовые шаблоны вынесены в отдельные файлы:

- `Templates/` внутри `00. Principia Rerum` — agent-шаблоны для создания заметок агентом.
- `Templates/` внутри `00. Obsidian Praxis` — минимальные шаблоны для стандартного механизма шаблонов Obsidian.
- [[Templates/Article Template|Agent Article Template]]
- [[Templates/Overview Template|Agent Overview Template]]
- [[00. Obsidian Praxis/Templates/Article Template|Obsidian Article Template]]
- [[00. Obsidian Praxis/Templates/Overview Template|Obsidian Overview Template]]

Каноническое правило: в каждой из этих двух директорий должно быть только по два шаблона — `Article Template` и `Overview Template`.

Не следует создавать тематические, доменные или технологические варианты шаблонов. Если заметка про особую тему, различие должно жить в её содержании и `frontmatter`, а не в отдельном template-файле.
