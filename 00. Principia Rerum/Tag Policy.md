# Tag Policy

## Назначение

Этот документ задаёт правила использования тегов в базе знаний.

Главный принцип:

- теги нужны для вторичных поперечных срезов;
- теги не являются основной системой классификации.

## Статус словаря тегов

Словарь тегов в текущей системе является `контролируемо-открытым`.

Это означает:

- есть набор уже подтверждённых рабочих тегов;
- новые теги допустимы только по правилам этого документа;
- новые теги не должны появляться стихийно внутри отдельных заметок.

## Уже подтверждённые теги

На текущем этапе как минимум подтверждены:

- `performance`
- `consistency`
- `security`
- `observability`
- `transactions`
- `reliability`

Если один из этих тегов подходит по смыслу, агент должен переиспользовать его, а не изобретать новый вариант.

## Для чего теги используются

Теги разрешены для дополнительных аналитических перспектив, которые могут проходить через разные разделы.

Типичные примеры:

- `performance`
- `consistency`
- `security`
- `observability`
- `transactions`
- `reliability`

## Для чего теги не используются

Теги не должны заменять:

- `area`
- `domain`
- `section`
- `parent`
- каноническое имя заметки

Нельзя строить через теги основную онтологию базы.

Также теги не должны заменять:

- каноническое разбиение на `overview` и `article`;
- структуру веток через `parent`;
- разбиение больших тем на `section`.

## Хороший тег

Хороший тег:

- встречается в разных разделах;
- выражает поперечный аспект, а не место заметки в структуре;
- помогает фильтровать заметки по дополнительной перспективе;
- остаётся полезным даже если заметки лежат в разных буквах и разных `section`.

Примеры хороших тегов:

- `performance`
- `observability`
- `security`
- `consistency`
- `transactions`
- `reliability`

## Плохой тег

Плохой тег:

- повторяет уже существующую классификацию;
- дублирует `domain` или `section`;
- нужен только одной заметке;
- является почти тем же самым словом, что и имя файла или родительской темы.

Примеры плохих тегов:

- `databases`
- `oltp`
- `database/oltp`
- `load-balancing`

Если заметка уже имеет:

```yaml
domain: databases
section: oltp
```

то тег `databases` или `oltp` почти наверняка избыточен.

## Когда добавлять новый тег

Новый тег допустим, если:

- это действительно поперечный аналитический срез;
- он будет полезен не одной, а нескольким заметкам;
- его нельзя естественно выразить через `domain`, `section`, `parent` или `related`;
- он не является просто новым синонимом уже существующего тега.

### Когда новый тег не нужен

Новый тег не нужен, если:

- тема уже естественно выражается через `section`;
- тег описывает не поперечный аспект, а локальный раздел;
- предполагается, что тег будет жить только в одной ветке;
- новый тег отличается от существующего только стилистически.

## Когда вместо тега использовать другой механизм

- использовать `domain`, если нужна главная дисциплинарная оптика;
- использовать `section`, если нужен локальный тематический кластер;
- использовать `parent`, если нужна иерархическая принадлежность;
- использовать `related`, если нужна содержательная поперечная связь;
- использовать `aliases`, если речь о другом названии той же темы.

## Как агент должен вводить новый тег

Если существующие теги не подходят, агент не должен просто молча добавлять новый.

Он обязан явно указать:

- какой новый тег предлагается;
- почему уже подтверждённые теги не подходят;
- почему этот тег полезен сразу для нескольких заметок;
- почему это именно поперечный аспект, а не скрытый `section` или `domain`.

## Практическое правило для агента

Перед добавлением тега агент должен проверить:

1. Не дублирует ли тег существующее поле.
2. Нужен ли этот тег хотя бы нескольким заметкам.
3. Является ли он поперечным аспектом, а не местом темы в онтологии.
4. Нет ли уже подходящего существующего тега.
5. Не описывает ли он на самом деле новый `section` или отдельную подветку `overview`.

Если хотя бы на один вопрос ответ отрицательный, новый тег лучше не добавлять.

## Что считается хорошим итоговым состоянием

Хорошая система тегов выглядит так:

- тегов немного;
- они повторно используются в разных частях базы;
- они помогают фильтровать, а не дублируют frontmatter;
- по тегу можно собрать полезный поперечный срез, а не просто ещё один список статей одного раздела.
