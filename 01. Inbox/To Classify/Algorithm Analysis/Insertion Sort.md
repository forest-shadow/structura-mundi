# Insertion Sort

## 1. Формальное определение

`Insertion Sort` (сортировка вставками) — это сравнением основанный (comparison-based) алгоритм сортировки, который строит отсортированную часть последовательности итеративно. На каждом шаге очередной элемент извлекается из неотсортированной части и вставляется в корректную позицию внутри уже упорядоченного префикса.

Более формально: после $i$-й итерации первые $i+1$ элементов находятся в отсортированном состоянии и содержат те же элементы, что и исходный префикс длины $i+1$, но уже в строго упорядоченном виде.

## 2. Этимология и происхождение термина

Термин `Insertion Sort` буквально означает «сортировка вставками». Название точно отражает центральную вычислительную операцию метода: новый элемент не обменивается с глобальным минимумом или максимумом всей коллекции, а целенаправленно вставляется в подходящее, локально вычисленное место внутри уже отсортированной части массива.

## 3. Историческое развитие и эволюция концепта

Алгоритм относится к числу классических базовых методов сортировки и исторически используется как один из канонических примеров локального упорядочивания последовательности. Эволюция метода в инженерной практике связана не с изменением математической модели, а с контекстом его архитектурного применения. Выделяются три исторические роли алгоритма:

- **Педагогический паттерн**: как базовый алгоритм для введения в концепции инвариантов циклов, локальных сдвигов памяти и строгого асимптотического анализа сложности.
    
- **Утилитарный метод**: как практически полезный алгоритм для коротких массивов, где отсутствие сложных структур управления оправдывает квадратичную асимптотику.
    
- **Компонент гибридных систем**: как вспомогательный вычислительный блок гибридных сортировок (например, Timsort или Introsort). В современных реализациях он применяется на малых подмассивах благодаря крайне низким константным издержкам и оптимальному поведению на почти отсортированных данных. Именно в гибридных алгоритмах наглядно демонстрируется принцип, согласно которому асимптотически слабый метод оказывается локально выгоден в системной реализации.
    

## 4. Формальные свойства, структура и ключевые признаки

**Ключевой механизм** На шаге с индексом $i$ алгоритм рассматривает элемент $A[i]$ как текущий вставляемый элемент и перемещает все строго большие элементы из диапазона $A[0..i-1]$ на одну позицию вправо, пока не будет найдена корректная позиция вставки.

**Инвариант алгоритма** Главный инвариант, обеспечивающий возможность формального доказательства корректности метода:

- Перед началом итерации $i$ префикс массива $A[0..i-1]$ уже отсортирован.
    
- После завершения итерации $i$ префикс массива $A[0..i]$ отсортирован.
    

**Ключевые структурные свойства** - **Сравнительный характер**: итоговый порядок элементов определяется исключительно через операции сравнения.

- **Локальность преобразования**: алгоритм работает с ближайшим окружением текущего элемента, не производя поиск глобального экстремума по всей неотсортированной части структуры.
    
- **Постепенное построение результата**: отсортированная область монотонно увеличивается слева направо.
    
- **Адаптивность к частичной упорядоченности**: количество необходимых сдвигов обратно пропорционально степени изначальной отсортированности входных данных.
    
- **Стабильность (Устойчивость)**: в стандартной реализации равные элементы сохраняют исходный относительный порядок, так как при сдвиге применяются только строгие сравнения вида $>$, предотвращающие избыточные перемещения дубликатов.
    
- **In-place (в мутационной парадигме)**: классическая мутационная реализация требует $O(1)$ дополнительной памяти, выполняя перестановки в пределах исходного массива.
    

**Дифференциация перемещений** Для точного понимания профиля производительности необходимо строго различать две операции:

- Сравнения: определяют предикат продолжения движения влево по префиксу.
    
- Сдвиги (shifts): физическое перемещение элементов вправо в памяти для высвобождения позиции под вставку.
    

Превосходство `Insertion Sort` над аналогами с квадратичной сложностью обусловлено не изменением асимптотики худшего случая, а резким снижением фактического числа сдвигов и сравнений на почти упорядоченных структурах.

## 5. Современное значение и интерпретации

В современной архитектуре программного обеспечения `Insertion Sort` редко выступает в качестве изолированного механизма для обработки массивов большого объема, однако его концептуальная значимость сохраняется в следующих аспектах:

- **Эталон адаптивности**: алгоритм является классической демонстрацией адаптивного поведения, где вычислительная стоимость динамически зависит от энтропии входных данных (степени упорядоченности), а не только от размера структуры $n$.
    
- **Локальный строительный блок**: применяется в гибридных комплексах сортировки. Стандартные библиотеки переключают контекст исполнения на сортировку вставками при достижении малых размеров подмассивов (пороговые значения от 16 до 64 элементов). На коротких диапазонах аппаратный выигрыш от отсутствия рекурсии и локальности кэша превосходит асимптотическое преимущество $O(n \log n)$ алгоритмов.
    
- **Аналитическая модель**: служит академической основой для изучения инвариантов структур данных, локального исправления нарушений порядка и анализа зависимости реального времени исполнения (Wall-clock time) от топологии данных.
    

## 6. Математическое и логико-формальное ядро

**Формальная схема вычислений** Для заданного массива $A$ длины $n$, итерация выполняется для каждого индекса $i$ от $1$ до $n-1$:

1. Сохраняется значение $key = A[i]$.
    
2. Инициализируется индекс сканирования $j = i - 1$.
    
3. Пока $j \ge 0$ и выполняется предикат $A[j] > key$, элемент $A[j]$ сдвигается вправо в позицию $A[j+1]$, а индекс $j$ декрементируется.
    
4. По завершении внутреннего цикла значение $key$ записывается в освободившуюся позицию $A[j+1]$.
    

**Корректность через математическую индукцию** Доказательство корректности алгоритма опирается на индукцию по длине отсортированного префикса:

- База индукции: префикс длины $1$ (состоящий из элемента $A[0]$) тривиально отсортирован по определению.
    
- Шаг индукции: предполагается, что префикс $A[0..i-1]$ отсортирован. Выполнение внутреннего цикла гарантирует, что элемент $A[i]$ будет интегрирован в позицию, не нарушающую монотонность последовательности. Следовательно, префикс $A[0..i]$ становится отсортированным.
    

**Математическая оценка сложности** - Худший случай: массив строго отсортирован в обратном порядке. На итерации $i$ элемент $A[i]$ сравнивается со всеми элементами префикса, что порождает арифметическую прогрессию числа операций: $1 + 2 + \dots + (n - 1) = \frac{n(n-1)}{2}$, что асимптотически равно $O(n^2)$.

- Лучший случай: массив предварительно отсортирован. Внутренний цикл прерывается после первой же проверки (так как $A[i-1] \le A[i]$). Общее число сравнений равно $n-1$, сдвиги отсутствуют. Сложность редуцируется до $O(n)$.
    

**Связь с теорией инверсий** Производительность метода строго коррелирует с числом математических инверсий. Инверсия определяется как пара индексов $(i, j)$, для которых $i < j$, но $A[i] > A[j]$. Каждое выполнение операции сдвига во внутреннем цикле алгоритма декрементирует общее число инверсий ровно на единицу. Таким образом, вычислительная сложность алгоритма прямо пропорциональна выражению $O(n + I)$, где $I$ — суммарное количество инверсий во входном массиве.

## 7. Профиль сложности

- **Худший случай (Worst case)**: $O(n^2)$
    
- **Средний случай (Average case)**: $O(n^2)$
    
- **Лучший случай (Best case)**: $O(n)$ (на строго отсортированном или массиве с числом инверсий $I \le c \cdot n$)
    
- **Дополнительная память (Space complexity)**:
    
    - $O(1)$ в классической in-place реализации.
        
    - $O(n)$ в функциональной (non-mutating) реализации из-за необходимости аллокации копии входного массива для изоляции состояний.
        

## 8. Псевдокод

Plaintext

```
insertionSort(A):
    for i from 1 to length(A) - 1:
        key = A[i]
        j = i - 1

        while j >= 0 and A[j] > key:
            A[j + 1] = A[j]
            j = j - 1

        A[j + 1] = key

    return A
```

## 9. Практические применения

- **Сортировка структур малого объема**: минимальные накладные расходы на управление потоком выполнения делают метод оптимальным для коротких массивов.
    
- **Обработка потоковых и почти отсортированных данных**: при малом числе инверсий алгоритм работает со скоростью, приближающейся к линейной, существенно опережая абстрактные оценки $O(n^2)$.
    
- **Интеграция в гибридные сортировки**: терминальная фаза для обработки локальных сегментов внутри рекурсивных алгоритмов (Merge Sort, Quick Sort).
    
- **Формальная верификация и академическое обучение**: прозрачность инвариантов обеспечивает удобную базу для доказательства корректности программ и понимания разрыва между асимптотическими пределами и реальной тактовой стоимостью операций процессора.
    

## 10. Сравнение с соседними алгоритмами

- **Insertion Sort vs Selection Sort**: `Selection Sort` выполняет минимально возможное количество обменов $O(n)$, но абсолютно не адаптивен к входным данным, всегда выполняя $O(n^2)$ сравнений. `Insertion Sort` высокочувствителен к топологии данных и досрочно прерывает сканирование, что делает его предпочтительным при любой степени изначальной упорядоченности.
    
- **Insertion Sort vs Bubble Sort**: Оба метода обладают квадратичной асимптотикой, однако `Insertion Sort` инженерно эффективнее. Он выполняет поиск позиции через экономичные сдвиги памяти (перезапись одного элемента за такт), тогда как `Bubble Sort` требует реализации полных обменов (swap) соседних элементов, что увеличивает нагрузку на шину памяти в три раза для каждой транзакции.
    
- **Insertion Sort vs $O(n \log n)$ алгоритмы**: На больших неструктурированных массивах методы `Merge Sort`, `Heap Sort` и `Quick Sort` демонстрируют критическое превосходство в масштабируемости. Ниша `Insertion Sort` ограничена короткими диапазонами и данными с искусственно заниженной энтропией, где константа при факторе $n \log n$ становится дороже квадратичной зависимости от малого $n$.
    

## 11. Типичные ошибки, заблуждения и ограничения

- **Отождествление эмпирической скорости с асимптотической эффективностью**: высокая скорость работы на частных случаях (почти отсортированные данные) не отменяет фундаментального квадратичного предела в худшем случае.
    
- **Игнорирование энтропии входных данных**: при выборе алгоритма оценка существующей структуры (количества инверсий) часто является более важным критерием, чем абстрактный объем массива.
    
- **Нарушение инварианта стабильности**: использование нестрогого неравенства ($\ge$ вместо $>$) в условии цикла `while` приводит к избыточным сдвигам эквивалентных элементов, необратимо разрушая свойство стабильности (stability) алгоритма.
    
- **Смешение семантики обменов (swap) и сдвигов (shift)**: реализация внутреннего цикла через последовательные `swap`-операции является грубой инженерной ошибкой, увеличивающей константу времени выполнения по сравнению со стандартным каскадным сдвигом.
    
- **Искажение профиля памяти в non-mutating парадигме**: концептуально алгоритм является in-place методом ($O(1)$), однако требование иммутабельности входных структур на уровне функции вынуждает аллоцировать $O(n)$ дополнительной памяти для работы с копией данных.
    
- **Архитектурное миспозиционирование**: использование `Insertion Sort` в качестве единственного или основного алгоритма обработки больших баз данных без гибридных оберток является антипаттерном проектирования.
    

## 12. Варианты реализаций на TypeScript и Go

В соответствии с требованиями к чистоте функций и предотвращению побочных эффектов (side-effects), представленные ниже реализации строго следуют правилу иммутабельности. Алгоритмы не модифицируют переданные по ссылке массивы, предварительно выделяя память под независимые копии.

### TypeScript

Обеспечение стабильности гарантируется строгим оператором `>`, сдвигающим исключительно те элементы префикса, которые больше текущего `key`.

TypeScript

```
function insertionSort(input: number[]): number[] {
  // Выделение памяти и создание изолированной копии
  const arr = [...input];

  for (let i = 1; i < arr.length; i++) {
    const key = arr[i];
    let j = i - 1;

    // Предикат строгого неравенства гарантирует стабильность сортировки
    while (j >= 0 && arr[j] > key) {
      arr[j + 1] = arr[j];
      j--;
    }

    arr[j + 1] = key;
  }

  return arr;
}
```

### Go

Копирование данных через идиому `append([]int(nil), input...)` гарантирует создание нового среза (slice) с независимым базовым массивом (backing array), полностью исключая мутацию исходного состояния.

Go

```
package main

// InsertionSort возвращает новый отсортированный срез, сохраняя исходный без изменений
func InsertionSort(input []int) []int {
	arr := append([]int(nil), input...)

	for i := 1; i < len(arr); i++ {
		key := arr[i]
		j := i - 1

		for j >= 0 && arr[j] > key {
			arr[j+1] = arr[j]
			j--
		}

		arr[j+1] = key
	}

	return arr
}
```

Анализ алгоритма доказывает, что локальная интеграция элемента в предварительно упорядоченный домен может быть практически эффективнее ресурсоемких глобальных методов, в особенности на данных малого объема или системах с низкой начальной энтропией. Благодаря этим свойствам, алгоритм выходит за рамки сугубо академического инструмента, выступая критически важным компонентом оптимизации внутри высокопроизводительных гибридных архитектур сортировки.

# Insertion Sort

## Краткое определение

`Insertion Sort` — это алгоритм сортировки, который строит отсортированную часть массива постепенно, вставляя каждый новый элемент в уже правильное место внутри этой части.

## Основная идея или механизм

- считать левую часть массива уже отсортированной;
- брать следующий элемент;
- сдвигать большие элементы вправо;
- вставлять текущий элемент в нужную позицию.

## Почему тема находится в ветке Sorting Algorithms

Эта заметка важна как переход от совсем наивных сортировок к алгоритму, который все еще прост, но уже хорошо ведет себя на почти отсортированных данных и часто используется как вспомогательный building block.

## Профиль сложности

- worst case: `O(n^2)`
- best case на почти отсортированном массиве: `O(n)`
- дополнительная память: обычно `O(1)`

## Рабочие свойства и границы темы

- обычно считается stable;
- обычно работает in-place;
- полезен на маленьких входах и как вспомогательная часть гибридных сортировок.
