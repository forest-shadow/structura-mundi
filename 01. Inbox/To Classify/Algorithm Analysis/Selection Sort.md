# 1. Selection Sort

> <font color="#fb4934">Selection Sort</font> — это детерминированный comparison-based алгоритм сортировки, который на каждой итерации выбирает экстремальный элемент из неотсортированной части последовательности и помещает его в следующую фиксируемую позицию отсортированной части.

В канонической форме при сортировке по возрастанию алгоритм последовательно ищет минимальный элемент на суффиксе массива и меняет его местами с первым элементом этого суффикса. Selection Sort относится к простым квадратичным алгоритмам сортировки и обычно рассматривается как учебный метод, позволяющий явно различать стоимость операций сравнения и стоимость операций обмена (мутации памяти).

# 2. Этимология и происхождение термина

Термин «selection» (выбор) прямо указывает на центральное семантическое действие алгоритма: на каждом шаге он не переставляет элементы локально в процессе обхода, а именно целенаправленно выбирает один экстремальный элемент из оставшейся неотсортированной части для его окончательного позиционирования.

# 3. Историческое развитие и эволюция концепта

Selection Sort относится к числу классических элементарных алгоритмов сортировки, которые исторически использовались прежде всего как базовые формальные модели для изучения алгоритмического мышления, концепций инвариантов цикла и основ асимптотического анализа.

В процессе эволюции алгоритмики его значение закрепилось не в качестве практического универсального метода, а как удобного аналитического эталона для сравнения с другими простыми сортировками, прежде всего с [[Bubble Sort]] и [[Insertion Sort]]. На историческом фоне развития вычислительных систем особенно важным стало наблюдение, вытекающее из анализа данного алгоритма: принципиально малое число операций обмена в памяти не компенсирует колоссального (квадратичного) числа операций сравнения в общей модели вычислительной стоимости.

# 4. Формальные свойства, структура, ключевые признаки

## Структура алгоритма

Алгоритм строго поддерживает и разделяет две логические области обрабатываемого массива:
- Префикс: область, уже приведённая в окончательное отсортированное состояние.
- Суффикс: область, которая ещё не обработана и содержит неотсортированные элементы.

На итерационном шаге с индексом $i$ алгоритм последовательно выполняет следующие действия:
1. Ищет индекс минимального элемента на диапазоне от $i$ до $n-1$.
2. Выполняет операцию обмена (swap) элементов `a[i]` и `a[minIndex]`.
3. Увеличивает длину отсортированного префикса ровно на единицу.

## Ключевой инвариант

После завершения $k$ итераций внешнего цикла первые $k$ элементов массива гарантированно содержат $k$ наименьших элементов исходной последовательности в строгом и правильном порядке. Этот инвариант является главным формальным математическим основанием для доказательства корректности алгоритма.

## Характерные признаки

- <b>Принадлежность к comparison-based алгоритмам:</b> итоговый порядок элементов определяется исключительно через операции попарного сравнения.
- <b>Пространственная эффективность:</b> алгоритм функционирует in-place, дополнительная память не зависит от размера ввода $n$, за исключением константной памяти для временной переменной при обмене.
- <b>Отсутствие стабильности:</b> алгоритм в канонической форме не является stable, так как обмен найденного глобального минимума с текущей позицией может деструктивно изменить исходный относительный порядок равных элементов.
- <b>Детерминированность профиля сравнений:</b> алгоритм делает фиксированно много сравнений для заданного $n$, абсолютно независимо от того, отсортирован входной массив изначально или нет.
- <b>Минимизация мутаций:</b> алгоритм делает относительно мало обменов, гарантируя не более одного обмена на каждую итерацию внешнего цикла.

# 5. Современное значение и интерпретации

В современной инженерной практике Selection Sort практически не используется как универсальный алгоритм сортировки для реальных рабочих нагрузок. Его квадратичная стоимость сравнений делает его существенно слабее итерированных алгоритмов уровня [[Merge Sort]], [[Quick Sort]] или оптимизированных гибридных библиотечных реализаций (например, Timsort или IntroSort).

Тем не менее, концепт сохраняет фундаментальное значение в трёх специфических контекстах:
- **Первый контекст (обучение)**: Алгоритм наглядно демонстрирует структурное различие между поиском экстремума на множестве и локальными соседними перестановками.
- **Второй контекст (анализ стоимости операций)**: Selection Sort выступает идеальным полигоном там, где необходимо концептуально разделять и отдельно обсуждать цену процессорной инструкции сравнения и цену физической записи или обмена в памяти.
- **Третий контекст (малые или специализированные задачи)**: В редких и узкоспециализированных аппаратных средах, где число циклов записи критично (из-за износа носителя или огромных задержек шины), а размер входного массива крайне мал, его специфический профиль «много сравнений, мало обменов» может представлять локальный инженерный интерес, хотя это и не делает алгоритм общим практическим выбором

# 6. Математическое или логико-формальное ядро

## Число сравнений

Для массива длины $n$ алгоритм сравнивает элементы в строго убывающей арифметической прогрессии:
- На первой итерации: $n - 1$ сравнений;
- На второй итерации: $n - 2$ сравнений;
- На предпоследней итерации: $1$ сравнение.

Итоговая сумма ряда выражается формулой:

$$(n - 1) + (n - 2) + \dots + 1 = \frac{n(n - 1)}{2}$$

Следовательно, точное число сравнений имеет асимптотический порядок:

$$\Theta(n^2)$$

Данная оценка является константой и абсолютно верна для best case, average case и worst case в стандартной реализации, так как внутренний цикл всегда проходит до конца суффикса.

## Число обменов

В канонической форме алгоритм выполняет не более $n - 1$ обменов, что математически выражается как:

$$O(n)$$

Если реализация включает условный оператор, исключающий холостой обмен в случае `minIndex == i`, фактическое число обменов может быть меньше максимального предела, однако асимптотический порядок линейной границы не меняется.

## Пространственная сложность

Дополнительная память в обычной in-place реализации строго ограничена хранением счетчиков циклов и буферной переменной:

$$O(1)$$

# 7. Псевдокод алгоритма


```Plaintext
selectionSort(A):
    n := length(A)

    for i := 0 to n - 2:
        minIndex := i

        for j := i + 1 to n - 1:
            if A[j] < A[minIndex]:
                minIndex := j

        if minIndex != i:
            swap A[i], A[minIndex]

    return A
```

# 8. Практические применения

Для прикладной backend- или frontend-разработки Selection Sort не является рекомендуемым выбором. Однако тема имеет подтвержденную практическую ценность как инструмент глубокого инженерного анализа.

## Применения в качестве учебной модели:

- Строгое математическое объяснение инвариантов сортировки.
- Различение операций сравнения и обмена как принципиально разных метрик вычислительной стоимости.
- Демонстрация контринтуитивного факта, что асимптотическая сложность может совершенно не зависеть от исходной степени упорядоченности входных данных (отсутствие адаптивности).

## Применения в качестве модели ограничений:

- Анализ аппаратных сценариев, где запись в физическую память (например, EEPROM или Flash) экспоненциально дороже или деструктивнее чтения/сравнения.
- Обсуждение архитектурного компромисса между общим числом сравнений и общим числом перестановок элементов.

- Построение базового фона для академического сравнения с [[Insertion Sort]], который в отличие от сортировки выбором обычно на порядки лучше использует частично отсортированный входной поток.
    

### 9. Типичные ошибки, заблуждения, ограничения

**Типичные заблуждения:**

- **Заблуждение**: Если алгоритм делает мало обменов, значит он автоматически эффективен.
    
    **Уточнение**: Selection Sort действительно делает мало обменов ($O(n)$), но почти всегда выполняет строго квадратичное число сравнений ($\Theta(n^2)$), и именно эта метрика обычно определяет его слабую практическую производительность на современных CPU.
    
- **Заблуждение**: Если входной массив уже почти отсортирован, алгоритм заметно ускорится.
    
    **Уточнение**: В своей стандартной форме Selection Sort абсолютно не выигрывает от частичной упорядоченности данных, поскольку инвариант требует полного итерирования через весь неотсортированный хвост для валидации минимума.
    
- **Заблуждение**: Selection Sort является stable алгоритмом.
    
    **Уточнение**: Обычная in-place реализация алгоритма нестабильна. Прямой обмен найденного глобального минимума с текущей границей префикса может перебросить элемент через другой элемент с таким же значением ключа, перманентно изменив их относительный порядок.
    

**Ограничения:**

- Квадратичное число математических сравнений делает алгоритм категорически непригодным для сортировки больших массивов данных.
    
- Алгоритм принципиально плохо адаптируется к частично отсортированным или почти отсортированным наборам данных.
    
- Архитектурная нестабильность существенно ограничивает его применимость в системах, где исходный относительный порядок равных элементов (например, сортировка по вторичному ключу) семантически значим.
    
- Даже при феноменально малом числе операций обмена, он в подавляющем большинстве случаев уступает более современным алгоритмам по метрикам суммарной производительности.

### 11. Примеры реализации на TypeScript и Go

Ниже приведены академически корректные реализации алгоритма. В соответствии с принципами чистой архитектуры и функционального подхода, входной массив в данных примерах не мутируется: все операции сортировки выполняются на предварительно выделенной изолированной копии.

**TypeScript:**

TypeScript

```
function selectionSort(input: number[]): number[] {
  const arr = [...input];
  const n = arr.length;

  for (let i = 0; i < n - 1; i++) {
    let minIndex = i;

    for (let j = i + 1; j < n; j++) {
      if (arr[j] < arr[minIndex]) {
        minIndex = j;
      }
    }

    if (minIndex !== i) {
      const temp = arr[i];
      arr[i] = arr[minIndex];
      arr[minIndex] = temp;
    }
  }

  return arr;
}
```

**Go:**

Go

```
package main

func SelectionSort(input []int) []int {
	arr := append([]int(nil), input...)
	n := len(arr)

	for i := 0; i < n-1; i++ {
		minIndex := i

		for j := i + 1; j < n; j++ {
			if arr[j] < arr[minIndex] {
				minIndex = j
			}
		}

		if minIndex != i {
			arr[i], arr[minIndex] = arr[minIndex], arr[i]
		}
	}

	return arr
}
```

### 12. Краткий вывод

Selection Sort — это простой in-place алгоритм сортировки с жестким квадратичным числом сравнений и строго линейным числом обменов. Его главное теоретическое значение состоит в том, что он алгоритмически ясно и однозначно отделяет процедуру поиска и выбора экстремума от процедуры физической перестановки элементов. Тем самым алгоритм наглядно доказывает фундаментальный тезис теории вычислимости: изолированно малая стоимость записей в память не тождественна малой общей стоимости работы алгоритма. Для практической сортировки реальных и больших наборов данных в production-средах он обычно не является предпочтительным выбором, но для академического анализа алгоритмов, профилирования ресурсов и систематизации базовых computer science идей неизменно остаётся важным и необходимым.

# Selection Sort

## Краткое определение

`Selection Sort` — это алгоритм сортировки, который на каждом шаге ищет минимальный элемент в неотсортированной части и ставит его в следующую правильную позицию.

## Основная идея или механизм

- разделить массив на отсортированную и неотсортированную части;
- найти минимум в неотсортированной части;
- обменять его с первым элементом этой части;
- повторять до конца массива.

## Почему тема находится в ветке Sorting Algorithms

Эта заметка полезна как отдельный учебный паттерн: в отличие от `Bubble Sort`, здесь центральна не серия локальных обменов, а выбор следующего минимального элемента.

## Профиль сложности

- best / average / worst case обычно: `O(n^2)`
- дополнительная память: обычно `O(1)`

## Рабочие свойства и границы темы

- как правило, не считается stable в обычной форме;
- обычно работает in-place;
- делает мало обменов, но все равно требует много сравнений.
