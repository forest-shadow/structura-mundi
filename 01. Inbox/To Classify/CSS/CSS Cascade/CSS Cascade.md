# 1. CSS Cascade

> <font color="#fb4934">CSS Cascade (Каскад CSS)</font> — это нормативный алгоритм и фундаментальный механизм языка Cascading Style Sheets, посредством которого пользовательский агент (браузер) определяет, какое именно декларативное значение свойства должно быть выбрано для конкретного элемента, если к этому элементу и этому свойству одновременно применимы несколько конкурирующих CSS-деклараций.

В строгом архитектурном смысле каскад представляет собой детерминированную систему разрешения конкуренции. Алгоритм упорядочивает множество кандидатных правил по строго регламентированной иерархии критериев и изолированно выбирает единственное победившее значение для каждого индивидуального свойства. Данный механизм не является исчерпывающим для всей модели вычисления стилей, а образует лишь одну из её центральных осей. В рамках единой экосистемы вычислений каскад занимает связующее положение между процессом сопоставления селекторов (selector matching) и финальным вычислением значений. Система функционирует в неразрывной связи с такими концептами, как [[CSS Specificity]], [[CSS Inheritance]], [[CSS Value Processing]] и [[CSS Cascade Origins, Importance, Layers, and Order]].

# 2. Этимология и происхождение термина

Термин **cascade** восходит к общеязыковому значению «каскад», «последовательное ниспадение», «водопад» или «ступенчатое прохождение». В контексте CSS (с момента предложения спецификации Хоконом Виумом Ли в 1994 году) он закрепился как метафорическое обозначение многоступенчатого механизма приоритизации. Итоговое решение алгоритма возникает не из единственного изолированного критерия, а в результате последовательного прохождения свойств через серию фильтров и сравнений между конкурирующими источниками стиля.

Название подчёркивает концептуальную основу языка: стиль не назначается элементу одномоментно и авторитарно. На DOM-узел «наслаиваются» декларации различных происхождений (браузер, пользователь, разработчик), уровней приоритета и степеней гранулярности, после чего машина логического вывода последовательно разрешает конфликты между ними, двигаясь от наиболее общих уровней к наиболее специфичным.

# 3. Историческое развитие и эволюция концепта

Каскад был заложен в основу CSS на самых ранних этапах стандартизации консорциумом W3C и являлся одной из его определяющих парадигм. В отличие от более примитивных императивных моделей оформления, где стиль воспринимается как непосредственная привязка атрибутов к элементу (например, тег `<font>` в раннем HTML), CSS изначально проектировался как язык, допускающий иерархическое сосуществование нескольких источников оформления. Эволюция концепта двигалась по вектору усложнения, детализации и строгой формализации критериев выбора:

- **Ранние спецификации (CSS1, CSS2):** Основное внимание уделялось базовому происхождению стиля, флагу важности (`!important`), расчету веса (специфичности) селекторов и лексическому порядку следования правил в исходном коде.
- **Уточнение источников (CSS2.1 / CSS3):** Модель была расширена строгим разделением на _origins_: стили пользовательского агента (браузера), пользовательские стили (в том числе для обеспечения доступности) и авторские стили (код разработчика).
- **Эра архитектурного контроля (CSS Cascading and Inheritance Level 5):** Внедрение каскадных слоев (`@layer`) позволило авторам декларативно управлять конфликтами независимо от специфичности селекторов, решив многолетнюю проблему «гонки весов».
- **Современная эволюция (Level 6 и далее):** Интеграция каскада с механизмами ограничения области видимости (`@scope`), а также с виртуальными правилами из модуля CSS Animations и Transitions. Каскад перестал быть исключительно «спором селекторов» и оформился в универсальный арбитражный механизм разрешения конкурирующих стилевых предписаний.

Развитие алгоритма отражает фундаментальный сдвиг: переход CSS от утилитарного языка форматирования текстовых документов к полноформатной декларативной системе визуального программирования сложных пользовательских интерфейсов.

# 4. Формальные свойства, структура и ключевые признаки

## 4.1. Предмет действия каскада

Каскад оперирует не целыми CSS-правилами (блоками), а исключительно **отдельными декларациями свойств**. Это архитектурное свойство означает, что одно и то же CSS-правило может оказаться «победителем» для одного свойства (например, `color`) и «проигравшим» для другого (например, `font-size`), если в системе присутствуют разные конкуренты с иными весами. Единицей сравнения в каскаде является строгий триплет:

1. Целевой DOM-элемент.
2. Конкретное CSS-свойство.
3. Множество всех релевантных деклараций, претендующих на задание значения этому свойству для данного элемента.

## 4.2. Фильтрация релевантности (Условие включения)

Каскадный алгоритм не обрабатывает все существующие декларации глобально. Первичным этапом является строгая фильтрация. Декларация признается валидным кандидатом для каскада исключительно при одновременном выполнении следующих условий:

- Селектор правила успешно сопоставлен с целевым элементом.
- Контекстные условные конструкции (at-rules), инкапсулирующие правило, возвращают истину в текущем состоянии окружения (`@media`, `@supports`, `@container`).
- Декларация синтаксически корректна и семантически применима к указанному свойству.
- Стилевая таблица, содержащая правило, включена в активную стилевую среду документа (не отключена через атрибуты или DOM API).

Собственно задача каскада (выбор победителя) инициируется только в отношении множества уже отфильтрованных, релевантных кандидатов.

## 4.3. Основные оси измерений (Критерии сравнения)

Современный каскад применяет критерии фильтрации строго последовательно. Как только на определенном уровне обнаруживается один доминирующий кандидат (или группа в одной категории), нижележащие критерии не вычисляются. Упрощенно-строгая иерархия выглядит следующим образом:

1. **Origin and Importance (Происхождение и Важность):** Оценка источника стиля (браузер, пользователь, автор) и наличия аннотации `!important`.
2. **Context and Encapsulation:** Учет контекста Shadow DOM. Стили, определенные в Shadow-корне, приоритетнее внешних стилей для инкапсулированных элементов.
3. **Layer Order (Порядок слоев):** Сравнение по положению внутри явно заданных каскадных слоев (`@layer`). Декларации вне слоев имеют наивысший приоритет над слоистыми декларациями (в обычном контексте автора).
4. **Scoping Proximity (Близость области видимости):** Применение критерия дистанции для конкурирующих правил, объявленных в рамках `@scope`. Меньшее расстояние в дереве от корня области видимости дает преимущество.
5. **Specificity (Специфичность селектора):** Математическое вычисление относительного веса селектора (ID, Классы/Атрибуты/Псевдоклассы, Теги/Псевдоэлементы).
6. **Order of Appearance (Порядок появления):** Детерминированный исход (tie-breaker) — побеждает декларация, обработанная парсером позже в исходном коде.

_Аксиома каскада: Специфичность является лишь одним из средних этапов алгоритма, а не синонимом всего процесса._

## 4.4. Происхождения стилей (Origins)

Классическая модель базируется на трех фундаментальных источниках (перечислены в порядке возрастания приоритета для обычных деклараций):

* **User-agent origin:** Базовые стили по умолчанию, предоставляемые самим браузером для обеспечения минимальной читаемости документа.
* **User origin:** Настройки, инжектированные пользователем (например, локальные таблицы стилей для высококонтрастного режима или пользовательские расширения).
* **Author origin:** Стили, предоставленные разработчиком веб-приложения (стили документа).

## 4.5. Уровень переопределения (Importance)

Аннотация `!important` не делает селектор «абсолютным победителем». Ее математический смысл заключается в инверсии приоритетов происхождения и слоев. Декларация перемещается в зеркально отраженный, высший класс сравнения. Важно отметить, что `!important` декларации пользователя превалируют над `!important` декларациями автора — это заложено W3C для обеспечения прав пользователей на доступность (Accessibility).

## 4.6. Архитектура каскадных слоев

Директива `@layer` ввела ортогональное измерение. Она позволяет авторам устанавливать приоритеты не как побочный артефакт роста специфичности или порядка загрузки файлов, а как осознанную архитектурную топологию.

_Пример типовой структуры:_ `base` -> `components` -> `utilities` -> `overrides`.

Внедрение слоев трансформировало каскад из пассивного механизма разрешения конфликтов в активный инструмент системной организации стилей.

# 5. Структура работы алгоритма каскада

Процесс разрешения конфликта алгоритмически разделен на детерминированные стадии:

1. **Сбор кандидатов:** Выполнение selector matching. Предикатный поиск всех правил, соответствующих графу (Element, Property).
2. **Нормативная фильтрация:** Отсечение кандидатов, находящихся в ложных ветвях Media Queries, Container Queries или имеющих синтаксические ошибки в значении.
3. **Разрешение конкуренции:** Лексикографическое упорядочивание оставшихся кандидатов по каскадному порядку приоритетов (Origins -> Layers -> Specificity -> Order).
4. **Формирование Specified Value:** Декларация, занявшая верхнюю позицию (максимум) в отсортированном множестве, назначается как _specified value_ (указанное значение) для данного свойства.
5. **Делегирование в модель значений:** Каскад завершает свою работу. Далее процессор значений применяет механизмы наследования (если каскад ничего не нашел или встретил `inherit`), подстановки начальных значений (initial values), и выполняет математические преобразования до стадий _computed_, _used_ и _actual values_. Фундаментальная граница концепта: каскад отвечает исключительно за выбор декларации, а не за вычисление ее финального пиксельного результата на матрице экрана.

# 6. Логико-формальное ядро

С математической точки зрения, каскад представляет собой функцию нахождения глобального максимума на множестве деклараций, где порядок задан лексикографически.

Пусть:
- $E$ — узел DOM-дерева;
- $P$ — вычисляемое CSS-свойство;
- $C(E, P)$ — отфильтрованное множество всех релевантных деклараций для пары $(E, P)$.

Тогда итоговое значение алгоритма:

$$winner(E, P) = \max(C(E, P), \prec_{cascade})$$

Где $\prec_{cascade}$ — не скалярное числовое значение, а строгий лексикографический порядок (кортеж весов), сравнивающий элементы по координатам: `[OriginImportance, LayerPosition, ScopingProximity, Specificity, SourceOrder]`. Попытка свести весь каскад к единому числу (как это делали в старых статьях про "вес селектора") методологически некорректна. Высокая специфичность никогда не сможет компенсировать проигрыш на более раннем этапе (например, на уровне `origin` или `layer`).

Для точной демонстрации логического ядра приведена формальная модель функции сравнения на TypeScript:

```TypeScript
// Определение векторного пространства специфичности
type SpecificityVector = [number, number, number];

// Внутренняя структура декларации-кандидата
interface CssCandidate {
    id: string;
    originImportanceWeight: number; // 1: UA, 2: User, 3: Author, 4: Author!important, 5: User!important...
    layerOrder: number;             // Индекс слоя (чем больше число, тем выше приоритет слоя)
    scopingProximity: number;       // Дистанция от корня скоупа (меньше = приоритетнее, Infinity = вне скоупа)
    specificity: SpecificityVector; // Вектор специфичности [ID, Class, Tag]
    sourceOrder: number;            // Индекс появления в исходном дереве парсера
}

/**
 * Лексикографический компаратор каскада CSS.
 * Оценивает две декларации и возвращает кандидата с наивысшим приоритетом.
 */
function resolveCascadeConflict(a: CssCandidate, b: CssCandidate): CssCandidate {
    // 1. Сравнение Origin & Importance
    if (a.originImportanceWeight !== b.originImportanceWeight) {
        return a.originImportanceWeight > b.originImportanceWeight ? a : b;
    }

    // 2. Сравнение слоев (Layers)
    if (a.layerOrder !== b.layerOrder) {
        return a.layerOrder > b.layerOrder ? a : b;
    }

    // 3. Сравнение близости Scope (меньшая дистанция = высший приоритет)
    if (a.scopingProximity !== b.scopingProximity) {
        return a.scopingProximity < b.scopingProximity ? a : b;
    }

    // 4. Сравнение векторов специфичности
    for (let i = 0; i < 3; i++) {
        if (a.specificity[i] !== b.specificity[i]) {
            return a.specificity[i] > b.specificity[i] ? a : b;
        }
    }

    // 5. Tie-breaker: Порядок в исходном коде
    return a.sourceOrder > b.sourceOrder ? a : b;
}
```

# 7. Современное значение и интерпретации

В парадигме современного системного веб-инжиниринга каскад интерпретируется не как рудиментарный механизм («кто кого переопределит» или «костыль для стилизации»), а как оркестратор источников состояния интерфейса. Значение алгоритма разворачивается в следующих академических измерениях:

- **Нормативная модель приоритета:** Формальный контракт браузера, математически гарантирующий предсказуемое разрешение конкуренции.
- **Архитектурный инструмент:** Благодаря абстракции `@layer` разработчики получили возможность проектировать каскад сверху вниз, контролируя приоритеты целых библиотек или подсистем без изменения HTML-разметки.
- **Механизм когнитивного сосуществования:** Обеспечивает неконфликтное слияние статичных стилей, динамических пользовательских преференций (например, `prefers-color-scheme`) и временных состояний виртуальной машины браузера (Web Animations API).
- **Система локального конфликторазрешения:** Гранулярность каскада (работа на уровне отдельного свойства, а не правила) предотвращает необходимость глобального подавления одного класса другим, позволяя гибко композировать свойства из множества мелких утилит.

Чем выше зрелость CSS-архитектуры в проекте, тем реже она опирается на побочный рост специфичности селекторов, и тем больше использует детерминированные границы каскада: слои, скоупинг и компонентную изоляцию.

# 8. Практические применения

1. **Архитектура дизайн-систем:** Построение устойчивой топологии стилей, где базовые нормализаторы (normalize/reset), дизайн-токены, компонентные абстракции и утилитарные классы распределены по соответствующим каскадным слоям. Без опоры на этот алгоритм глобальные дизайн-системы неизбежно коллапсируют в набор конфликтующих исключений.
2. **Управление энтропией в больших монорепозиториях:** Разрешение коллизий между Legacy-кодом, изолированными микрофронтендами, внешними библиотеками UI (например, Bootstrap или Material) и локальными правками продуктовых команд. Инжиниринг каскада радикально снижает потребность в аварийном использовании флагов важности.
3. **Инкапсуляция и контролируемая проницаемость:** Даже в средах, использующих CSS-in-JS, CSS Modules или изоляцию через Shadow DOM, инженер сталкивается с необходимостью инжектировать внешние темы в закрытые компоненты. Каскад остается базовым протоколом взаимодействия между "внутренней" приватной областью элемента и "внешней" публичной системой стилей приложения.
4. **Аналитика и визуальная отладка:** Профессиональная диагностика в DevTools сводится к обратному инжинирингу каскада: определению состава множества $C(E, P)$ и выявлению точной координаты (слой, специфичность или лексический порядок), на которой проиграла искомая декларация.

# 9. Типичные ошибки, заблуждения и ограничения

1. **Редукция каскада до расчета специфичности:** Самая массовая анти-паттерн интерпретация. Специфичность — лишь локальный этап (обычно 4-й или 5-й шаг). Она игнорируется, если конфликт уже разрешен на уровне происхождения или слоя. Высшая специфичность авторского класса не переопределит `!important` правило пользовательского агента.
2. **Абсолютизация исходного порядка (Source Order):** Заблуждение о том, что «последний селектор в файле всегда побеждает». Лексический порядок является исключительно резервным tie-breaker механизмом. Он вступает в силу _только_ при полном математическом равенстве всех вышестоящих критериев.
3. **Деструктивное применение флага важности:** Вера в то, что `!important` является архитектурным инструментом фиксации стиля. Систематическое внедрение флага свидетельствует о потере контроля над графом каскада. Он нарушает предсказуемость лексикографической сортировки, делая локальную отладку экспоненциально сложнее.
4. **Логическое смешение с наследованием (Inheritance):** Каскад выбирает победителя среди деклараций, непосредственно адресованных элементу селектором. Наследование — принципиально иной процесс, активирующийся _после_ каскада, заполняющий пустые значения от родительского узла графа.
5. **Концептуальные ограничения алгоритма:** Каскад разрешает только декларативные приоритетные конфликты. Он не способен:
	- Автоматически исправить плохую компонентную архитектуру.
	- Устранить необходимость семантического структурирования HTML.
	- Предотвратить логические коллизии при небрежной организации пространств имен (если не используются слои или скоупы).

# 10. Связанные понятия

- **[[CSS Specificity]]**: Частный алгоритмический механизм вычисления веса индивидуального селектора, применяемый как один из фильтров внутри широкой конвейерной модели каскада.
- **[[CSS Inheritance]]**: Механизм передачи вычисленных значений свойств от родительского DOM-узла к дочернему. Выступает запасным протоколом (fallback), срабатывающим в случае, если каскад не вернул победившей декларации.
- **[[CSS Cascade Origins, Importance, Layers, and Order]]**: Строгая детализация внутренних векторов приоритизации, по осям которых алгоритм осуществляет сортировку массива кандидатов.
- **[[CSS Selectors]]**: Лексические паттерны, служащие предикатами для включения декларации в исходное множество кандидатов до инициализации каскадного сравнения.
- **[[CSS Value Processing]]**: Обширная цепочка трансформации значений, стартующая на выходе из каскада (перевод относительных единиц в абсолютные, интерполяция, расчет геометрии).
- **[[CSS @layer]]**: Директива, предоставляющая API для внедрения кастомных осей приоритета (каскадных слоев), изолирующих группы правил друг от друга.
- **[[CSS @scope]]**: Интерфейс ограничения области влияния правил по границам поддеревьев DOM, модифицирующий расчет приоритета через введение метрики близости (scoping proximity).
