# Go Testing

## Краткое определение

> <font color="#fb4934">Тестирование в Go</font> — это встроенная часть стандартного инструментария языка: `go test` автоматически запускает функции вида `TestXxx(*testing.T)` из файлов `*_test.go`; в той же модели поддерживаются subtests, benchmarks, fuzz tests и example tests. Тестовые файлы могут лежать либо в том же пакете, либо в отдельном пакете с суффиксом `_test`, что соответствует, соответственно, white-box и black-box стилю проверки. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

## Центральная идея

Наиболее практичная ментальная модель для Go звучит так: **тест — это исполнимая спецификация контракта функции**, а не повторение её реализации. В терминах практики это означает, что тест должен отвечать на три вопроса: какие входы допустимы, какое наблюдаемое поведение обещано, и по какому критерию мы понимаем, что контракт выполнен или нарушен. Эта модель особенно полезна на live coding, потому что она снижает когнитивную нагрузку: вместо попытки “придумать тесты вообще” вы формализуете обещания API и проверяете их по ветвям поведения. Такой подход согласуется и с рекомендациями Go по table-driven tests, subtests и проверке именно стабильных, семантически значимых результатов. ([Go](https://go.dev/blog/subtests "Using Subtests and Sub-benchmarks - The Go Programming Language"))

## Что является объектом тестирования

В Go полезно различать не “виды тестов” как абстрактную классификацию, а **виды обещаний**, которые код даёт внешнему миру.

Первый вид — **точный результат**. Функция получает вход и обязана вернуть конкретное значение или структуру.

Второй вид — **инвариант**. Например, результат должен сохранять определённое свойство при любом допустимом входе. Именно такой тип проверки особенно хорошо ложится на fuzzing, который в Go встроен в `go test` и запускает функцию на случайных входах, чтобы искать ошибки, не покрытые обычными unit tests. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

Третий вид — **семантика ошибки**. Важно не только то, что ошибка возвращается, но и то, что она означает программно. Официальные рекомендации Go предостерегают от хрупких тестов, сравнивающих ошибки по строкам, и советуют тестировать именно error semantics, а не текст сообщения как таковой. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

Четвёртый вид — **побочный эффект**, например изменение окружения, файловой системы или временных артефактов. Для таких случаев `testing.T` предоставляет `Cleanup`, `TempDir` и `Setenv`, что позволяет строить тесты как локальные, самовосстанавливающиеся сценарии. `Setenv` при этом нельзя использовать в параллельных тестах, потому что он влияет на весь процесс. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

## Базовая ментальная схема

На практике самая полезная внутренняя схема состоит из пяти шагов.

### 1. Сначала формулируется контракт

Перед написанием теста нужно ответить не на вопрос “как тестировать”, а на вопрос “что функция обязана гарантировать”. Для функции-конструктора это обычно набор ветвей: успешное создание одного типа, успешное создание другого типа, и ошибочный сценарий. Для парсера это может быть корректный разбор валидного ввода, отказ на невалидном вводе и сохранение определённых инвариантов.

### 2. Затем контракт разбивается на независимые сценарии

Здесь возникает простая, но очень мощная привычка: **каждая смысловая ветвь поведения должна быть названа и помещена в свой test case**. Именно это делает table-driven tests в Go столь естественными: одна и та же проверочная логика применяется к серии кейсов. Go blog прямо отмечает, что table-driven подход является типичным способом писать родственные проверки, а subtests через `t.Run` позволяют оформить их структурно и удобно запускать по имени. ([Go](https://go.dev/blog/subtests "Using Subtests and Sub-benchmarks - The Go Programming Language"))

### 3. Для каждого сценария выбирается “оракул”

Оракул — это критерий истинности. Он может быть разным:

- точное значение;
    
- ожидаемая структура;
    
- выполнение инварианта;
    
- `errors.Is` / `errors.As`-семантика;
    
- отсутствие panic;
    
- наличие побочного эффекта в контролируемом виде.
    

Именно на этом этапе тест становится строгим. Без явного оракула тест превращается в неформальное наблюдение.

### 4. Только после этого выбирается форма теста

Если много входов проверяются по одинаковой логике, берётся table-driven test. Если нужны отдельные имена сценариев и тонкий запуск через CLI, добавляются subtests. Если тестируется свойство на широком пространстве входов, разумен fuzz test. Если интересует производительность, нужен benchmark. Все эти формы встроены в стандартный пакет `testing`. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

### 5. И лишь затем пишется код теста

На уровне микропаттерна это почти всегда:

**Arrange → Act → Assert**

То есть: подготовить данные, вызвать тестируемый код, затем проверить контракт. Эта схема не является специфически “go-only”, но в Go она особенно хорошо сочетается с лаконичной тестовой функцией и table-driven стилем.

## Почему table-driven tests так важны в Go

Table-driven tests в Go — не просто идиома, а удобный способ представить тест как **таблицу контрактов**: имя сценария, вход, ожидаемый результат, ожидаемая ошибка, особая проверка. Go Wiki рекомендует table-driven tests там, где множество кейсов проверяются одной и той же логикой, а если логика проверок начинает сильно расходиться, лучше разделять тесты на несколько функций или групп subtests. Это очень полезное правило для live coding: не надо насильно помещать всё в одну таблицу, если условия проверки уже стали разнородными. ([Go](https://go.dev/blog/subtests "Using Subtests and Sub-benchmarks - The Go Programming Language"))

Практический вывод для интервью таков: когда вы видите функцию с несколькими ветвями поведения, **по умолчанию думайте в сторону table-driven test**, но не превращайте таблицу в мини-интерпретатор с множеством `if` внутри `check`.

## Как правильно выбирать гранулярность тестов

Есть два конкурирующих импульса: писать один “большой” тест на всё и писать много мелких тестов. Практичная модель такова.

Если сценарии отличаются только входом и ожидаемым результатом, но не логикой проверки, нужен **один table-driven test**.

Если сценарии требуют разных критериев успеха, лучше писать **несколько test functions** или отдельные subtests. Go Wiki прямо предупреждает, что когда разные строки таблицы требуют уже разных правил проверки, тест становится трудночитаемым. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

Это особенно важно на live coding: читаемость теста ценится не меньше покрытия.

## White-box и black-box мышление

Go допускает два режима тестирования. Если файл лежит в том же пакете, тест может обращаться к неэкспортируемым идентификаторам; если файл лежит в пакете `mypkg_test`, он видит только экспортируемый API и ведёт себя как внешний потребитель. Это означает, что ещё до написания теста вы должны понять, **что именно вы хотите проверять: внутреннюю механику или внешний контракт**. Для unit tests бизнес-логики чаще полезнее второй взгляд: внешний контракт важнее внутреннего устройства. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

## Что проверять в первую очередь

Наиболее практичный порядок при написании unit test в Go такой.

Сначала проверяется **ошибка**: есть она или нет, и соответствует ли она ожидаемой семантике.

Затем проверяется **основной результат**.

Потом — **дополнительные свойства**, если они являются частью контракта.

Такой порядок хорош потому, что ошибка почти всегда определяет дальнейшую ветку рассуждения. Если функция должна была завершиться успешно, а вернула error, нет смысла дальше сравнивать основной результат как если бы он был валиден.

## Как проверять ошибки

Официальные рекомендации Go советуют не строить хрупкие тесты на сравнении текста ошибки, если вас интересует именно её программный смысл. Правильнее разделять “сообщение для человека” и “семантику для кода”. Отсюда следует практическое правило: если ошибка является частью контракта, её обычно лучше представлять как sentinel error или типизированную ошибку и проверять через `errors.Is` / `errors.As`, а не через `err.Error() == ...`. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

Строковое сравнение допустимо лишь тогда, когда сам текст является частью проверяемого поведения, например когда вы действительно хотите убедиться, что сообщение содержит имя аргумента или определённый фрагмент описания. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

## Как проверять результаты

Go Wiki рекомендует сравнивать **полные структуры**, а не проверять каждое поле по отдельности, если можно собрать ожидаемое значение целиком и сравнить его одним шагом. Для сложных структур в рекомендациях Go предлагается предпочитать `cmp.Equal` / `cmp.Diff`, а не `reflect.DeepEqual`, поскольку `cmp` даёт более удобные и устойчивые различия. Кроме того, следует сравнивать именно **стабильные, семантически значимые результаты**, а не случайные детали форматирования вроде точной сериализации JSON, если формат не обещан контрактом. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

Это важный ментальный сдвиг: тест должен быть не “детектором изменений”, а “детектором нарушения контракта”.

## Как писать сообщения об ошибках

Официальные рекомендации Go советуют строить failure messages так, чтобы в них были:

- имя проверяемой функции;
    
- входные данные, если они короткие и существенные;
    
- порядок `got` перед `want`. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))
    

Хороший шаблон:

```go
t.Errorf("NewObject(%q, %+v) = %v, want %v", kind, p, got, want)
```

или, для ошибок:

```go
t.Fatalf("NewObject(%q, %+v) error = %v, want %v", kind, p, err, wantErr)
```

Это кажется мелочью, но на live coding такие сообщения сразу показывают зрелость инженерного мышления.

## Errorf versus Fatalf

В `testing.T` методы `Errorf` и `Fatalf` различаются по управлению выполнением. `Errorf` логирует ошибку и помечает тест как failed, но выполнение продолжается; `Fatalf` эквивалентен `Logf` с последующим `FailNow`, то есть останавливает текущий тест немедленно. Практическое правило простое: используйте `Fatalf`, когда дальнейшие проверки бессмысленны или небезопасны, и `Errorf`, когда вы хотите собрать несколько независимых нарушений в одном прогоне. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

Внутренне это можно помнить так: **Fatalf — для разрушенного контекста, Errorf — для испорченного результата**.

## Роль subtests

`T.Run` позволяет создавать subtests, а это даёт три сильных преимущества: читаемые имена сценариев, выборочный запуск через `go test -run`, и контроль параллелизма. Документация Go подчёркивает, что subtests могут использоваться и для управления параллельностью, а родительский тест завершится только после завершения дочерних subtests. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

Для live coding это означает следующее: даже если тест маленький, `t.Run` часто стоит использовать уже ради ясных названий кейсов.

## Параллелизм в тестах

`T.Parallel` позволяет запускать тест параллельно с другими параллельными тестами. Но это не просто “ускоритель”, а изменение модели изоляции: любые общие ресурсы должны стать безопасными. Официальная документация отдельно предупреждает, что `Setenv` нельзя использовать в параллельных тестах или у тестов с параллельными предками, потому что переменные окружения общие для всего процесса. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

Отсюда практическое правило: параллелизм добавляется только тогда, когда тест уже детерминирован и изолирован.

## Вспомогательные механизмы, которые снижают хаос

`T.Helper` помечает функцию как test helper, и тогда её собственная строка не засоряет место ошибки: отчёт будет указывать на вызывающий код. Это особенно полезно, когда вы выносите повторяющиеся проверки в маленькие helper-функции. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

`T.Cleanup` регистрирует код завершения и запускает его после окончания теста и всех его subtests, причём в LIFO-порядке. Это делает логику очистки локальной и структурной. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

`T.TempDir` создаёт временную директорию, автоматически удаляемую после теста. Это избавляет от ручной уборки файловых артефактов. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

`T.Setenv` временно меняет переменную окружения и автоматически восстанавливает её через cleanup. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

С инженерной точки зрения эти методы формируют правильную привычку: **тест должен замыкать свои эффекты внутри самого себя**.

## Fuzzing как расширение ментальной модели

Fuzzing в Go встроен в `go test` и ориентирован на случаи, когда проверяются не отдельные примеры, а свойства, которые должны выполняться на большом пространстве входов. Документация Go описывает fuzzing как технику автоматического поиска неожиданных входов, приводящих к ошибкам, и подчёркивает, что fuzzing особенно полезен для нахождения edge cases и уязвимостей. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

Практически это означает: если unit test отвечает на вопрос “работает ли на этих образцах?”, то fuzz test отвечает на вопрос “не нарушается ли инвариант на произвольных входах?”. В ментальной модели это уже следующий уровень зрелости.

## Benchmarks как отдельный класс проверки

Benchmark-функции имеют форму `BenchmarkXxx(*testing.B)` и запускаются через `go test -bench`. В актуальной документации Go новые benchmarks рекомендуется писать через `b.Loop()`, а не через старый `b.N`-style, поскольку этот способ объявлен более надёжным и эффективным. Benchmarks — это не unit tests и не должны смешиваться в голове с проверкой корректности: они отвечают не на вопрос “верно ли?”, а на вопрос “сколько стоит?”. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

Для live coding это вторично, но как часть общей картины важно.

## Example tests как средство проверки документации

Example functions с комментариями `Output:` или `Unordered output:` исполняются и сверяются с ожидаемым выводом. Это особый тип теста, полезный тогда, когда вы хотите, чтобы пример использования был одновременно документацией и исполнимой проверкой. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

## Особенность цикла и subtests

Исторически в Go одной из частых ошибок при `for range` и subtests была случайная захваченная loop variable. Go blog описывает эту проблему, а начиная с Go 1.22 новая семантика цикла устраняет этот распространённый класс ошибок для нового кода. В современных проектах это снижает риск, но при чтении старых кодовых баз и mixed-version окружений всё равно полезно держать этот источник багов в уме. ([Go](https://go.dev/blog/loopvar-preview "Fixing For Loops in Go 1.22 - The Go Programming Language"))

## Ментальная модель именно для live coding interview

Для интервью наиболее практичен следующий алгоритм.

### Шаг 1. Назвать контракт вслух

Например: “Функция должна создать `Base`, создать `Child` или вернуть ожидаемую ошибку для неизвестного типа”.

Это сразу показывает, что вы мыслите поведением, а не синтаксисом.

### Шаг 2. Разбить контракт на минимальные сценарии

Обычно это:
- happy path №1;
- happy path №2, если поведение действительно отличается;
- error case.

Этого набора почти всегда достаточно, чтобы показать системность.

### Шаг 3. Для каждого сценария определить оракул

Что именно будет доказательством корректности:
- конкретный тип;
- конкретные поля;
- `errors.Is`;
- инвариант;
- отсутствие побочного эффекта.

### Шаг 4. Выбрать форму теста

Если сценарии похожи по структуре, берите table-driven test с `t.Run`. Это в Go выглядит естественно и профессионально. ([Go](https://go.dev/blog/subtests "Using Subtests and Sub-benchmarks - The Go Programming Language"))

### Шаг 5. Сначала написать assertions на ошибку

Потом — на полезный результат. Такой порядок снижает путаницу в ветвлениях.

### Шаг 6. Использовать `Fatalf`, если без ранней проверки дальше идти нельзя

Например, если type assertion не прошёл, бессмысленно читать поля. Это типичный момент для `t.Fatalf`, а не `t.Errorf`. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

### Шаг 7. Сделать failure messages диагностическими

Нужно, чтобы по одной строке было видно: какой вход, какая функция, что получили, что ожидали. Официальные рекомендации Go прямо поддерживают такой стиль сообщений. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

## Канонический шаблон мышления

Его удобно запомнить как формулу:

**Contract → Cases → Oracle → Form → Assertions**

Расшифровка:
- `Contract`: что функция обещает;
- `Cases`: на какие независимые сценарии это обещание распадается;
- `Oracle`: по какому критерию я узнаю истину;
- `Form`: table, subtests, fuzz, benchmark;
- `Assertions`: в каком порядке и с какой степенью жёсткости это проверять.

Это и есть наиболее практичная ментальная модель для Go.

## Что чаще всего ломает хорошие тесты

1. тест проверяет реализацию, а не контракт.
2. тест сравнивает нестабильный вывод, например точную сериализацию, хотя контракт обещает лишь семантическую эквивалентность. Go Wiki отдельно советует сравнивать стабильные результаты. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))
3. тест сравнивает ошибки по строкам, хотя важна их программная семантика. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))
4. таблица тестов превращается в сложный интерпретатор с десятком флагов и условных ветвей. В таких случаях лучше разбивать тесты. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

Пятая проблема — сообщения об ошибках не позволяют быстро понять, что именно сломалось. Официальные рекомендации Go предлагают явно печатать функцию, вход и порядок `got`/`want`. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))

## Практические правила высокого приоритета

1. Сначала тестируйте контракт, а не внутреннее устройство.
2. Если кейсов много и логика одна, используйте table-driven test; если логика разная, разделяйте тесты. ([Go](https://go.dev/blog/subtests "Using Subtests and Sub-benchmarks - The Go Programming Language"))
3. Ошибки проверяйте по семантике, а не по строке, если строка не является частью контракта. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))
4. Сравнивайте стабильные результаты, а не случайные детали форматирования. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))
5. Сообщения об ошибках делайте диагностическими: функция, вход, `got`, `want`. ([Go](https://go.dev/wiki/TestComments "Go Wiki: Go Test Comments - The Go Programming Language"))
6. Используйте `Helper`, `Cleanup`, `TempDir`, `Setenv` для локализации тестовой инфраструктуры. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))
7. Параллелизм включайте только в изолированных тестах. `Setenv` в параллельных тестах недопустим. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))
    

## Сжатый operational checklist для интервью

Перед тем как писать тест, быстро прогоняйте в голове:

- Какой у функции контракт?
    
- Какие у него независимые ветви?
    
- Что будет оракулом для каждой ветви?
    
- Нужен один test, table-driven test или subtests?
    
- Где нужен `Fatalf`, а где достаточно `Errorf`?
    
- Какое сообщение об ошибке сразу объяснит сбой?
    

Если на эти шесть вопросов есть ответ, потеряться на live coding уже значительно труднее.

## Итог

Зрелая ментальная модель тестирования в Go строится не вокруг набора трюков из `testing`, а вокруг дисциплины контрактного мышления. Стандартный пакет `testing` даёт для этого почти всё необходимое: именованные test functions в `*_test.go`, same-package и `_test` package режимы, `t.Run` для subtests, `t.Helper`, `t.Cleanup`, `t.TempDir`, `t.Setenv`, fuzzing, benchmarks и examples. Но центральная практическая идея остаётся одной и той же: **сначала формализовать обещание кода, затем разложить его на сценарии, выбрать оракулы и только после этого писать assertions**. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))
