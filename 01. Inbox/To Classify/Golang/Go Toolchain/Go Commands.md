# 1. Go Commands

> <font color="#fb4934">Go Commands</font> — это совокупность команд командной строки, доступных через исполняемый файл `go`, которые образуют стандартный интерфейс управления Go-проектом: компиляцией, запуском, тестированием, форматированием, модульной структурой и зависимостями.

В инженерном смысле `go` CLI является не отдельной утилитой для одной операции, а фасадом над Go toolchain. Через него разработчик взаимодействует с исходным кодом, пакетами, модулями, кэшем сборки, тестовым рантаймом, форматтером и вспомогательными инструментами. Официальная документация Go определяет `go` как программу, которая управляет Go source code и запускает остальные команды toolchain; типичная форма вызова имеет вид `go <command> [arguments]`. ([pkg.go.dev](https://pkg.go.dev/cmd/go "go command - cmd/go - Go Packages"))

`Go Commands` находятся между уровнем исходного кода и уровнем компилятора/runtime-инструментов. Разработчик обычно не вызывает компилятор, линкер, `gofmt`, `cover` или низкоуровневые инструменты напрямую; вместо этого он использует унифицированные команды `go build`, `go run`, `go test`, `go fmt`, `go mod` и другие подкоманды. Официальная документация описывает этот слой как набор программ для сборки и обработки Go-кода, которые обычно вызываются через программу `go`, а не напрямую. ([Go](https://go.dev/doc/cmd "Command Documentation - The Go Programming Language"))

## 2. Историческое развитие и эволюция подхода

Go изначально проектировался не только как язык, но и как экосистема с предписанным рабочим циклом. Поэтому значительная часть повседневной разработки была встроена в стандартный toolchain: форматирование, сборка, тестирование, документация, статические проверки и работа с пакетами.

Ранний Go-workflow был тесно связан с `GOPATH`: код размещался в определённой директории workspace, а импортные пути и зависимости интерпретировались через эту структуру. Такой подход упрощал начальную модель, но плохо масштабировался для версионирования зависимостей и воспроизводимых сборок.

Существенный перелом произошёл с появлением Go modules. В Go 1.11 была добавлена предварительная поддержка modules как альтернативы `GOPATH` с интегрированной поддержкой версионирования и распространения пакетов. ([Go](https://go.dev/doc/go1.11?utm_source=chatgpt.com "Go 1.11 Release Notes")) Позднее модульная модель стала основной практикой управления зависимостями: модуль определяется через `go.mod`, содержит набор пакетов, имеет module path и dependency requirements. ([Go](https://go.dev/ref/mod "Go Modules Reference - The Go Programming Language"))

Эволюция `Go Commands` показывает характерную особенность Go: инфраструктурные решения не выносятся полностью во внешние инструменты, а по возможности закрепляются в стандартном CLI. Поэтому форматирование, тестирование, dependency management и сборка воспринимаются не как набор произвольных соглашений проекта, а как часть языка и его инженерной культуры.

## 3. Формальные свойства, структура и ключевые признаки

### 3.1. Единая точка входа

`go` CLI задаёт единую точку входа для большинства операций над проектом. Команды имеют общую синтаксическую форму: `go <command> [arguments]`.

Это снижает фрагментацию workflow. Вместо отдельного выбора build system, test runner, formatter и dependency manager базовый Go-проект уже имеет стандартные команды для основных стадий разработки.

### 3.2. Package-oriented модель

Go-команды работают прежде всего с пакетами, а не с произвольными файлами. Даже когда команда принимает список `.go`-файлов, она интерпретирует их как часть одного пакета из одной директории. Это важно для понимания `go build`, `go test`, `go fmt` и `go run`.

Пакет в Go — это не просто директория с файлами, а компилируемая единица. Официальное руководство описывает package как коллекцию исходных файлов в одной директории, которые компилируются вместе. ([Go](https://go.dev/doc/code "How to Write Go Code - The Go Programming Language"))

### 3.3. Module-aware поведение

Современные команды `go build`, `go run`, `go test` и другие выполняются в контексте модуля, если проект содержит `go.mod`. Модуль задаёт корень dependency graph, module path, версии зависимостей и область видимости пакетов внутри проекта.

`go mod` не является единственным местом, где работает модульная система. Поддержка модулей встроена во все основные команды `go`; сама документация подчёркивает, что module support встроен не только в `go mod`, но и в остальные команды. ([pkg.go.dev](https://pkg.go.dev/cmd/go "go command - cmd/go - Go Packages"))

### 3.4. Разделение команд по типу операции

Основные команды повседневного цикла можно разделить на пять групп:

|Команда|Основная функция|Тип операции|
|---|--:|---|
|`go build`|Компиляция пакетов и зависимостей|Build operation|
|`go run`|Компиляция и немедленный запуск `main`-пакета|Execution shortcut|
|`go test`|Сборка и запуск тестовых бинарников|Verification workflow|
|`go fmt`|Форматирование исходного кода|Code normalization|
|`go mod`|Управление модулем и зависимостями|Dependency/module maintenance|

Это разделение удобно для практического мышления: `build` проверяет собираемость и производит бинарник, `run` нужен для быстрого запуска, `test` проверяет поведение, `fmt` нормализует исходный код, `mod` управляет dependency context.

## 4. Основные команды

### 4.1. `go build`

`go build` компилирует указанные пакеты вместе с их зависимостями, но не устанавливает результат в install-директорию. Если собирается один `main`-пакет, результатом является исполняемый файл; если собирается non-main package или несколько пакетов, команда выполняет проверку собираемости и отбрасывает объектный результат. При обычной сборке файлы `*_test.go` игнорируются. ([pkg.go.dev](https://pkg.go.dev/cmd/go "go command - cmd/go - Go Packages"))

Практический смысл `go build`:

- проверить, что пакет или весь проект компилируется;
    
- получить исполняемый бинарник для `package main`;
    
- выполнить сборку с флагами вроде `-o`, `-race`, `-tags`, `-trimpath`;
    
- использовать команду в CI как быстрый structural check до запуска тестов.
    

Пример:

```bash
go build ./...
go build -o bin/app ./cmd/app
go build -race ./...
```

`go build ./...` обычно используется как проверка всех пакетов текущего модуля и его подпакетов. `go build -o bin/app ./cmd/app` явно задаёт путь итогового бинарника. `go build -race ./...` включает race detector для поддерживаемых платформ.

### 4.2. `go run`

`go run` компилирует и запускает указанный `main`-пакет. Обычно пакет задаётся как `go run .`, путь к директории, import path или список `.go`-файлов из одной директории. Команда предназначена для быстрого запуска программы без отдельного ручного шага запуска бинарника. ([pkg.go.dev](https://pkg.go.dev/cmd/go "go command - cmd/go - Go Packages"))

Практический смысл `go run`:

- быстро запустить локальное приложение;
    
- проверить поведение небольшого `main`-пакета;
    
- выполнить одноразовую служебную программу;
    
- запустить external command через module-aware механизм, например с version suffix.
    

Пример:

```bash
go run .
go run ./cmd/api
go run ./cmd/migrate
go run example.com/tool@latest
```

Важно различать `go run` и `go build`: `go run` оптимизирован для удобства запуска, а `go build` — для явного получения артефакта сборки. Для production-сборки, контейнерного образа или release pipeline обычно используют `go build`, а не `go run`.

### 4.3. `go test`

`go test` автоматизирует тестирование Go-пакетов. Команда пересобирает пакет вместе с файлами, соответствующими шаблону `*_test.go`, и запускает отдельный тестовый бинарник для каждого указанного пакета. Тестовые файлы могут содержать test functions, benchmark functions, fuzz tests и example functions. ([pkg.go.dev](https://pkg.go.dev/cmd/go "go command - cmd/go - Go Packages"))

Пакет `testing` предназначен для совместного использования с `go test`: тестовые функции имеют форму `func TestXxx(*testing.T)`, а файлы тестов имеют суффикс `_test.go`. ([pkg.go.dev](https://pkg.go.dev/testing "testing package - testing - Go Packages"))

Практический смысл `go test`:

- запуск unit-тестов и integration-тестов;
    
- запуск benchmark-тестов через `-bench`;
    
- сбор coverage-профиля через `-cover` и `-coverprofile`;
    
- проверка data races через `-race`;
    
- получение машинно-читаемого вывода через `-json`.
    

Пример:

```bash
go test ./...
go test -v ./internal/service
go test -race ./...
go test -cover ./...
go test -bench=. ./internal/cache
```

`go test ./...` является базовой командой для CI. `go test -race ./...` дороже по времени выполнения, но полезен для поиска data races в конкурентном коде. `go test -bench=.` запускает benchmarks, но их результаты нужно интерпретировать осторожно: на них влияют CPU, фоновые процессы, настройки окружения и стабильность входных данных.

### 4.4. `go fmt`

`go fmt` форматирует исходники пакетов, фактически запуская `gofmt -l -w` для указанных пакетов и печатая имена изменённых файлов. Для более специальных опций нужно вызывать `gofmt` напрямую. ([pkg.go.dev](https://pkg.go.dev/cmd/go "go command - cmd/go - Go Packages"))

Практический смысл `go fmt`:

- привести код к стандартному формату Go;
    
- устранить форматирование как предмет code review;
    
- обеспечить единый стиль в команде;
    
- встроить форматирование в pre-commit hook или editor integration.
    

Пример:

```bash
go fmt ./...
```

`go fmt` выражает важную инженерную норму Go: форматирование не является областью индивидуального вкуса. Единый формат снижает шум в diff, упрощает ревью и делает кодовую базу визуально однородной.

### 4.5. `go mod`

`go mod` предоставляет операции для обслуживания модулей. В него входят подкоманды `download`, `edit`, `graph`, `init`, `tidy`, `vendor`, `verify`, `why`. ([pkg.go.dev](https://pkg.go.dev/cmd/go "go command - cmd/go - Go Packages"))

Практический смысл `go mod`:

- создать новый модуль через `go mod init`;
    
- синхронизировать `go.mod` и `go.sum` через `go mod tidy`;
    
- скачать зависимости в module cache через `go mod download`;
    
- проверить содержимое загруженных модулей через `go mod verify`;
    
- объяснить необходимость зависимости через `go mod why`;
    
- построить граф зависимостей через `go mod graph`.
    

Пример:

```bash
go mod init example.com/project
go mod tidy
go mod download
go mod verify
go mod why example.com/dependency
```

`go mod tidy` является одной из ключевых команд повседневной работы. Она добавляет недостающие зависимости и удаляет неиспользуемые из `go.mod` / `go.sum`, приводя module metadata в соответствие с фактическими импортами проекта.

## 5. Современное значение и инженерная интерпретация

В современной Go-разработке `Go Commands` выполняют роль стандартного project workflow. Они задают минимальный набор операций, который одинаково понятен локальной среде разработки, CI/CD, Docker-сборке, monorepo-структуре и production-oriented backend-проекту.

Для backend-разработки на Go этот слой особенно важен по нескольким причинам.

Во-первых, `go build` и `go test` образуют базовую линию качества: проект должен собираться, тесты должны запускаться, race-sensitive участки должны проверяться отдельными режимами.

Во-вторых, `go fmt` фиксирует единый стиль исходного кода. Это снижает стоимость сопровождения, потому что форматирование перестаёт быть локальным соглашением команды.

В-третьих, `go mod` делает dependency graph явным. Для production-сервисов это критично: зависимости должны быть воспроизводимыми, проверяемыми и понятными при обновлениях.

В-четвёртых, `go run` поддерживает быстрый feedback loop. Он удобен для локального запуска API-сервера, мигратора, CLI-утилиты или небольшого диагностического инструмента.

## 6. Логико-формальная модель

На формальном уровне `Go Commands` можно рассматривать как операции над состоянием проекта.

Проект задаётся набором исходных файлов, пакетной структурой, модульной метаинформацией, dependency graph, build cache и внешним окружением. Команды `go` преобразуют это состояние или проверяют его свойства.

- `go build`: отображает исходные пакеты и зависимости в build artifacts или в факт успешной компиляции.
    
- `go run`: отображает `main`-пакет во временный build artifact и затем в процесс выполнения.
    
- `go test`: отображает пакет и его тестовые файлы в тестовый бинарник и результат проверки.
    
- `go fmt`: отображает исходный код в канонически отформатированную форму.
    
- `go mod`: преобразует или анализирует module metadata и dependency graph.
    

Такой взгляд полезен на интервью: команды не являются случайным набором CLI-операций. Каждая из них работает с определённым уровнем модели Go-проекта: source files, packages, modules, binaries, tests, dependency graph.

## 7. Практические применения

### 7.1. Локальная разработка

Типичный локальный цикл backend-разработчика выглядит так:

```bash
go fmt ./...
go test ./...
go run ./cmd/api
```

Этот цикл покрывает три базовые проверки: код приведён к стандартному формату, тесты проходят, приложение запускается.

### 7.2. CI/CD pipeline

В CI команды обычно разделяются на независимые стадии:

```bash
go mod download
go fmt ./...
go test ./...
go build -o app ./cmd/app
```

В production pipeline `go build` часто запускается внутри Docker multi-stage build. При этом итоговый runtime-образ может содержать только собранный бинарник и минимальное окружение, без исходников и полного toolchain.

### 7.3. Работа с несколькими `main`-пакетами

В backend-проектах часто используется структура с несколькими entrypoints:

- `./cmd/api`: HTTP/gRPC API server;
    
- `./cmd/worker`: background worker;
    
- `./cmd/migrate`: migration runner;
    
- `./cmd/cli`: служебная CLI-утилита.
    

Для такой структуры команды становятся способом адресовать конкретный executable target:

```bash
go run ./cmd/api
go build -o bin/worker ./cmd/worker
go test ./...
```

### 7.4. Проверка конкурентного кода

Для сервисов с goroutines, shared state, caches, background workers и fan-out/fan-in patterns важно периодически запускать тесты с race detector:

```bash
go test -race ./...
```

Эта команда не доказывает отсутствие всех конкурентных ошибок, но помогает обнаружить data races, которые проявляются при конкретных путях выполнения тестов.

### 7.5. Обслуживание зависимостей

После добавления или удаления импортов обычно выполняется:

```bash
go mod tidy
go test ./...
```

Такой порядок позволяет сначала синхронизировать module metadata, а затем проверить, что проект остаётся собираемым и тесты проходят.

## 8. Типичные ошибки, заблуждения и ограничения

### 8.1. Смешение `go build` и `go run`

`go run` не является заменой production-сборки. Он удобен для быстрого запуска, но release artifact должен создаваться через `go build`, потому что production pipeline должен явно контролировать имя бинарника, build flags, target OS/architecture, debug information, reproducibility и содержимое итогового контейнера.

### 8.2. Ожидание, что `go build` учитывает тестовые файлы

Обычная сборка игнорирует `*_test.go`. Поэтому успешный `go build ./...` не означает, что тестовый код компилируется. Для проверки тестовых файлов нужен `go test ./...`.

### 8.3. Использование `go fmt` как optional style tool

В Go `go fmt` не является косметической рекомендацией. Это стандартная часть workflow. Неформатированный код в Go-проекте воспринимается как нарушение базовой инженерной дисциплины.

### 8.4. Ручное редактирование `go.sum` без понимания module verification

`go.sum` содержит контрольные данные для проверки модулей. Его не следует воспринимать как мусорный lock-файл, который можно произвольно чистить. Обычно корректное состояние достигается через `go mod tidy`, `go mod download`, `go test` и нормальный module-aware workflow.

### 8.5. Слепое использование `go test ./...` как полной гарантии качества

`go test ./...` проверяет только то, что покрыто тестами и достижимо в текущей конфигурации. Он не заменяет race checks, integration tests, static analysis, load testing, contract testing и observability в production.

### 8.6. Непонимание package patterns

Паттерн `./...` означает текущий пакет и подпакеты. Он не всегда эквивалентен «весь репозиторий», особенно если в репозитории несколько модулей, nested `go.mod` или workspace-структура.

## 9. Связанные понятия

## 10. Итоговое определение

`Go Commands` — это стандартный командный интерфейс Go-экосистемы, через который исходный код превращается в проверяемые пакеты, исполняемые бинарники, тестовые запуски, отформатированные файлы и воспроизводимую модульную структуру. Его значение состоит не только в удобстве CLI, но и в унификации инженерного workflow: Go-проект по умолчанию имеет встроенный способ сборки, запуска, тестирования, форматирования и управления зависимостями.

## Краткое определение области

`Go Commands` - это обзорная ветка про основные команды CLI `go`, через которые обычно происходит повседневная работа с Go-проектом: сборка, запуск, тестирование, форматирование и управление модулями.

## Что входит в эту тему

## Как устроена ветка

- `go build` отвечает за сборку пакетов и исполняемых файлов.
- `go run` нужен для быстрого запуска программы без отдельного ручного шага исполнения бинарника.
- `go test` является входом в testing workflow, но сама модель тестирования раскрывается шире в `[[Go Testing]]`.
- `go fmt` выражает встроенную норму форматирования в экосистеме Go.
- `go mod` управляет модульной структурой и зависимостями, но опирается на более общую тему `[[Go Packages and Modules]]`.

## Рекомендуемый маршрут чтения

1. Начать с `[[go build]]` и `[[go run]]`, чтобы понять базовый цикл сборки и запуска.
2. Затем перейти к `[[go test]]` и сопоставить его с `[[Go Testing]]`.
3. После этого читать `[[go fmt]]` как часть idiomatic workflow.
4. Завершить `[[go mod]]`, если нужен dependency и module context.
