# 1. Machine Code

> <font color="#fb4934">Machine Code (машинный код)</font> — это набор низкоуровневых, архитектурно-зависимых инструкций конкретной архитектуры набора команд (`[[Instruction Set Architecture]]`, ISA), представленный в виде закодированных бинарных последовательностей, предназначенных для непосредственного чтения, декодирования и исполнения аппаратным обеспечением центрального процессора (CPU).

![[computer-system-abstraction-hierachy.jpg|Computer System Abstraction Hierarchy and Machine Code placement in it]]

В строгом смысле, machine code представляет собой последовательность битов или байтов, которая интерпретируется аппаратным декодером не как абстрактные данные или текст, а как структурированный набор машинных инструкций и связанных с ними полей операндов (<i>operand fields</i>). К таким полям относятся коды операций (<i>opcodes</i>), идентификаторы регистров, непосредственные значения (<i>immediates</i>), адресные смещения и иные элементы, определённые правилами кодирования (<i>encoding</i>) в рамках целевой ISA.

В контексте программной трансляции ([[Compilation]]) машинный код выступает конечным результатом фазы генерации кода (`[[Code Generation]]`). Именно на этом уровне архитектурно-независимые промежуточные представления (Intermediate Representation, IR) трансформируются в архитектурно-специфическое исполнимое содержимое. При этом машинный код не тождественен исполнимому файлу (`[[Executable Binary]]`): первый обозначает исключительно поток машинных инструкций, тогда как второй является файловым контейнером операционной системы, включающим секции, заголовки, таблицы импорта/экспорта и метаданные.

# 2. Этимология и происхождение термина

Термин образован слиянием слов «machine» (машина, аппаратный механизм) и «code» (код, система символов для передачи информации) и буквально обозначает кодировку, предназначенную для восприятия вычислительной машиной, а не человеком. Исторически данное понятие закрепилось для концептуального противопоставления бинарного представления программы её символьным формам записи: изначально — ручному описанию вычислительных операций в числовом виде, затем — языкам ассемблера (assembly language), а в дальнейшем — высокоуровневым языкам программирования.

# 3. Историческое развитие и эволюция концепта

Эволюция формирования, хранения и доставки машинного кода отражает процесс повышения уровней абстракции в вычислительной технике:
- <b>Аппаратная коммутация и ручное кодирование</b>: На ранних этапах развития вычислительных систем (например, ЭНИАК) алгоритмы задавались физическим изменением топологии цепей. Позднее машинный код стал практически единственной формой программирования: инструкции вводились в машинно-числовом (бинарном или шестнадцатеричном) виде через переключатели, перфокарты, перфоленты. В этом контексте машинный код выступал одновременно формой хранения программы, средством её написания и формой непосредственного исполнения.
- <b>Символическая абстракция (появление ассемблеров)</b>: Разработка языков ассемблера не привела к устранению машинного кода, но добавила поверх него человеко-читаемую символическую нотацию (мнемоники). Это разделение стало фундаментальным этапом: язык ассемблера специфицирует архитектурные действия в текстовом виде, в то время как машинный код остаётся их фактическим бинарным воплощением (encoding).
- <b>Автоматическая трансляция (развитие компиляторов)</b>: С появлением высокоуровневых языков роль машинного кода сместилась из области прямого ручного конструирования в область автоматизированной backend-трансляции. Прохождение исходного кода через `[[Program Translation Pipeline]]` сделало машинный код целевым артефактом (target artifact), генерируемым для выбранной платформы.
- <b>Эволюция схем кодирования (Instruction Encoding)</b>: Развитие различных ISA привело к диверсификации форматов инструкций. Ряд архитектур (CISC) сделал ставку на плотные, вариативные форматы, тогда как другие (RISC) перешли к регулярным и формально предсказуемым схемам. Несмотря на эти различия, машинный код сохранил статус нижнего программно-значимого уровня перед микроархитектурной (microarchitectural) реализацией.
- <b>Динамическая генерация</b>: В современной инженерной практике концепт машинного кода расширился в область runtime-систем. Код больше не ограничивается статической (Ahead-of-Time) генерацией, а создаётся динамически во время исполнения: JIT-компиляция (Just-In-Time), runtime patching и специализированные системы профилированной оптимизации.

# 4. Уровень абстракции и архитектурные границы

<font color="#fb4934">Машинный код позиционируется на стыке между архитектурной семантикой программы и физической реализацией её исполнения внутри процессора.</font> 

![[computer-system-abstraction-hierachy.jpg|Computer System Abstraction Hierarchy and Machine Code placement in it]]

Его место в стеке трансляции можно строго зафиксировать:

* <b>Выше машинного кода</b>: исходный код (source code), промежуточные представления (IR), символьная запись assembly, а также фазы трансляции компилятора (выбор инструкций, аллокация регистров, layout-построение памяти).
* <b>Ниже машинного кода</b>: аппаратный тракт процессора (fetch/decode/execute path), конвейер (pipeline), исполнительные устройства (execution units), внутренние микрооперации (micro-ops) и иные механизмы микросхемной кремниевой реализации.

> [!warning] Машинный код описывает исключительно архитектурный поток процессорных инструкций
> Он не отвечает за системную оркестрацию памяти (задача операционной системы и системного загрузчика) и не должен трактоваться как «электрический сигнал процессора». Это последняя программно-значимая форма инструкции, выступающая интерфейсом к аппаратному интерпретатору.

# 5. Формальные свойства, структура, ключевые признаки

> [!important] 
> <font color="#fb4934">Ключевое свойство машинного кода</font> — абсолютная и строгая зависимость от целевой архитектуры (ISA). Одинаковая high-level программа после трансляции для архитектур x86-64 и AArch64 (ARM) представляет собой принципиально разные последовательности двоичных данных, которые, сохраняя семантику на уровне исходной задачи, не совпадают ни по бинарному представлению, ни по допустимым формам инструкций (instruction forms).

<font color="#d3869b">Фундаментальные признаки машинного кода:</font>
- <font color="#d79921">Архитектурная специфицированность</font>: Допустимость, размер и интерпретация инструкции всецело определяются правилами конкретной ISA, а не универсальными свойствами вычислимых функций.
- <b>Бинарная кодировка</b>: Отсутствие текстовой репрезентации; инструкции представлены битовыми полями, подлежащими аппаратному декодированию.
- <b>Последовательная исполнимость</b>: <font color="#d3869b">Логика организуется как поток инструкций (instruction stream), навигация по которому осуществляется посредством продвижения архитектурного программного счётчика (architectural program counter / IP).</font>
- <b>Семантическая привязка к architectural state</b>: Каждая атомарная инструкция осуществляет трансформацию строго определённых архитектурно-видимых компонентов: регистров, ячеек памяти, флагов состояния (flags).
- <b>Отсутствие семантических абстракций высокого уровня</b>: В машинном коде отсутствуют концепции типов данных, объектов, структур или управляющих конструкций (циклов, исключений). Память рассматривается как плоский массив байтов, управление потоком выполнения — как графы переходов (jumps/branches).
- <b>Ограниченная переносимость</b>: Машинный код фундаментально непереносим между различными архитектурами. Более того, он часто непереносим между различными средами исполнения (execution environments) одной архитектуры без соблюдения бинарных интерфейсов приложений (ABI), конвенций вызовов (calling convention) и платформенных ограничений (platform constraints).

<b>Структурно машинный код формируется из закодированных инструкций (encoded instructions), каждая из которых может включать следующие элементы:</b>
- <b>Opcode (код операции)</b>: Битовое поле или префикс, детерминирующее класс выполняемой аппаратной операции.
- <b>Operand specifiers (спецификаторы операндов)</b>: Поля, указывающие на целевые регистры, режимы адресации памяти или иные параметры операндов.
- <b>Immediates и displacements (непосредственные значения и смещения)</b>: Встроенные константы, абсолютные или относительные адреса.
- <b>Дополнительные encoding bits</b>: Архитектурно-специфические модификаторы, такие как префиксы длины, модификаторы размера операнда, поля условного исполнения (condition fields).

<font color="#d3869b">С точки зрения топологии кодирования (encoding models) выделяют два основных подхода:</font>
- <b>Fixed-length instructions (инструкции фиксированной длины)</b>: Все команды имеют строго одинаковый размер (например, 32 бита), что радикально упрощает стадию аппаратного декодирования (decode stage) и предсказуемость границ инструкций.
- <b>Variable-length instructions (инструкции переменной длины)</b>: Размер команд варьируется (например, от 1 до 15 байт в x86), что позволяет достичь более высокой плотности кода (code density) и выразительности, однако существенно усложняет логику аппаратного декодера.

# 6. Математическое и логико-формальное ядро

С формальной точки зрения машинный код может быть описан как упорядоченная конечная последовательность байтов (или битовых векторов):

$$M = \langle b_0, b_1, \dots, b_n \rangle$$

Допустимость и семантическая интерпретация каждого фрагмента этой последовательности строго определяются функцией декодирования $D_{\text{ISA}}$, специфичной для конкретной архитектуры. Функция принимает на вход поток байтов $M$ и текущее значение архитектурного счётчика $pc$:

$$D_{\text{ISA}}(M, pc) \to (instr, pc') \lor \bot$$

Где:
- $instr$ — успешно декодированная атомарная машинная инструкция;
- $pc'$ — новое значение программного счётчика (вычисленное с учётом длины декодированной инструкции или логики control flow);
- $\bot$ — терминальное состояние, обозначающее недопустимую комбинацию битов (invalid encoding state), приводящее к аппаратному исключению (trap) или невозможности декодирования.

В контексте теории автоматов исполнение корректно декодированного машинного кода задаётся как функция перехода архитектурного состояния $S_{\text{ISA}}$. Состояние процессора и памяти $\Sigma$ трансформируется в новое состояние $\Sigma'$ под воздействием инструкции $instr$:

$$\Sigma' = S_{\text{ISA}}(\Sigma, instr)$$

Где состояние $\Sigma$ включает программный счётчик, значения всех архитектурных регистров, состояние оперативной памяти, флаги и иные архитектурно-видимые компоненты.

Данная формализация фиксирует два важнейших свойства: во-первых, машинный код — это декодируемая битовая последовательность, а не текст; во-вторых, семантика этого кода не является имманентным свойством самих битов, а возникает исключительно в связке «правила декодирования + семантика исполнения» ($D_{\text{ISA}} + S_{\text{ISA}}$).

# 7. Современное значение и междисциплинарные интерпретации

В парадигме современной вычислительной инженерии машинный код — это центральная точка сопряжения трёх фундаментальных систем:
- ISA выступает как строгий архитектурный контракт;
- Компилятор или JIT-система выступает как математический генератор (producer) инструкционного потока, удовлетворяющего контракту;
- Микропроцессор (CPU) выступает как аппаратный интерпретатор (interpreter) допустимых форм кодирования.

Для инженерии компиляторов (compiler engineering) машинный код является окончательной и метрически измеримой целью фазы backend-генерации. Для системного программирования (systems programming) машинный код представляет собой фактическую форму материи, в которой существуют и оперируют ядра операционных систем, среды выполнения (runtimes), драйверы, загрузчики (boot code) и встроенные (embedded) программы. Для обратной разработки (reverse engineering) он является первичным объектом дизассемблирования, анализа потока управления (control flow graph analysis) и восстановления семантики закрытых бинарных артефактов.

В инфраструктурном аспекте (облачные вычисления, контейнеризация, оркестрация) машинный код сохраняет базовое значение платформенной зависимости. Виртуализация не устраняет архитектурную природу кода: бинарный образ, собранный под одну систему команд, невозможно запустить на узле кластера с другой архитектурой без трансляции или полной программной эмуляции.

# 8. Практические применения

Взаимодействие с машинным кодом в явном виде применяется в следующих инженерных доменах:
- <b>Инженерия компиляторов и компоновщиков (Toolchains)</b>: Формирование бинарного потока как результат выбора инструкций (instruction selection), распределения регистров (register allocation), планирования (scheduling) и финальной эмиссии кода. Включает кросс-платформенную сборку, где выбор целевой архитектуры (target architecture) детерминирует генерацию диалекта машинного кода.
- <b>JIT-компиляция и Runtime-системы</b>: Динамическая генерация машинных инструкций в оперативной памяти в ответ на профиль исполнения или типовую специализацию (повышение скорости интерпретируемых языков).
- <b>Системное программирование</b>: Разработка ядер ОС, прошивок (firmware), гипервизоров (hypervisors) и загрузчиков (bootloaders), которые инициализируют аппаратное обеспечение и напрямую управляют процессорным контекстом.
- <b>Отладка и анализ производительности (Profiling)</b>: Профилировщики (например, perf), отладчики и дизассемблеры работают непосредственно с machine code для точного выяснения реального пути исполнения (instruction path), анализа задержек конвейера и оптимизации циклов.
- <b>Информационная безопасность и бинарный анализ (Security)</b>: Исследование вредоносного ПО (malware analysis), сканирование сигнатур, анализ возвратно-ориентированного программирования (ROP-chains) и разработка механизмов защиты (exploit mitigation).
- <b>Бинарная инструментация</b>: Внедрение дополнительных диагностических инструкций в готовый машинный код «на лету» (например, Valgrind, DynamoRIO) для мониторинга выделения памяти или профилирования без необходимости доступа к исходным текстам.

Следствием этих применений является факт непостоянства бинарного результата: один и тот же исходный код на языке высокого уровня может компилироваться в множество радикально отличающихся вариантов машинного кода (variants) в зависимости от ISA, целевого ABI, выбранного компилятора, уровня оптимизации и целевого микроархитектурного профиля.

# 9. Типичные ошибки, заблуждения, ограничения

- <b>Отождествление машинного кода и языка ассемблера</b>: Одно из наиболее распространённых заблуждений. Язык ассемблера — это текстовая, человеко-читаемая символическая форма записи; машинный код — это её исключительно бинарное кодирование.
- <b>Отождествление машинного кода и исполнимого файла</b>: Исполнимый файл (ELF, PE, Mach-O) — это сложный контейнер, содержащий машинный код лишь в специализированных секциях (например, `.text`), но также включающий таблицы символов, метаданные релокаций (relocation metadata) и заголовки ОС.
- <b>Машинный код как байт-код (Bytecode)</b>: Байт-код предназначен для интерпретации абстрактной программной виртуальной машиной (VM), в то время как машинный код предназначен для прямого исполнения кремниевой логикой физического CPU.
- <b>Иллюзия «прямого электрического действия»</b>: Машинный код часто считают непосредственным сигналом к исполнению. Однако современные суперскалярные процессоры осуществляют сложную внутреннюю трансляцию, переупорядочивание (Out-of-Order execution) и декомпозицию архитектурного машинного кода в набор внутренних микроопераций (micro-ops или macro-ops). Машинный код является финальным интерфейсом для программного стека, но не обязательно совпадает с внутренней микросхемной реализацией аппаратуры.
- <b>Ложное ожидание переносимости (Portability myth)</b>: Исполняемый поток инструкций жестко привязан к конкретной экосистеме исполнения (execution ecosystem). Попытка выполнить код, скомпилированный для одной платформы, на другой гарантированно приведет к ошибке декодирования или сегментации.
- <b>Низкая человекочитаемость</b>: Без применения процессов дизассемблирования и использования таблиц символов, «сырой» машинный код представляет собой бинарный шум, крайне сложный для анализа, сопровождения или модификации специалистом.

# 10. Связанные понятия

- [[Compilation]] — общая родительская парадигма, в рамках которой машинный код является конечным результатом backend-трансляции.
- `[[Code Generation]]` — стадия компилятора, ответственная за выбор архитектурно-специфических инструкций и формирование их бинарного encoding.
- [[Instruction Set Architecture (ISA)]] — архитектурный контракт (спецификация инструкций, регистров и семантики), задающий правила, по которым машинный код получает смысл и способность к декодированию.
- `[[Assembly Language]]` — символьная мнемоническая репрезентация программы, семантически изоморфная машинному коду, но отличающаяся текстовой формой.
- `[[Executable Binary]]` — системный файловый контейнер, инкапсулирующий машинный код и метаданные загрузки для конкретной операционной системы.
- `[[Program Translation Pipeline]]` — полный технологический маршрут преобразования абстрактного исходного текста в целевой аппаратный артефакт исполнения.

# 11. Краткое концептуальное резюме

Машинный код представляет собой структурно сложный архитектурный поток (instruction stream), который:
- Является финальным продуктом целевой генерации кода, завершающим этап понижения абстракций;
- Строго детерминируется синтаксическими и семантическими правилами конкретной ISA;
- Исполняется физическим процессором через цикл декодирования (decode) и применения архитектурной семантики (architectural semantics);
- Выступает ядром бинарных исполнимых артефактов, но не тождественен их полной контейнерной структуре;
- Образует фундаментальную интерфейсную границу между программными абстракциями компилятора и физическим микроархитектурным аппаратным исполнением.
