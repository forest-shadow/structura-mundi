# 1. ELF (Executable and Linkable Format)

> <font color="#fb4934">ELF (Executable and Linkable Format)</font> — это стандартизованный бинарный формат представления программных артефактов в системных средах семейства Unix и других Unix-подобных операционных системах. Он представляет собой расширяемую структуру контейнера, предназначенного для хранения машинного кода, таблиц символов, relocation-данных (данных для перемещения), метаданных загрузки и отладочной информации.

***

![[elf-01.jpg]]

**Общая компоновка ELF-файла**: хорошо показывает `ELF header`, таблицу заголовков сегментов, типичные секции вроде `.text`, `.rodata`, `.data`, `.bss`, а также секции символов и debug-данных, которые обычно не загружаются в память процесса. Это удобно для базового визуального введения в тему.

***

С формальной точки зрения ELF является спецификацией интерфейса двоичных приложений (ABI), обеспечивающей переносимость объектных файлов между различными вычислительными средами. В инженерном смысле ELF следует понимать как универсальный контейнерный формат, который задаёт согласованную внутреннюю организацию бинарного файла так, чтобы разные компоненты toolchain могли интерпретировать один и тот же артефакт в собственном режиме работы:

- Компоновщик (<i>linker</i>) интерпретирует его как объект компоновки, опираясь на секции и символы.
- Загрузчик (<i>loader</i>) интерпретирует его как описание загружаемого образа, оперируя сегментами памяти.
- Отладчик (<i>debugger</i>) интерпретирует его как носитель символической и отладочной информации.

ELF не тождественен конкретным типам файлов, таким как `[[Object File]]`, `[[Shared Library]]` или `[[Executable Binary]]`: он представляет собой общий базовый формат, в рамках которого выражаются различные по своему назначению бинарные артефакты.

# 2. Этимология и происхождение термина

Название **ELF** является акронимом от _Executable and Linkable Format_. Эта формулировка строго фиксирует двойную направленность и природу формата: он изначально спроектирован так, чтобы быть пригодным как для непосредственной загрузки в память и исполнения процессором (Executable), так и для использования в качестве входных данных редактором связей при объединении с другими модулями (Linkable).

Термин и сам стандарт были разработаны в конце 1980-х годов подразделением UNIX System Laboratories (USL) компании AT&T в рамках создания операционной системы System V Release 4 (SVR4).

# 3. Историческое развитие и эволюция концепта

Формат ELF возник как более гибкая, масштабируемая и расширяемая альтернатива ранним форматам бинарных файлов Unix, в первую очередь `a.out` и `COFF` (<i>Common Object File Format</i>). Эволюция подхода была продиктована необходимостью преодоления архитектурных ограничений `COFF`, который не обладал достаточной выразительностью для поддержки динамического связывания, разделяемых библиотек и современных переносимых ABI-соглашений.

Ключевые этапы развития:
- **Стадия становления (конец 1980-х)**: Разработка формата в AT&T для унификации представления бинарных артефактов в System V. <font color="#d3869b">Исторически наиболее важным архитектурным решением стало концептуальное разделение точек зрения на один и тот же файл (views), что сделало формат пригодным одновременно для этапов compile-time, link-time и load-time, не смешивая их в единую жесткую структуру.</font>
- **Стандартизация (1990-е)**: Принятие спецификации комитетом Tool Interface Standard (TIS) в качестве межплатформенного стандарта для 32-битных систем архитектуры Intel (x86).
- **Расширение и доминирование (2000-е — настоящее время)**: Адаптация формата для 64-битных архитектур (стандарт LP64) и его закрепление как де-факто стандарта во всех основных дистрибутивах Linux, значительной части BSD-экосистемы, Android, а также в качестве формата встроенного программного обеспечения (firmware). Параллельно ELF стал основным контейнером для стандартизированных отладочных данных, таких как [[DWARF]].

# 4. Формальные свойства, структура и ключевые признаки

Общая структурная идея ELF организует бинарный файл как строго упорядоченный контейнер с формально заданными заголовками и индексируемыми областями данных. Базовая логика формата строится вокруг трех уровней представления:

1. **ELF Header (Заголовок файла)**: Корневой заголовок, расположенный в самом начале файла (смещение 0x00). Он описывает сам файл как объект формата ELF и содержит следующие обязательные метаданные:
    - Магическое число (сигнатура): `0x7F 'E' 'L' 'F'`.
    - Класс/разрядность представления: 32-bit (`ELFCLASS32`) или 64-bit (`ELFCLASS64`).
    - Порядок байтов (Endianness): little-endian или big-endian.
    - Версия формата.
    - Тип файла (ET_REL, ET_EXEC и др.).
    - Целевая машинная архитектура (ISA).
    - Виртуальный адрес точки входа (Entry point).
    - Смещения (offsets) в файле, указывающие на начало таблиц Program Header Table и Section Header Table.
    - ABI-специфичные флаги.
2. **Section Header Table (Таблица заголовков секций)**: Задает набор секций и их характеристики. Эта структура необходима инструментарию на этапах link-time и analysis-time.
3. **Program Header Table (Таблица заголовков программ)**: Задает набор сегментов. Данная таблица является критической для load-time, так как именно она используется системным загрузчиком для построения образа процесса в виртуальной памяти.

## Модель «двойного представления» (Dual View)

<font color="#fb4934">Фундаментальным свойством ELF является строгое различие между секциями (<i>sections</i>) и сегментами (<i>segments</i>)</font>, обеспечивающее возможность интерпретировать один и тот же файл двумя способами:

- **Linking View (Представление для связывания / Секции)**: Логическая единица организации данных внутри файла. Секции ориентированы на компилятор, ассемблер и линкер. Они содержат атомарные единицы информации: код, инициализированные данные, неинициализированные данные, таблицы символов, таблицы строк и инструкции по релокации.
- **Execution View (Представление для исполнения / Сегменты)**: Единица загрузочного представления. Сегмент описывает не логическую классификацию данных, а способ отображения непрерывного фрагмента файла в адресное пространство процесса. Сегмент объединяет одну или несколько секций с одинаковыми требованиями к загрузке (адрес, выравнивание, права доступа RWX).

**Две проекции одного и того же ELF**: `linking view` и `execution view`. `sections` нужны прежде всего linker/tooling, а `segments` — loader’у и механизму отображения файла в память.

## Типология ELF-артефактов

Различие между типами файлов выражается не сменой формата контейнера, а сменой значения поля `e_type` в ELF Header и наличием/отсутствием определенных таблиц:

- **ET_REL (Relocatable file)**: Объектный файл (`.o`), содержащий код и данные, требующие связывания. Еще не приведен к конечному образу выполнения.
- **ET_EXEC (Executable file)**: Готовый к запуску исполнимый файл, содержащий точку входа и строго заданные виртуальные адреса для сегментов.
- **ET_DYN (Shared object file)**: Разделяемая библиотека (`.so`) или позиционно-независимый исполняемый файл (PIE). Может динамически подгружаться линкером времени выполнения (`ld.so`) по произвольному базовому адресу.
- **ET_CORE (Core file)**: Дамп состояния виртуальной памяти и регистров процесса, генерируемый операционной системой при критических сбоях для последующего post-mortem анализа.

## Типичные семантические классы секций

***

Несмотря на гибкость формата, спецификация и соглашения ABI определяют стандартный набор секций:

- **Кодовые секции**: `.text` (исполняемый машинный код).
- **Секции инициализированных данных**: `.data` (глобальные переменные с начальными значениями).
- **Секции read-only данных**: `.rodata` (константы, строковые литералы).
- **Секции неинициализированных данных**: `.bss` (Block Started by Symbol — занимает место только в памяти, но не в файле).
- **Таблицы символов и строки имен**: `.symtab` (статическая таблица символов), `.strtab` (строки имен), `.dynsym` и `.dynstr` (для динамического связывания).
- **Секции перемещений (Relocations)**: `.rel.text`, `.rela.data` и др.
- **Секции отладки**: `.debug_info`, `.debug_line` (обычно содержат данные стандарта `[[DWARF]]`).

## Таблицы символов и relocation-данные

Инженерная ценность ELF как промежуточного представления для build pipeline заключается в способности хранить символическую информацию и relocation-описания. Это позволяет компоновщику:

1. Сопоставлять внешние (imported) и внутренние (exported) символы между разрозненными модулями.
2. Разрешать перекрестные ссылки между объектными файлами.
3. Корректировать адресозависимые инструкции (релоцировать код) после того, как определено окончательное размещение всех секций в едином линейном адресном пространстве.

## Архитектурная и ABI-зависимость

ELF является универсальным каркасом, но содержимое конкретного файла строго привязано к целевой архитектуре (ISA) и ABI-контексту. Формат не является архитектурно-независимым представлением: секция `.text` внутри ELF для архитектуры x86-64 содержит принципиально иной машинный код, чем аналогичная секция для AArch64. Стандарт регламентирует способ упаковки и метаданные, но не абстрагирует само бинарное содержимое.

# 5. Современное значение и интерпретации

В современной парадигме системной инженерии ELF воспринимается не просто как пассивный формат хранения на диске, а как фундаментальный интерфейсный контракт между всеми слоями низкоуровневой инфраструктуры Unix-like систем:

- Компилятор (GCC, Clang) и ассемблер генерируют ELF-совместимые объектные файлы.
- Статический компоновщик (Linker) потребляет, трансформирует и объединяет ELF-структуры.
- Системный загрузчик (Loader) в ядре ОС парсит Program Headers для настройки страниц памяти процесса (через системный вызов `execve` и механизм `mmap`).
- Инструментарий динамического связывания (Runtime ecosystem) опирается на стандартизованное представление Shared Objects для разрешения зависимостей "на лету".
- Отладчики (GDB, LLDB) и профилировщики читают метаданные секций для восстановления контекста выполнения.

Современная интерпретация подчеркивает мультиплексную природу формата: один физический файл существует одновременно в нескольких аналитических измерениях (как массив байтов, как иерархия логических секций, как карта памяти, как граф символических зависимостей, как носитель ABI-контракта).

# 6. Математическое и логико-формальное ядро

В строгом логико-структурном выражении формат ELF можно представить как математическое объединение структур данных, где ядро системы образуют функции отображения (mappings).

Общая формальная структура:

$$ELF = \text{Header} + \text{PHT} + \text{SHT} + \text{Payload}$$

Где $\text{PHT}$ — Program Header Table, $\text{SHT}$ — Section Header Table, а $\text{Payload}$ — объединение непересекающихся диапазонов данных файла.

Формат задает систему инъективных и сюръективных отображений метаданных на физические смещения и виртуальные адреса:

- **Отображение секций**: функция $f_{sec}: \text{SectionID} \rightarrow [\text{offset}_{file}, \text{offset}_{file} + \text{size}]$, ставящая в соответствие идентификатору секции непрерывный блок байтов на диске.
- **Отображение сегментов**: функция $f_{seg}: \text{SegmentID} \rightarrow ([\text{offset}, \text{offset} + \text{size}]_{file}, [V_{addr}, V_{addr} + M_{size}]_{mem})$, регламентирующая проекцию байтов файла в страницы виртуальной памяти процесса.
- **Отображение символов**: связь между строковым идентификатором (именем функции или переменной) и смещением относительно начала определенной секции.
- **Отображение перемещений (Relocations)**: алгебраические правила применения смещений. Например, вычисление абсолютного адреса во время компоновки по формуле $A = S + A_{ddend}$, где $S$ — значение символа, а $A_{ddend}$ — константная добавка, извлекаемая из `.rela` секции.

С этой точки зрения ELF — это не просто сериализация данных, а декларативная система метаописаний, наделяющая «сырую» байтовую последовательность строгой семантикой для независимых программных потребителей.

# 7. Практические применения

Формат выступает базовым инфраструктурным элементом в следующих практических доменах:

- **`[[Object File]]` (Модульная компиляция)**: Хранение результатов трансляции исходного кода в машинный код на этапе до финальной компоновки. Обеспечивает инкрементальную сборку крупных проектов.
- **`[[Executable Binary]]` (Исполнение программ)**: Инкапсуляция готового программного продукта с указанием `entry point` и параметров инициализации памяти, необходимых ядру ОС для создания нового процесса.
- **`[[Shared Library]]` (Динамическое связывание)**: Реализация динамически подключаемых библиотек с семантикой импорта/экспорта символов. Позволяет нескольким процессам делить одну копию `.text` сегмента библиотеки в физической оперативной памяти, радикально экономя ресурсы.
- **Core dump (Анализ сбоев)**: Фиксация точного слепка состояния виртуальной памяти процесса, содержимого регистров процессора и стека вызовов на момент фатального исключения (Segmentation fault).
- **Debugging and Profiling (Отладка и профилирование)**: Хранение отладочной информации (чаще всего интегрированного стандарта `[[DWARF]]`), позволяющей соотносить машинные инструкции с конкретными строками исходного кода на языках высокого уровня.
- **Binary Analysis (Реверс-инжиниринг и безопасность)**: Использование утилитами статического и динамического анализа (например, `readelf`, `objdump`, дизассемблеры) для инспекции секций, таблиц релокаций, аудита безопасности и валидации ABI-характеристик.

# 8. Типичные ошибки, заблуждения и ограничения

Несмотря на широкую распространенность, понимание формата часто сопровождается концептуальными ошибками:
- **ELF не равен исполнимому файлу**: Это абстрактный контейнер. Внутри него с равным успехом может лежать как исполняемая программа, так и объектный файл, который ОС запустить не сможет.
- **ELF не описывает весь процесс компоновки**: Формат предоставляет лишь статические структуры (таблицы символов, списки релокаций). Алгоритмы разрешения конфликтов имен и оптимизации кода (LTO) остаются зоной ответственности самого компоновщика.
- **ELF не тождественен runtime loading**: Наличие сегментов (`Program Headers`) необходимо для загрузки, но полная модель инициализации процесса определяется ядром ОС, работой динамического загрузчика (`ld-linux.so`) и правилами конкретного ABI.
- **ELF не равен `[[DWARF]]`**: Это ортогональные стандарты. ELF — это формат "конверта" и системных секций, а `[[DWARF]]` — формат представления графа отладочных данных, который этот "конверт" может в себе переносить.
- **Платформенная зависимость**: ELF стандартизирован как каркас, но его содержимое не является переносимым (архитектурно-нейтральным). Файл `ET_EXEC`, собранный под Linux x86-64, требует для корректной интерпретации строго определенного окружения, соответствующего endianness, ISA и системным вызовам целевой ОС.
- **Сложность разбора**: Формат требует полноценного парсера структур и заголовков. По этой причине в крайне ограниченных средах (например, на самых ранних стадиях работы bootloader'а контроллеров) вместо ELF часто используют "плоские" бинарные файлы (raw binary), лишенные каких-либо метаданных.

# 9. Связанные понятия

- [[Program Binary Formats]]: Концептуальная макро-категория форматов представления бинарных программных артефактов (включает ELF, PE/COFF, Mach-O).
- `[[Object File]]`: Промежуточный артефакт трансляции (в рамках Unix — файл типа `ET_REL`).
- `[[Executable Binary]]`: Конечный результат работы статического компоновщика (файл типа `ET_EXEC` или позиционно-независимый `ET_DYN`).
- `[[Shared Library]]`: Динамически связываемый модуль (файл типа `ET_DYN`), загружаемый в адресное пространство процесса во время выполнения.
- `[[Linking]]`: Вычислительный процесс разрешения графа символов и применения правил релокации, опирающийся на структуры ELF.
- `[[Machine Code]]`: Архитектурно-зависимые инструкции процессора, выступающие полезной нагрузкой (payload) исполняемых секций ELF.
- `[[DWARF]]`: Формат представления отладочных метаданных, инкапсулируемый в выделенные секции ELF (например, `.debug_info`).
