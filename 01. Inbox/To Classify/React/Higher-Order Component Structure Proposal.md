# Higher-Order Component Structure Proposal

## Назначение

Этот файл фиксирует минимальную иерархию заметок для темы `Higher-Order Component` по правилам `Principia Rerum`.

Тема естественно встраивается не как отдельная корневая React-ветка, а как часть уже существующей компонентной модели React.

Для этой ветки подходит уже подтвержденная словарная рамка:

- `area: computer-science`
- `domain: frontend-engineering`
- `section: react`

## Рекомендуемая иерархия

```text
React (overview)
└── React Components (overview)
    ├── Props in React (article)
    ├── Functional Components (article)
    ├── Component Composition (article)
    ├── Class Components (article)
    └── Higher-Order Component (article)
```

## Почему структура именно такая

- `Higher-Order Component` относится к компонентной модели, потому что его главным объектом является сам компонент: HOC принимает компонент и возвращает новый компонент.
- Тема не требует отдельного `sub-overview`: одной статьи пока недостаточно, чтобы оправдать новый обзорный слой вроде `React Reuse Patterns`.
- `Higher-Order Component` не стоит растворять внутри `Component Composition`. Композиция описывает общий принцип сборки UI из компонентов, а HOC описывает отдельный паттерн обёртывания и расширения компонентного поведения.
- `Higher-Order Component` также не стоит помещать в `React Hooks`, потому что это более ранний и иной по форме паттерн переиспользования логики.
- `React.memo` полезно связывать с HOC через `related`, но не подчинять ему иерархически. Несмотря на HOC-форму, по смыслу `React.memo` живёт в ветке `React Rendering Model`, так как относится к управлению повторными рендерами.

## Что пока не нужно создавать

- отдельный `overview` уровня `React Reuse Patterns`, если под ним пока нет плотного корпуса статей;
- отдельную статью `React HOC vs Custom Hooks`, если различие можно раскрыть внутри самой статьи `Higher-Order Component`;
- отдельную ветку только ради `withRouter`, `connect` или других частных исторических примеров.

## Предлагаемое физическое размещение после нормализации

Если заметки будут доведены до канонического состояния, их физическое размещение должно следовать алфавитному правилу:

```text
02. Corpus Mundi/
├── H/
│   └── Higher-Order Component.md
├── R/
│   ├── React.md
│   └── React Components.md
```

Логическая ветка при этом всё равно собирается через `parent`, а не через физическую близость файлов.
