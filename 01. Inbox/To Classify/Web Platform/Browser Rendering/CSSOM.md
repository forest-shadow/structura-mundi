# 1. CSSOM (CSS Object Model)

> <font color="#fb4934">CSSOM (CSS Object Model)</font> — это внутреннее структурированное представление CSS-данных, формируемое браузером на основе таблиц стилей для интерпретации правил оформления, их сопоставления с узлами [[DOM]] и участия в построении [[Render Tree]]. В строгом инженерном смысле CSSOM представляет собой не просто текст CSS, а разобранную и организованную модель, состоящую из объектов таблиц стилей (stylesheets), правил (rule-objects), селекторов (selectors), деклараций (declarations) и связанных с ними стилевых зависимостей, пригодную для алгоритмического применения в процессе browser rendering.

В контексте [[Browser Rendering]] CSSOM выступает как обязательное промежуточное представление между исходными стилевыми источниками и последующими стадиями визуального конвейера. Без него браузер лишен возможности корректно определить, какие стилистические правила применимы к конкретным DOM-узлам, какие элементы исключены из визуального вывода и в каком виде должна формироваться итоговая отображаемая структура страницы.

# 2. Этимология и происхождение термина

Термин CSSOM образован по прямой аналогии с DOM и расшифровывается как **CSS Object Model** (Объектная модель CSS). Понятие «объектная модель» в данном контексте обозначает не парадигму объектно-ориентированного программирования, а формализованное представление содержимого документа или стилевой системы в виде взаимосвязанных программно доступных структур. В то время как DOM моделирует документ как дерево узлов, CSSOM моделирует стилевую составляющую документа как систему таблиц стилей, правил и деклараций.

# 3. Историческое развитие и эволюция концепта

Появление CSSOM обусловлено переходом браузерных движков от линейного текстового чтения CSS к полноценной программной и алгоритмической работе со стилями. На ранних этапах развития веба CSS воспринимался преимущественно как декларативный язык оформления. Однако усложнение логики отображения потребовало создания внутренней структуры, способной решать следующие задачи:

- Разбор CSS в машинно-обрабатываемую структуру.
- Вычисление применимости правил к элементам.
- Разрешение механизмов каскада (cascade), наследования (inheritance) и приоритетов (priority).
- Обеспечение динамических изменений стилей через JavaScript.
- Синхронизация стилевой модели с процессом построения отображаемой структуры.

Эволюция подхода развивалась по двум ключевым векторам:
1. **Стандартизация программного доступа**: создание унифицированной модели, позволяющей скриптам считывать и модифицировать таблицы стилей и вычисленные значения в рантайме.
2. **Оптимизация Rendering Pipeline**: превращение CSSOM в критически важный внутренний этап движка, напрямую влияющий на производительность парсинга, пересчета стилей (style recalculation) и инвалидацию рендеринга.

# 4. Формальные свойства, структура и ключевые признаки

## 4.1. Базовая структура и композиция

CSSOM конструируется из различных источников стилевой информации: инлайновых стилей (inline styles), встроенных блоков стилей (embedded stylesheets) и внешних файлов (external stylesheets). Типовая структура CSSOM включает следующие элементы:

- **Stylesheet**: единица подключения и обработки, представляющая отдельную таблицу стилей.
- **Rule**: объект отдельного правила внутри таблицы стилей.
- **Selector**: формальное условие применимости правила к узлам документа.
- **Declaration block**: совокупность свойств и значений, назначаемых при совпадении селектора.
- **At-rules**: конструкции типа `@media`, `@supports`, `@font-face`, вводящие дополнительные условия или описания.
- **Связи с DOM**: ассоциации между правилами и узлами, возникающие на этапе style matching.

## 4.2. Непрямой характер представления

CSSOM не является изоморфным текстовому CSS и не представляет собой простое дерево. Итоговый визуальный результат зависит от совокупности множества факторов: различных уровней каскада, медиа-условий и механизмов наследования. Это структурированная модель правил и зависимостей, а не линейная копия исходных файлов.

## 4.3. Связь с каскадом (Cascade Resolution)

<font color="#fb4934">Фундаментальный признак CSSOM — его пригодность для разрешения каскада.</font> 

Модель позволяет вычислить итоговое значение свойства для конкретного элемента, учитывая:

- Уровень источника (origin).
- Специфичность селекторов (specificity).
- Порядок появления в исходном коде (source order).
- Наследование (inheritance).
- Условную применимость (conditional applicability) через медиа-запросы.
- Приоритеты, заданные через `!important`.

## 4.4. Взаимодействие с DOM

CSSOM и [[DOM]] — это две независимые внутренние модели страницы. DOM описывает структуру узлов документа, в то время как CSSOM определяет пространство потенциально применимых к ним правил. Их пересечение критически необходимо для формирования [[Render Tree]].

## 4.5. Контекстная зависимость

Значение CSSOM динамически зависит от условий среды:
- Размеры области просмотра (viewport).
- Состояние медиа-запросов (media query state).
- Поддерживаемые возможности браузерного движка.
- Состояние псевдоклассов и атрибутов DOM-узлов.

# 5. Архитектурная роль в Browser Rendering

Процесс трансформации данных в визуальном конвейере выглядит следующим образом:

`CSS source → parsing → CSSOM → style matching against DOM → computed style resolution → [[Render Tree]] → дальнейшие стадии pipeline`.

В этой архитектуре CSSOM выполняет четыре основные функции:
1. **Нормализация**: перевод текстовых описаний в формальную внутреннюю модель.
2. **Алгоритмическое применение**: обеспечение эффективного сопоставления селекторов и разрешения конфликтов деклараций.
3. **Исключение узлов**: участие в определении элементов, не участвующих в визуальном выводе (например, через `display: none`).
4. **Динамическое обновление**: поддержка частичной или полной переработки стилевой информации при изменении классов, атрибутов или JS-манипуляциях.

# 6. Логико-формальное ядро

Формально CSSOM можно представить как отображение между множеством правил и множеством элементов документа.

Пусть:
- $D$ — множество узлов DOM.
- $R$ — множество CSS-правил.
- $S(r, d)$ — предикат применимости селектора правила $r$ к элементу $d \in D$.
- $Decl(r)$ — множество деклараций правила $r$.

Множество кандидатов на применение для элемента $d$:

$Applicable(d) = \{ r \in R \mid S(r, d) = \text{true} \}$

Итоговое вычисленное состояние (Computed Style) определяется функцией разрешения каскада:

$ComputedStyle(d) = ResolveCascade(Applicable(d), \text{context})$

где $\text{context}$ включает параметры специфичности, порядка и наследования. CSSOM здесь выступает как структурированная область определения для функции $ResolveCascade$.

# 7. Практические применения и сценарии

## 7.1. Построение и обновление интерфейса

- **Построение Render Tree**: определение визуальных характеристик на основе сопоставления DOM и CSSOM.
- **Динамические изменения**: при добавлении класса элементу меняется подмножество применимых к нему правил, что инициирует цикл пересчета стилей.
- **Медиа-условия**: активация медиа-запроса при изменении размеров окна меняет контекст интерпретации CSSOM, делая ранее неактивные правила активными.

## 7.2. Программный доступ и Performance Engineering

JavaScript использует CSSOM API для динамического тематизирования, адаптации UI и инспекции стилей. С точки зрения производительности, организация CSSOM напрямую влияет на стоимость операций сопоставления селекторов. Глубокие или универсальные селекторы увеличивают вычислительную нагрузку на этапе style matching.

# 8. Типичные ошибки и заблуждения

- **Заблуждение**: CSSOM — это просто «дерево стилей».
    - _Уточнение_: Это сложная модель правил и условий, а не простая древовидная копия CSS.
- **Заблуждение**: CSSOM является частью DOM.
    - _Уточнение_: Это параллельные структуры; их объединение происходит только при формировании [[Render Tree]].    
- **Заблуждение**: CSSOM тождественен Computed Styles.
    - _Уточнение_: CSSOM — это база правил, а Computed Style — результат применения этих правил к конкретному узлу.    
- **Заблуждение**: Браузер может обойтись без CSSOM, накладывая стили сразу на DOM.
    - _Уточнение_: Без промежуточной модели разрешение каскада и конфликтов стилей вычислительно невозможно.    

# 9. Связанные понятия

- [[Browser Rendering]]: общий контекст визуального конвейера. 
- [[DOM]]: структурная модель документа.
- [[Render Tree]]: результат пересечения DOM и CSSOM.
- **Cascade**: алгоритм разрешения приоритетов правил.
- **Computed Style**: итоговые значения свойств для конкретного элемента.

# 10. Концептуальная карта (Flow)

1. **CSS source**: текстовое описание.    
2. **CSSOM**: внутренняя модель правил.
3. **DOM**: структура документа.
4. **Style resolution**: процесс сопоставления.
5. **[[Render Tree]]**: подготовленная для отрисовки структура.
6. **Layout / Paint**: последующие визуальные этапы.
