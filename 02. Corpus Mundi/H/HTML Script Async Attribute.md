# 1. Async Script

> <font color="#fb4934">Async Script</font> — это недетерминированный режим фоновой загрузки и приоритетного исполнения JavaScript-кода, задаваемый посредством булева атрибута `async` у HTML-элемента `<script>`. Для классических внешних скриптов (external classic scripts) наличие данного атрибута инструктирует браузер осуществлять сетевую загрузку ресурса параллельно с синтаксическим анализом (парсингом) HTML-документа, не блокируя построение DOM-дерева на фазе скачивания (fetch). Исполнение загруженного кода инициируется немедленно после завершения сетевого запроса, что влечет за собой синхронную приостановку HTML-парсера (если он еще не завершил работу) на время интерпретации и выполнения скрипта.

В инженерном контексте атрибут `async` представляет собой механизм реорганизации расписания (scheduling) жизненного цикла скрипта, состоящего из трех фаз: обнаружения ресурса, загрузки и исполнения. Атрибут устраняет задержку, вызываемую блокирующей загрузкой, однако принципиально не вводит предсказуемого порядка исполнения между несколькими асинхронными скриптами и не предоставляет гарантий наличия определенных узлов в DOM-дереве к моменту выполнения логики.

# 2. Историческое развитие и эволюция концепта

Изначально базовой вычислительной моделью для классических скриптов в HTML являлся строго синхронный (parser-blocking) режим. При обнаружении тега `<script src="...">` браузер был обязан полностью приостановить продвижение парсера, инициировать сетевой запрос, дождаться ответа и выполнить полученный код до возобновления разбора последующих HTML-тегов. Данный подход обеспечивал строгую консистентность состояния (скрипт всегда "видел" DOM до себя и никогда — после), но создавал критическое узкое место в производительности начальной загрузки страницы.

Атрибуты `async` и `defer` были введены в спецификацию HTML5 как стандартизированные механизмы декаплинга (развязки) сетевой загрузки скрипта и продвижения HTML-парсера, нацеленные на оптимизацию критического пути рендеринга (Critical Rendering Path).

В современной архитектуре веб-платформы семантика `async` претерпела эволюцию и была расширена на модульные скрипты (`module script`). Для `<script type="module">` (которые по умолчанию имеют поведение, аналогичное `defer`) добавление атрибута `async` предписывает браузеру загружать сам модуль и весь его граф зависимостей параллельно парсингу, а затем выполнять их немедленно по мере полной готовности графа. Таким образом, разделение режимов загрузки между классическими и модульными скриптами стало фундаментальным встроенным элементом современной спецификации WHATWG HTML.

# 3. Формальные свойства, структура, ключевые признаки

Свойства режима `async` классифицируются через набор поведенческих инвариантов:
- **Область применимости**: Атрибут функционален исключительно для скриптов с внешним источником (заданным через атрибут `src`) в классическом режиме, а также для модульных скриптов (где он влияет на весь граф импортов).
- **Семантика сетевой загрузки (Fetch)**: Загрузка ресурса делегируется фоновому сетевому потоку параллельно с парсингом HTML. Полная остановка парсера в точке обнаружения тега не производится.
- **Семантика исполнения (Execution)**: Код исполняется в главном потоке немедленно после готовности ресурса. Если в этот момент построение DOM-дерева еще продолжается, парсер прерывается. Режим `async` устраняет `network contention` (блокировку сетью), но сохраняет `main thread contention` (блокировку вычислительным потоком) на этапе исполнения.
- **Отсутствие гарантий топологического порядка (Race condition)**: При наличии множества тегов `<script async>` лексический порядок их следования в документе игнорируется. Очередность выполнения диктуется исключительно скоростью ответа сети и размером файлов (выполняется тот, кто загрузился первым). Режим математически несовместим с графами строгих зависимостей.
- **Агностичность к состоянию DOM-дерева**: Момент исполнения недетерминирован. Скрипт может быть выполнен на ранних стадиях построения документа, следовательно, обращение к узлам DOM, расположенным ниже по дереву, может привести к ошибкам доступа.
- **Асинхронность относительно событий жизненного цикла**: Парсер документа и событие `DOMContentLoaded` не ожидают завершения загрузки или исполнения `async`-скриптов. Однако глобальное событие `Window.load` учитывает их, так как ожидает загрузки всех запрошенных ресурсов страницы.

# 4. Сопоставление с соседними режимами загрузки (Async vs Defer)

Для точного инженерного выбора стратегии необходимо прямое сопоставление `async` с синхронным режимом и режимом `defer` по ключевым осям планирования.

- **Синхронный скрипт (без атрибутов)**: Блокирует парсер и на этапе сетевой загрузки, и на этапе исполнения. Гарантирует строгий порядок по документу и известное состояние DOM. Является наиболее "дорогим" по влиянию на время первой отрисовки (First Paint).
- **Defer Script (`defer`)**: Осуществляет параллельную загрузку без блокировки парсера. Гарантирует, что исполнение начнется строго после полного завершения парсинга HTML, но до наступления события `DOMContentLoaded`. Для классических скриптов сохраняет исходный порядок следования в документе.
- **Async Script (`async`)**: Осуществляет параллельную загрузку. Гарантирует максимально раннее исполнение по факту готовности ресурса, прерывая парсинг при необходимости. Жертвует предсказуемостью порядка и гарантиями готовности DOM ради минимизации задержки выполнения конкретного скрипта.

# 5. Поведение для inline-скриптов и динамически вставляемых скриптов

Атрибут `async` демонстрирует специфическое поведение в зависимости от метода внедрения кода в документ:

- **Внутренние (inline) скрипты**: Конструкции вида `<script async> console.log("Init"); </script>` игнорируют атрибут `async`. Внутренний код не требует сетевой загрузки, поэтому выполняется синхронно в момент прохождения парсера. Для отложенного выполнения внутренних скриптов применение атрибута бессмысленно; требуется использование событийных подписок (например, на `DOMContentLoaded`) или паттерна отложенного вызова.
- **Динамически вставляемые скрипты**: Скрипты, создаваемые через DOM API (`document.createElement('script')`), в современных браузерах по умолчанию неявно получают поведение `async = true`. Они будут загружены параллельно и выполнены в произвольном порядке по мере готовности. Если для динамических скриптов требуется строгий порядок, разработчик обязан явно установить `script.async = false`, что заставит браузер выполнять их в порядке добавления в DOM.

# 6. Современное значение и интерпретации

В парадигме современной frontend-инженерии `async` интерпретируется не как универсальная оптимизация для любого JavaScript-кода, а как узкоспециализированный инструмент для полностью инкапсулированных подключений.

Архитектурный консенсус диктует, что основной application bundle (прикладной код), логика рендеринга интерфейса и библиотеки со сложным графом зависимостей не должны загружаться через `async`. Для них стандартом де-факто является `defer` (или поведение `<script type="module">`), обеспечивающее сохранность порядка и безопасный доступ к построенному DOM.

Сфера применения `async` сужается до стороннего кода (third-party scripts), который ортогонален основному приложению: внешняя аналитика, телеметрия, системы A/B тестирования, рекламные пиксели и скрипты наблюдаемости (observability). Это продиктовано тем, что данные системы соответствуют двум критериям: они не имеют внешних зависимостей и их функционирование не требует немедленного доступа к полностью сформированному документу.

# 7. Практические применения

## Сценарий 1: Независимая аналитика или телеметрия (Идеальный случай)

Скрипты, которым важен лишь факт регистрации хита или сбора начальных метрик среды, идеально ложатся на семантику `async`.

В данном случае сеть не блокирует парсер, а произвольный момент исполнения не нарушает логику основного интерфейса.

## Сценарий 2: Адаптация независимого скрипта к состоянию DOM (Защитный паттерн)

Если асинхронный скрипт всё же должен манипулировать DOM-деревом (например, сторонний виджет чата), его код должен быть спроектирован с учетом недетерминированности среды. Поскольку скрипт может выполниться как до, так и после события `DOMContentLoaded`, безопасная инициализация требует явной проверки свойства `document.readyState`.

## Сценарий 3: Зависимые скрипты (Антипаттерн)

Если `/app.js` опирается на глобальные переменные или функции, предоставляемые `/vendor.js`, данная схема приведет к "плавающим" ошибкам (Heisenbugs). Если `/app.js` загрузится на несколько миллисекунд быстрее (например, из-за особенностей сетевой маршрутизации), он будет немедленно выполнен и потерпит крах с `ReferenceError`.

# 8. Типичные ошибки, заблуждения, ограничения

- **Заблуждение о "быстром defer"**: Распространенной ошибкой является трактовка `async` как более производительной версии `defer`. Это принципиально разные контракты: `defer` жертвует немедленностью исполнения ради детерминизма (порядок + DOM), тогда как `async` жертвует детерминизмом ради предельно раннего старта логики.
- **Заблуждение об ожидании `DOMContentLoaded`**: Инженерная ошибка полагать, что событие готовности документа будет отложено до загрузки `async`-скриптов. Обработчики на `DOMContentLoaded` часто выполняются до того, как тяжелый асинхронный скрипт будет скачан, что приводит к рассинхронизации состояний в приложении.
- **Ограничение главного потока (Main Thread Contention)**: `async` устраняет исключительно сетевую блокировку. В момент, когда скрипт готов, его интерпретация и компиляция выполняются в главном потоке V8/SpiderMonkey, что неизбежно блокирует рендеринг, сборку мусора и обработку пользовательского ввода (Total Blocking Time). Для сверхтяжелых вычислений атрибут не спасает от фризов интерфейса — в таких случаях требуется перенос логики в Web Workers.
- **Применение к критическому приложению**: Использование режима для загрузки ядра SPA (Single Page Application) делает систему хрупкой, усложняя отладку из-за недетерминированной последовательности инициализации модулей.

# 9. Связанные понятия

- **Defer Script**: Смежный атрибут загрузки, обеспечивающий параллельный fetch, но откладывающий исполнение до завершения HTML-парсинга с сохранением исходного порядка тегов.
- **Critical Rendering Path (Критический путь рендеринга)**: Комплекс шагов, предпринимаемых браузером для конвертации HTML, CSS и JS в пиксели на экране; оптимизация этого пути является первопричиной существования неблокирующих скриптов.
- **Module Script (`<script type="module">`)**: Современная парадигма модульного JavaScript, имеющая встроенную семантику отложенной загрузки, в рамках которой `async` форсирует немедленное исполнение всего графа зависимостей модуля.
- **DOMContentLoaded**: Триггерное событие жизненного цикла документа, разорванное по смыслу с окончанием работы `async`-скриптов.
- **Browser Loading Optimization Mechanisms**: Мета-система браузера, включающая спекулятивный парсинг (preload scanners), resource hints и атрибуты загрузки, управляющая приоритизацией ресурсов.
