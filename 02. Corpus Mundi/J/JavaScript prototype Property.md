# 01.  `prototype` property (Функции-конструкторы)


> `prototype` — это обычное публичное свойство функции-конструктора, значение которого представляет собой объект-прототип, используемый для реализации прототипного наследования и размещения разделяемых свойств (shared properties). Когда конструктор создаёт новый объект через оператор `new`, этот объект неявно ссылается на значение `constructor.prototype` для разрешения ссылок на свойства в процессе поиска по цепочке прототипов.

```JavaScript
function User(name) {
  this.name = name; // own property экземпляра
}

// Размещаем разделяемый метод на User.prototype
User.prototype.sayHi = function () {
  return `Hi, ${this.name}`;
};

const u = new User("Alice");

// 1. У экземпляра есть собственное свойство
console.log(u.name); // "Alice"
console.log(Object.hasOwn(u, "name")); // true

// 2. У экземпляра нет собственного sayHi
console.log(Object.hasOwn(u, "sayHi")); // false

// 3. Но sayHi находится через prototype chain
console.log(u.sayHi()); // "Hi, Alice"

// 4. Прототип экземпляра равен User.prototype
console.log(Object.getPrototypeOf(u) === User.prototype); // true

// 5. Это и есть ключевая идея:
// new User(...) создаёт объект,
// чей [[Prototype]] указывает на User.prototype
```

В строгом спецификационном смысле (согласно ECMA-262) свойство `prototype` у функции не тождественно другим механизмам прототипной модели языка:
- `prototype` — это публичное свойство самой функции-конструктора.
- `[[Prototype]]` — это скрытый внутренний слот созданного объекта (internal slot).
- `__proto__` — это legacy-аксессор (accessor property), определенный на `Object.prototype`, чей геттер и сеттер работают через внутренние методы `[[GetPrototypeOf]]` и `[[SetPrototypeOf]]`.

# 2. Историческое развитие / эволюция концепта

JavaScript изначально проектировался как язык с прототипным наследованием (prototype-based inheritance). В базовой терминологии спецификации, конструктор — это функция, создающая и инициализирующая объекты, а значение её публичного свойства `prototype` используется для реализации наследования. Следовательно, свойство `prototype` не является поздним декоративным соглашением или синтаксической надстройкой; оно относится к фундаментальной, базовой модели объектного устройства языка.

В ходе дальнейшей эволюции стандарта спецификация ECMAScript уточнила два критических аспекта:
- Было строго формализовано архитектурное различие между публичным свойством `prototype` и внутренним слотом `[[Prototype]]`.
- Было математически закреплено, что новый объект получает своё значение `[[Prototype]]` через специализированные абстрактные операции `OrdinaryCreateFromConstructor` и `GetPrototypeFromConstructor`. Эти операции алгоритмически читают именно публичное свойство конструктора `prototype`, а не какой-либо скрытый метадескриптор или внутреннее состояние функции.

# 3. Формальные свойства, структура, ключевые признаки

- **Онтологический статус свойства**: Свойство `prototype` — это обычное свойство функции, доступное в коде через выражение `constructor.prototype`. Оно принципиально не относится к классу внутренних слотов (internal slots) и не требует специального синтаксиса спецификации для обращения. Именно поэтому его спецификационный статус допускает чтение, замену и расширение на уровне обычного ECMAScript-кода.
- **Отношение к функциям-конструкторам**: Спецификация прямо фиксирует, что у каждого конструктора по умолчанию есть свойство `prototype`. Для функций, созданных через `FunctionDeclaration` (объявление функции) и `FunctionExpression` (функциональное выражение), свойство `prototype` создаётся движком автоматически, поскольку такая функция потенциально может быть вызвана с оператором `new` и использована как конструктор.
- **Требование осмысленного (meaningful) прототипа**: Из факта существования функций не следует, что любое вызываемое (callable) значение обязано иметь свойство `prototype`. Спецификация отдельно указывает, что встроенные объекты-функции (built-in function objects), которые не являются конструкторами (например, стрелочные функции или методы встроенных API), обычно не имеют свойства `prototype`, если в стандарте явно не сказано иное. Тема свойства `prototype` относится исключительно к конструкторам и тем функциям, которые спецификация делает конструируемыми (constructable).
- **Связь с созданием экземпляра**: Абстрактная операция `OrdinaryCreateFromConstructor` создаёт обычный объект (ordinary object), чей внутренний слот `[[Prototype]]` извлекается из свойства `prototype` конструктора, при условии, что это значение является объектом. Операция `GetPrototypeFromConstructor` формализует тот же алгоритм: она читает значение через `Get(constructor, "prototype")` и, если полученное значение не является объектом, подставляет внутренний прототип по умолчанию (intrinsic default prototype) соответствующего контекста выполнения (realm). Таким образом, свойство `prototype` не хранит экземпляры внутри себя и не «содержит» внутренний слот; оно выступает в роли конфигурационного параметра — задаёт исходное значение, из которого инициализируется `[[Prototype]]` нового объекта при его рождении.
- **Различие между `F.prototype` и `F.[[Prototype]]`**: Для функции `F` одновременно существуют два разных уровня онтологической связи. Как `function object`, функция `F` сама является объектом и имеет собственный внутренний слот наследования `[[Prototype]]`. Обычное создание функции выполняется через абстрактную операцию `OrdinaryFunctionCreate(%Function.prototype%, ...)`, что означает, что функция `F` сама наследует от `Function.prototype`. Однако, помимо этого, у неё есть отдельное публичное свойство `F.prototype`, предназначенное уже не для самой функции, а для делегирования будущих объектов, создаваемых через `new F(...)`. Это два семантически разных отношения с разными векторами направленности и архитектельными ролями.
- **Различие между `F.prototype` и `F.__proto__`**: Вызов `F.__proto__` в нормальных условиях наблюдает состояние `F.[[Prototype]]`, поскольку `__proto__` — это legacy-аксессор, чей геттер возвращает результат вызова `O.[[GetPrototypeOf]]()`. Следовательно, `F.__proto__` описывает саму функцию как объект и её место в иерархии прототипов, тогда как `F.prototype` описывает конфигурацию для будущих объектов, порожденных этой функцией. Их совпадение в штатном режиме не предполагается и отсутствует.
- **Внутренняя асимметрия модели**: Свойство `prototype` — это «вход» в процесс создания нового объекта, в то время как `[[Prototype]]` — это «результат» инициализации уже созданного экземпляра. Связь между ними имеет строгую асимметричную направленность: не `obj.[[Prototype]]` определяет состояние `Constructor.prototype`, а строго наоборот. Значение `Constructor.prototype` считывается интерпретатором в момент конструирования, после чего навсегда (до явной мутации) используется для установки `obj.[[Prototype]]`. Это различие принципиально для глубокого понимания семантики оператора `new`.

# 4. Границы темы и разграничение зон ответственности

Для корректной систематизации необходимо четко определить границы применимости концепта свойства `prototype` и точки перехода к смежным архитектурным механизмам:

- **Что относится к теме непосредственно**: Свойство `prototype` у функций-конструкторов; механизмы автоматического создания этого свойства для обычных функций (объявлений и выражений); его алгоритмическое использование абстрактными операциями `OrdinaryCreateFromConstructor` и `GetPrototypeFromConstructor`; его архитектурная роль как контейнера для разделяемых методов и общего поведения экземпляров.
- **Где тема переходит к внутреннему устройству объекта**: Концептуальная граница проходит ровно там, где инженерное обсуждение перестаёт касаться свойства функции-конструктора и начинает касаться внутреннего слота экземпляра. Как только речь заходит о фактическом значении `[[Prototype]]` созданного объекта, фокус смещается от свойства `prototype` к теме `Internal Prototype Slot`. Спецификация проводит жесткую демаркацию этих уровней: `GetPrototypeFromConstructor` читает свойство конструктора, после чего `ordinary object` хранит полученную итоговую ссылку автономно, уже в собственном `[[Prototype]]`.
- **Где тема переходит к пользовательскому legacy-доступу**: Когда техническая дискуссия смещается к использованию `__proto__`, речь идёт уже не о свойстве конструктора, а о legacy-аксессоре на уровне `Object.prototype`, который предоставляет интерфейс к внутренним методам `[[GetPrototypeOf]]` и `[[SetPrototypeOf]]`. Следовательно, `__proto__` принадлежит другой терминологической ветке: наблюдение и мутация прототипа конкретного инстанса, а не конфигурация будущих экземпляров на уровне фабрики/конструктора.
- **Где тема переходит к prototype chain lookup**: Полный процесс разрешения свойств (lookup) по цепочке прототипов начинается строго после того, как `[[Prototype]]` объекта был успешно установлен. Само свойство `constructor.prototype` лишь поставляет исходную точку, первоначальную координату для этой связи. Алгоритм дальнейшего поиска недостающих свойств принадлежит уже домену `Prototype Chain` и разрешению внутренних слотов, а не теме свойства `prototype` в узком спецификационном смысле.

# 5. Современное значение и интерпретации

В современной инженерной практике и архитектуре веб-приложений прямое взаимодействие со свойством `prototype` часто скрыто за более высокоуровневыми синтаксическими формами записи (например, синтаксис `class` из ES2015). Однако спецификационный статус механизма остаётся фундаментальным: синтаксический сахар классов транслируется в те же самые прототипные парадигмы под капотом. Даже если разработчик концептуализирует код в терминах «конструкторов», «экземпляров» и «общих методов», каноническое объяснение движка по-прежнему проходит через то, что конструктор предоставляет объект-шаблон через своё свойство `prototype`, а экземпляр получает ссылку на этот объект в качестве значения собственного слота `[[Prototype]]`.

Современная академическая интерпретация этой темы требует полного отказа от псевдоклассовой иллюзии, при которой свойство `prototype` воспринимается как «строгий тип» или «описание структуры класса». В строгом спецификационном смысле `prototype` не является ни типом данных, ни метаклассом; это исключительно стандартизированное свойство функции, чьё значение утилитарно используется виртуальной машиной для настройки делегирующей связи вновь создаваемых объектов.

# 6. Математическое или логико-формальное ядро

Процесс связывания прототипов поддается строгой формализации. Пусть $C$ — функция-конструктор, а $P$ обозначает объект, хранящийся в её прототипе ($P = C.prototype$). Тогда для создаваемого объекта $O$ общая формальная схема абстрактного алгоритма выглядит следующим образом:
$$P := \text{Get}(C, \text{"prototype"})$$
$$O := \text{OrdinaryObjectCreate}(P)$$
_Условие применимости:_ Данная модель применяется, если $P$ является объектом (Object type); в противном случае вместо $P$ используется внутренний прототип по умолчанию (`intrinsic default prototype`) из соответствующего контекста выполнения (`realm`). Это не является полной пошаговой репрезентацией оператора `new`, но служит точной абстракцией того участка алгоритма, который связывает свойство конструктора со слотом создаваемого объекта.

Из этого алгоритма выводится критически важное логическое различие:

$$C.prototype \neq C.[[Prototype]]$$

И в наблюдаемой (runtime) форме через legacy-аксессор:

$$C.prototype \neq C.\text{\_\_proto\_\_}$$

Левая часть уравнений всегда относится к собственному публичному свойству функции, предоставляемому для делегирования другим объектам, в то время как правая часть описывает собственную позицию функции $C$ в цепочке наследования в качестве объекта-функции (делегирование к `Function.prototype`).

# 7. Практические применения

Свойство `prototype` имеет конкретные паттерны применения в архитектуре программного обеспечения:
- **Размещение разделяемых методов (Shared Methods Placement)**: Свойство `prototype` используется как каноническое, стандартизированное место в памяти для хранения методов, которые должны разделяться всеми экземплярами конструктора. Это математически соответствует базовой идее спецификации ECMAScript: объект-прототип предоставляет `shared properties` для других объектов, а свойства, единожды добавленные на прототип, становятся неявно доступны всем объектам, разделяющим данный прототип через внутренний слот наследования.

```JavaScript
function User(name) {
  this.name = name;
}

User.prototype.sayHi = function () {
  return `Hi, ${this.name}`;
};
```

- **Динамический контроль над прототипом создаваемых объектов**: Прямая замена значения `Constructor.prototype` до вызова оператора `new` радикально меняет тот объект, который будет использован как исходное значение `[[Prototype]]` для всех новых экземпляров. Это поведение логически вытекает из факта, что операция `GetPrototypeFromConstructor` производит чтение свойства в момент инстанцирования. Как следствие, изменение `Constructor.prototype` влияет исключительно на будущие экземпляры, но не переопределяет и не мутирует ретроактивно уже созданные объекты. Последнее является прямым следствием описанного выше алгоритма — уже существующий объект автономно хранит свой собственный `[[Prototype]]`.

```JavaScript
function A() {}
const x = new A();

A.prototype = { value: 42 };
const y = new A();

// Объекты x и y будут связаны с совершенно разными объектами-прототипами в памяти.
```

- **Ясное разделение уровней проектирования**: Глубокое понимание спецификационного различия между `prototype`, `__proto__` и `[[Prototype]]` является критическим требованием для корректной низкоуровневой отладки, рефакторинга сложных конструкторов и статического анализа поведения оператора `new`. Без этого структурного разграничения разработчик рискует смешать три разных уровня архитектуры языка: публичное свойство функции, скрытый внутренний слот экземпляра и legacy-механизм наблюдения за этим слотом.
    

# 8. Типичные ошибки, заблуждения, ограничения

В инженерной практике систематически возникают следующие антипаттерны и заблуждения, связанные с неверной трактовкой концепта:
- **Отождествление `prototype` и `__proto__`**: Наиболее распространённая концептуальная ошибка. Свойство `prototype` — это конфигурационное свойство функции-конструктора; `__proto__` — это legacy-аксессор, наблюдающий или мутирующий `[[Prototype]]` конкретного объекта. Они оперируют на разных уровнях объектной модели.
- **Отождествление `prototype` и `[[Prototype]]`**: Фундаментально неверно. Свойство `prototype` принадлежит функции-конструктору как обычное публичное `property`; `[[Prototype]]` принадлежит самому экземпляру как внутренний слот `ordinary object`. Между этими сущностями существует однонаправленная функциональная связь при инициализации, но отсутствует тождество.
- **Мнение, что каждая функция имеет полноценное свойство `prototype`**: Спецификация не подтверждает этот постулат. Встроенные функции (`built-in functions`), которые не являются конструкторами, обычно не инициализируются свойством `prototype`. Следовательно, наличие этого свойства нужно связывать исключительно со способностью функции к конструированию (`constructability`), а не с самим фактом её вызываемого (`callable`) поведения.
- **Представление о том, что `prototype` описывает саму функцию**: Сама функция, являясь объектом, делегирует свои вызовы (наследует) стандартному `Function.prototype`, что задаётся через её собственный слот `[[Prototype]]`. Напротив, свойство `F.prototype` не описывает саму функцию `F`, а является шаблоном (дескриптором поведения), который будет использован исключительно при создании её экземпляров.
- **Игнорирование случая не-объектного значения в `constructor.prototype`**: Если свойству `prototype` конструктора присвоить примитив (значение, не являющееся объектом), спецификация не позволяет сделать его прототипом экземпляра. Вместо выброса исключения, алгоритм безопасно откатывается к использованию `intrinsic default prototype` из `realm` конструктора. Это жесткое спецификационное ограничение часто упускается в упрощённых учебных материалах.
- **Путаница вокруг объекта `Function.prototype`**: Глобальный объект `Function.prototype` — это специальный `Function prototype object`, который сам технически является `function object`, однако он парадоксальным образом не имеет свойства `prototype` и не может быть использован в качестве конструктора с оператором `new`. Этот спецификационный нюанс выступает в роли идеального предельного примера, доказывающего, что понятия «быть функцией» и «иметь meaningful prototype для конструирования» — это независимые характеристики в топологии языка.

# 9. Связанные понятия

- **`[[Prototype]]`**: Скрытый внутренний слот (internal slot) обычного объекта (`ordinary object`), физически хранящий фактическую ссылку на прототип данного объекта и непосредственно участвующий в разрешении свойств при наследовании.
- **`__proto__` / `[[Proto Accessor]]`**: Стандартизированный в качестве исторического наследия (legacy-accessor) геттер/сеттер на объекте `Object.prototype`, который предоставляет пользовательскому коду прямой доступ к внутренним методам `[[GetPrototypeOf]]` и `[[SetPrototypeOf]]`.
- **Constructor (Конструктор)**: Специализированный `function object`, задачей которого является создание и инициализация новых объектов. Значение его свойства `prototype` используется движком для обеспечения цепочки наследования и предоставления разделяемых свойств (`shared properties`).
- **`OrdinaryCreateFromConstructor`**: Фундаментальная абстрактная операция спецификации, отвечающая за алгоритм создания объекта и выбор корректного значения для его `[[Prototype]]` путем извлечения свойства `prototype` из переданного конструктора.
- **`GetPrototypeFromConstructor`**: Абстрактная операция, осуществляющая безопасное чтение `constructor.prototype` и математически определяющая (с учетом фоллбэков на `realm`), какой именно прототип будет назначен рождающемуся объекту.
- **Prototype Chain (Цепочка прототипов)**: Логическая последовательность связанных объектов, которая формируется исключительно после того, как слот `[[Prototype]]` уже был успешно установлен; отвечает за весь последующий `lookup` свойств во время выполнения программы.

# 10. Архитектурное резюме (Итог)

Свойство `prototype` в JavaScript представляет собой канонический архитектурный механизм, посредством которого функция-конструктор декларативно задаёт прототип для своих будущих экземпляров. Его семантическое значение не совпадает ни с аксессором `__proto__`, ни с внутренним слотом `[[Prototype]]`. Первое (`__proto__`) относится к функции как к объекту и наблюдает её собственную позицию в глобальной цепочке наследования; второе (`[[Prototype]]`) является защищенным внутренним слотом уже созданного и инициализированного экземпляра; в то время как `prototype` неизменно остаётся публичным конфигурационным свойством самого конструктора, утилитарно используемым виртуальной машиной в момент рождения новых объектов.

Строгое академическое и инженерное понимание этой темы требует безошибочного удержания следующей темпоральной последовательности инициализации:

1. Изначально существует функция-конструктор и её свойство `prototype`.
2. В момент инициирования оператора `new` при создании экземпляра, значение этого свойства алгоритмически считывается движком.
3. Полученное значение (если оно является объектом) используется абстрактными операциями для прямой инициализации внутреннего слота `[[Prototype]]` нового инстанса.
4. Исключительно после завершения этого процесса начинает функционировать механизм разрешения свойств (`lookup`) по цепочке прототипов (`prototype chain`).

Именно такое хронологическое и онтологическое разграничение делает объектную модель ECMAScript формально ясной, устраняя типичную инженерную путаницу между тремя внешне схожими, но семантически независимыми сущностями языка.
