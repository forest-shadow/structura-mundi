# 1. React Components

> <font color="#fb4934">React Components (Компоненты React)</font> — это фундаментальная концепция и структурная модель библиотеки React, в которой пользовательский интерфейс (UI) строится как иерархическое дерево из изолированных, декларативных, переиспользуемых и композиционно сочетаемых единиц описания.

> В строгом смысле <font color="#fb4934">компонент React</font> — это программная абстракция, инкапсулирующая логику отображения и внутреннее состояние. Она получает входные данные, участвует в формировании виртуального дерева интерфейса и выступает атомарной единицей композиции. 
> * Данная концепция фиксирует саму парадигму мышления об архитектуре интерфейса, а не сводится к описанию отдельного API или конкретной формы реализации.

# 2. Этимология и происхождение концепта

Термин _component_ восходит к парадигме компонентно-ориентированного программирования (Component-Based Software Engineering, CBSE) и общей инженерной идее составного объекта, который интегрируется в более крупную систему как самостоятельная часть с чётко определённой ролью, контрактом и границей ответственности.

До появления React в UI-разработке доминировало технологическое разделение (HTML, CSS, JS), а элементы интерфейса строились на базе шаблонов, контролов или виджетов, управляемых преимущественно императивно. React переопределил это понятие: в его контексте компонент закрепился прежде всего как единица декларативного описания и композиции UI, объединяющая разметку и логику (Separation of Concerns на уровне функциональных блоков), а не просто как визуальный контрол.

# 3. Историческое развитие и эволюция модели

Компонентная модель существовала задолго до React, однако именно эта библиотека придала ей специфическую форму, прошедшую несколько этапов эволюции:

- **Эпоха фабрик (React 15 и ниже):** использование `React.createClass()` и примесей (Mixins) для переиспользования логики.
- **Объектно-ориентированная эпоха (React 15.5 – 16.7):** доминирование `[[Class Components]]`, основанных на спецификации ES6. Компонент понимался как объектно-ориентированная сущность. Управление состоянием, побочными эффектами и жизненным циклом осуществлялось через жестко заданные методы класса.
- **Функциональная эпоха (React 16.8 – настоящее время):** внедрение хуков (Hooks). Центральную роль заняли `[[Functional Components]]`. Изначально они рассматривались лишь как упрощённая форма для «presentational» (глупых) компонентов, но с появлением хуков стали основной современной формой. Это изменило саму интерпретацию модели:
    - Компонент стал пониматься прежде всего как математическая функция декларативного описания UI.
    - Локальное состояние, эффекты и повторное использование логики были нативно встроены в функциональную парадигму.
    - `Class components` сохранились как важная историческая (legacy) реализация для поддержки старых баз кода, но перестали быть ядром современного React.
- **Серверная эпоха (React 18+):** внедрение React Server Components (RSC). Разделение компонентной модели на серверные (асинхронные, выполняемые вне браузера) и клиентские (интерактивные) узлы, что расширило границы ответственности компонентов до уровня серверной инфраструктуры.

Параллельно этому процессу React последовательно утверждал принцип _Composition over Inheritance_ (Композиция вместо наследования): сложный UI должен строиться через вложение и специализацию, а не через иерархии классов.

# 4. Формальные свойства, структура, ключевые признаки

Компонентная модель React обладает строгими инвариантами и архитектурными признаками:

- **Компонент как единица декларативного описания:** компонент не описывает императивную последовательность манипуляций с DOM. Он задаёт спецификацию того, _каким_ должен быть интерфейс при текущих входных данных и состоянии
- **Идемпотентность (целевая):** при идентичных входных данных (`props`) и внутреннем состоянии (`state`) функция рендеринга должна возвращать строго идентичное поддерево элементов, не производя непредвиденных мутаций.
- **Компонент как единица композиции:** компоненты образуют направленный ациклический граф (DAG). Любой компонент может включать в себя другие компоненты, координируя общую структуру UI.
- **Граница ответственности и инкапсуляция:** компонент вводит строгую локальную границу между:
    - входными данными (контрактом);
    - внутренней бизнес-логикой и локальным состоянием (строго изолированным от прямого внешнего изменения);
    - описанием возвращаемого поддерева интерфейса.
- **Однонаправленность передачи данных (Unidirectional Data Flow):** данные передаются строго сверху вниз через `props`. Дочерний компонент не владеет состоянием родителя, а получает данные и callback-механизмы в рамках явно выраженного контракта.
- **Сегрегация уровней абстракции:** критически важно различать три сущности:
    - _Компонент_: абстракция уровня React, задающая логику и алгоритм рендеринга.
    - _React Element_: иммутабельное объектное описание конкретного узла UI (результат вызова компонента).
    - _DOM Node_: реальный узел браузерного документа, являющийся результатом фиксации изменений (commit phase). Компонент не тождественен HTML-тегу.
- **Независимость от переиспользуемости и файловой системы:** хотя компоненты часто создаются ради принципа DRY, переиспользуемость не является абсолютным критерием. Компонент оправдан даже при единичном использовании, если он формирует семантическую границу архитектуры. Кроме того, факт вынесения кода в отдельный файл — это организационное решение, а не структурное требование компонентной модели.

# 5. Структура компонентной системы

Компонентная модель React представляет собой целостную систему, концептуальная архитектура которой опирается на следующие взаимосвязанные элементы:

- `[[Props in React]]`: механизм входных данных и строгий контракт передачи информации от родительского узла к дочернему.
- `[[Functional Components]]`: доминирующая современная реализация компонентной модели, выступающая математической функцией вычисления интерфейса и инкапсуляции логики.
- `[[Component Composition]]`: базовый принцип архитектурной сборки UI, объясняющий координацию и вложение узлов в дереве интерфейса без использования жесткого связывания.
- `[[Class Components]]`: исторически значимая (legacy) реализация с иной внутренней организацией, основанная на объектно-ориентированном программировании и методах жизненного цикла.
- `[[Higher-Order Component]]`: паттерн обёртывания компонента компонентом, используемый для повторного использования и наращивания поведения на границе компонентного контракта.

# 6. Современное значение и интерпретации

В современной инженерной практике компонент интерпретируется преимущественно как единица декларативной модульности UI.

1. доминирует функциональная парадигма: компонент — это чистая (в идеале) функция, синхронизирующая побочные эффекты с состоянием.
2. компонент перестал быть исключительно визуальной единицей. Это контейнер локализации логики, где сосредоточены интеграция с хуками, координация дочерних элементов и обработка состояний загрузки/ошибок.
3. качество компонента сегодня оценивается не по количеству строк кода или визуальному размеру, а по качеству его структурной границы: насколько ясен его входной контракт и насколько изолирована его ответственность (Single Responsibility Principle).

# 7. Логико-формальное ядро

На концептуальном уровне компонентную модель React можно выразить через формальные отображения множеств.

Базовая функциональная абстракция описывается как:

$$V = f(P, S)$$

Где $V$ (View) — генерируемое представление, $f$ — алгоритм компонента, $P$ — иммутабельные внешние свойства, $S$ — мутабельное локальное состояние.

В более строгой и детальной записи, отражающей архитектуру современного React, компонент выступает как отображение $C : I \rightarrow E$, где:

$$E = C(P, X, L)$$

Где:

- $C$ — вызываемый компонент.
- $P$ (Props) — множество переданных входных параметров.
- $X$ (Context) — неявно доступный контекст выполнения.
- $L$ (Local State) — локальное состояние экземпляра компонента.
- $E$ (Element) — объектное описание результирующего поддерева интерфейса.

Итоговый пользовательский интерфейс формируется как направленный граф (дерево), составленный из узлов-компонентов:

$$UI = T(C_1, C_2, \dots, C_n)$$

Из этого математического ядра следуют два базовых инварианта React: структура носит строго иерархический характер, а финальный UI возникает не как плоский список императивных операций, а как рекурсивная композиция декларативных поддеревьев.

# 8. Практические применения

- **Архитектура сложного интерфейса:** декомпозиция монолитных экранов (включая SPA) на управляемые, изолированные фрагменты, что упрощает поддержку и локализует влияние багов.
- **Стандартизация и Design Systems:** создание библиотек UI-примитивов (кнопки, модальные окна, layout-контейнеры) для обеспечения визуальной и поведенческой консистентности в рамках крупных продуктов.
- **Синхронизация данных:** благодаря инкапсуляции состояния, изменения бизнес-данных отражаются в UI автоматически через механизм реактивности, исключая необходимость ручной синхронизации DOM.
- **Организация инженерной работы:** компонентная декомпозиция создаёт естественные границы для распределения задач между разработчиками, проведения Code Review, изолированного тестирования (Unit-тесты компонентов) и безопасного рефакторинга.
  
# 9. Типичные ошибки, заблуждения, архитектурные антипаттерны

- **Сведение компонента к «кусочку JSX» или DOM-узлу:** грубая редукция. JSX — лишь синтаксический сахар. Один компонент может возвращать логическое ветвление, пустой фрагмент или несколько компонентов, не генерируя собственного DOM-узла.
- **Смешение темы компонентов с темой props:** `props` — лишь механизм контракта. Компонент не тождественен функции получения `props`, это структурная единица с более широким композиционным смыслом.
- **Избыточная декомпозиция:** предположение, что любой микро-фрагмент UI обязан быть выделен в отдельный файл. Это приводит к фрагментации кодовой базы и усложнению когнитивной нагрузки. Выделение оправдано только при наличии ясной архитектурной границы.
- **Антипаттерн «God Component» (Божественный компонент):** нарушение границ ответственности путём концентрации избыточного объёма логики, сетевых запросов и разметки в одном узле.
- **Антипаттерн «Prop Drilling»:** передача данных через глубокие цепочки промежуточных компонентов, которым эти данные не нужны, вместо использования контекста или композиции слотов.
- **Нарушение чистоты рендера (Impure Render):** выполнение мутаций глобальных переменных или инициирование сетевых запросов непосредственно в теле функции компонента вне специализированных механизмов (например, `useEffect`).
