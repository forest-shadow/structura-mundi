# 01. Higher-Order Component

> <font color="#fb4934">Higher-Order Component, HOC (Компонент высшего порядка)</font> — это продвинутый паттерн проектирования в библиотеке React, при котором функция принимает компонент в качестве входного аргумента и возвращает новый компонент, расширяющий, модифицирующий или повторно использующий некоторое поведение исходного компонента.

> <font color="#fb4934">Higher-Order Component (HOC)</font> — это чистая функция-преобразователь, выражаемая базовым преобразованием $Component \to EnhancedComponent$. В более точной функциональной записи сигнатура имеет вид $f(C_{Base}) \to C_{Enhanced}$.

* С инженерной и архитектурной точек зрения HOC представляет собой компонентную обёртку уровня определения, а не уровень непосредственного JSX-комбинирования. HOC не является частью API React, но эксплуатирует композиционную природу функциональных и классовых компонентов. 
* Исходный компонент не изменяется in-place; вместо этого создаётся новый компонент, который инкапсулирует дополнительную сквозную функциональность (cross-cutting concerns) и делегирует рендеринг обёрнутому компоненту.

# 2. Этимология и происхождение термина

Термин _higher-order_ восходит к функциональному программированию, в частности к концепции функций высшего порядка (Higher-Order Functions, HOF). HOF — это функция, принимающая другие функции как аргументы или возвращающая их как результат. В экосистеме React эта идея была перенесена на уровень компонентов: вместо преобразования функций общего назначения выполняется преобразование компонентов как единиц UI-абстракции.

Следовательно, название Higher-Order Component подчёркивает не внутреннюю природу самого компонента, а способ его получения через функцию более высокого порядка.

# 3. Историческое развитие и эволюция концепта

Развитие паттерна в экосистеме React прошло через несколько стадий, определяемых эволюцией самой библиотеки:

- **Mixins (устаревший этап):** Изначальный механизм композиции в `React.createClass`. Был признан вредным из-за неявных зависимостей, конфликтов имён в разделяемом пространстве состояний и сложности масштабирования.
- **Период доминирования HOC:** До появления встроенных хуков HOC стал основным универсальным механизмом абстракции и повторного использования логики. В этот период паттерн активно применялся для решения типовых задач: внедрение данных, подключение к внешнему состоянию, добавление подписок, авторизации, feature flags, а также повторное использование container-like логики без наследования. Типичными историческими примерами стали `connect(...)` в React Redux, `withRouter(...)` в ранних поколениях React Router, `injectIntl(...)` в react-intl.
- **Эпоха Custom Hooks (современность):** С выходом React 16.8 акцент сместился в сторону более локального и прозрачного повторного использования логики через функции вида `useX()`. <font color="#d3869b">HOC утратили статус основного универсального механизма, однако не исчезли. В современной практике они применяются точечно, главным образом там, где важно оборачивать именно компонент как целую единицу интерфейса, а не просто переиспользовать его внутреннюю логику вычислений.</font>

# 4. Формальные свойства, структура, ключевые признаки

## Базовая структура и минимальная сигнатура

> Фундаментальная схема HOC фиксирует три ключевых признака: HOC является функцией (а не компонентом), результатом является новый компонент, и этот новый компонент прокидывает входные параметры (props) в обёрнутый узел.

Полноценная минимальная реализация на TypeScript:

```typescript
import React, { ComponentType, FC } from 'react';

function withEnhancement<T extends object>(
  WrappedComponent: ComponentType<T>
): FC<T> {
  const EnhancedComponent: FC<T> = (props) => {
    return <WrappedComponent {...props} />;
  };
  
  EnhancedComponent.displayName = `withEnhancement(${WrappedComponent.displayName || WrappedComponent.name || 'Component'})`;
  return EnhancedComponent;
}
```

## Инвариант непрямого расширения (Чистота)

Корректный HOC не модифицирует исходный компонент напрямую (через прототип или мутации), а создаёт внешний слой поведения. Это делает композицию предсказуемой и сохраняет _referential separation_ (ссылочное разделение) между исходным и расширенным представлением.

## Прозрачность интерфейса (Pass-through)

HOC стремится сохранить внешний контракт обёрнутого компонента. Все пропсы, не относящиеся непосредственно к инфраструктурной логике HOC, должны передаваться в оборачиваемый компонент без искажений. Как только HOC начинает перехватывать, удалять или неявно добавлять пропсы, он изменяет внешний контракт результирующего компонента, что требует строгого описания типов.

## Декоративная природа

HOC выступает формой декларативного декорирования. Он добавляет слои поведения поверх исходной реализации (например: управление доступом, инъекцию данных, логирование, адаптацию пропсов).

## Композиционность

Результат вызова одного HOC является валидным компонентом, что позволяет использовать его как входное значение для другого HOC:

```typescript
const Enhanced = withAuth(withTracking(withTheme(MyComponent)));
```

Данное свойство позволяет поэтапно наращивать поведение, но генерирует риск избыточной глубины дерева (wrapper hell).

## Статическая композиция

Вызов HOC обязан происходить строго вне метода `render` или тела функционального компонента. Динамическое создание HOC внутри цикла рендеринга приводит к уничтожению идентичности компонента (изменению референса) при каждом рендере, что вызывает полную потерю внутреннего состояния и перемонтирование всего поддерева.

## Разделение ролей

В архитектуре HOC чётко выделяются два функциональных слоя:
- **Wrapped component:** предметный компонент, отвечающий за собственное изолированное UI-поведение.
- **Enhancing wrapper:** внешний инфраструктурный слой, отвечающий за внедряемую повторно используемую логику.

# 5. Современное значение и интерпретации

В современной инженерной практике React концепт HOC интерпретируется не как универсальный default-механизм абстракции логики, а как специализированный паттерн компонентной трансформации. Его применение строго локализовано в следующих контекстах:

- Интеграция с библиотеками, архитектурно построенными вокруг компонентных обёрток.
- Необходимость стандартизированно декларировать однотипное внешнее поведение поверх множества компонентов (например, централизованные Error Boundaries или Suspense-обёртки).
- Работа с boundary-like механизмами, где требуется именно внешнее перехватывание контекста дерева.
- Поддержка и миграция legacy React-кода.

В новых проектах HOC конкурирует не с монолитным кодом, а с современными примитивами: Custom Hooks, локальной Component Composition и Render Props.

# 6. Математическое или логико-формальное ядро

В абстрактной форме HOC описывается как оператор отображения над множеством компонентов:
$$H: \mathcal{C} \to \mathcal{C}$$
Где $\mathcal{C}$ — множество всех валидных компонентных представлений в заданной модели React.

При наличии конфигурации или зависимости от внешних параметров (каррирование), форма становится двухступенчатой:
$$H_{config}: \mathcal{P}_{extra} \to (\mathcal{C} \to \mathcal{C})$$
Логически HOC представляет собой оператор расширения контракта рендеринга, способный:
- сохранять исходный интерфейс тождественным;
- расширять интерфейс новыми обязательными или опциональными входами;
- скрывать исходные параметры (если HOC сам удовлетворяет зависимость компонента);
- накладывать математические предусловия на отображение (условный рендеринг).

# 7. Сравнение с соседними понятиями

* **Higher-Order Component и Component Composition**. Component Composition — это фундаментальный, более общий принцип организации интерфейса через комбинирование структуры дерева компонентов. HOC является более узким функциональным паттерном. Композиция размещает узлы внутри друг друга в JSX, тогда как HOC программно производит новый компонентный класс или функцию. Всякий HOC опирается на идею композиции, но не всякая композиция является функцией высшего порядка.
* **Higher-Order Component и Custom Hooks**. Хуки инкапсулируют логику состояния и сайд-эффектов на уровне функций, встраиваясь непосредственно внутрь компонента. HOC же повторно использует поведение через внешнюю трансформацию самого компонента. Хуки обеспечивают более прозрачный поток данных и исключают "wrapper hell", поэтому HOC применяется только тогда, когда логика требует именно внешнего, стандартизированного управления всем компонентом целиком.
* **Higher-Order Component и React.memo**. `React.memo` представляет собой специализированный встроенный API, который технически действует как HOC-подобная обёртка, возвращающая мемоизированную версию компонента для оптимизации рендеринга на основе поверхностного сравнения пропсов. Он не заменяет общую архитектурную модель HOC, а является частным оптимизационным случаем с фиксированной семантикой.

# 8. Практические применения и реализации

HOC применяются в архитектурных сценариях, требующих многократного применения единообразного поведения на границах компонентов.

## Авторизация и контроль доступа (Кондициональный рендеринг)

Реализация HOC, который проверяет наличие прав пользователя перед рендерингом защищенного компонента:

```typescript
import React, { ComponentType, FC } from 'react';

interface AuthContextProps {
  isAuthenticated: boolean;
}

// HOC требует, чтобы оборачиваемый компонент не ожидал props авторизации извне,
// так как HOC сам их обрабатывает, либо просто блокирует рендер.
export function withAuth<T extends object>(
  WrappedComponent: ComponentType<T>
): FC<T & AuthContextProps> {
  const WithAuth: FC<T & AuthContextProps> = ({ isAuthenticated, ...props }) => {
    if (!isAuthenticated) {
      return <div>Access denied: Insufficient privileges.</div>;
    }
    
    // Приведение типов требуется из-за того, что TypeScript не всегда корректно
    // выводит остаточные параметры в дженериках
    return <WrappedComponent {...(props as T)} />;
  };

  WithAuth.displayName = `withAuth(${WrappedComponent.displayName || WrappedComponent.name || 'Component'})`;
  return WithAuth;
}
```

## Инъекция данных и адаптация (Параметризованный HOC / Currying)

Пример HOC, который принимает конфигурацию и модифицирует сайд-эффекты интерфейса (изменение заголовка документа):

```typescript
import React, { ComponentType, FC, useEffect } from 'react';

// Первая функция принимает конфигурацию (title), вторая - компонент
export function withTitle(title: string) {
  return function wrap<T extends object>(WrappedComponent: ComponentType<T>): FC<T> {
    const WithTitle: FC<T> = (props) => {
      useEffect(() => {
        const originalTitle = document.title;
        document.title = title;
        
        return () => {
          document.title = originalTitle;
        };
      }, []);

      return <WrappedComponent {...props} />;
    };

    WithTitle.displayName = `withTitle(${WrappedComponent.displayName || WrappedComponent.name || 'Component'})`;
    return WithTitle;
  };
}
```

## Обработка состояний загрузки

Адаптация интерфейса, централизующая fallback-UI.

```typescript
import React, { ComponentType, FC } from 'react';

interface LoadingProps {
  isLoading: boolean;
}

export function withLoading<T extends object>(
  WrappedComponent: ComponentType<T>
): FC<T & LoadingProps> {
  const WithLoading: FC<T & LoadingProps> = ({ isLoading, ...props }) => {
    if (isLoading) {
      return <div className="spinner">Загрузка данных...</div>;
    }

    return <WrappedComponent {...(props as T)} />;
  };

  WithLoading.displayName = `withLoading(${WrappedComponent.displayName || WrappedComponent.name || 'Component'})`;
  return WithLoading;
}
```

# 9. Типичные ошибки, заблуждения, ограничения

- **Ошибочное отождествление с композицией JSX:** Обёртка компонента внутри JSX-дерева (передача `children`) не делает его HOC. HOC — это строго функция генерации компонента.
- **Нарушение прозрачности props и Prop Collisions:** Если HOC неявно перезаписывает пропсы с именами, которые уже использует целевой компонент (или внешний вызывающий код), происходит потеря данных или изменение семантики. Потеря неиспользуемых пропсов также ломает контракт компонента.
- **Wrapper Hell (Ад обёрток):** Чрезмерная композиция HOC приводит к глубокой вложенности абстракций, усложнению иерархии в React DevTools, затруднению трассировки потока данных и локального рассуждения о поведении (local reasoning).
- **Сложность отладки без displayName:** Потеря или отсутствие явной установки статического свойства `displayName` приводит к тому, что в DevTools отображаются цепочки анонимных компонентов (`<Unknown>`).
- **Потеря ссылок (Ref Loss):** Пропс `ref` не является стандартным параметром и не прокидывается автоматически через rest-оператор (`...props`). Для сквозной передачи ссылок на внутренние DOM-узлы требуется использование `React.forwardRef` внутри HOC.
- **Потеря статических методов:** При оборачивании классового компонента его статические методы не копируются автоматически в HOC-обёртку. Эта проблема исторически решалась библиотекой `hoist-non-react-statics`.
- **Мутация исходного компонента (Антипаттерн):** Прямая модификация прототипа или свойств переданного компонента внутри HOC строго запрещена, так как это нарушает предсказуемость и делает компонент непригодным для повторного использования без этих мутаций.
- **Избыточность абстракции:** Использование HOC там, где логика полностью локальна и может быть выражена через Custom Hooks, приводит к необоснованному усложнению архитектуры.

# 10. Практические рекомендации по проектированию HOC

## *Критерии уместности HOC:

- Требуется применить однотипное, стандартизированное внешнее поведение на уровне контракта компонента (например, Error Boundaries).
- Логика должна быть декларативно применима извне, без модификации внутреннего кода компонента.
- Архитектура требует интеграции с legacy API или строгого разделения слоёв ответственности.

## Антипаттерны применения (когда HOC избыточен):

- Требуемая логика вычислений, состояний или эффектов локальна для одного функционального узла.
- Добавляемое поведение кардинально меняет исходный контракт пропсов компонента.
- Система уже страдает от избыточной вложенности, и логика легко переносится в композицию через `children`.

## Строгие правила реализации:

1. Прокидывать все неизвестные пропсы (`...props`) явно и без потерь.
2. Изолировать и документировать резервируемые HOC пропсы для предотвращения коллизий.
3. Обязательно устанавливать `displayName` с указанием имени исходного компонента.
4. Проектировать механизм `forwardRef`, если HOC предполагает обёртывание узлов, с которыми взаимодействуют через ссылки.
5. Соблюдать чистоту функции-преобразователя (не мутировать входной `Component`).
