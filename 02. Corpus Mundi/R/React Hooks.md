# 01. React Hooks

> <font color="#fb4934">React Hooks</font> — это функциональный интерфейс (API) и фундаментальный механизм библиотеки React, предоставляющий функциональным компонентам и пользовательским хукам (custom hooks) доступ к состоянию, контексту, ссылкам, эффектам, средствам оптимизации и иным возможностям React без необходимости использования классовой архитектуры. 
> * В официальной документации хуки определяются как функции, позволяющие использовать различные возможности React непосредственно из компонентов; при этом встроенные примитивы можно комбинировать для построения собственных, композитных хуков.

> <font color="#fb4934">В строгом инженерном смысле хук (hook)</font> — это специальная точка подключения компонентной логики к внутренней модели рендеринга и согласования (reconciliation) React. Хук не является произвольной вспомогательной функцией: его вызов подчинён строгим формальным ограничениям среды исполнения React и жестко встроен в контракт между декларативным описанием пользовательского интерфейса (UI) и циклом повторных рендеров компонента.

# 2. Этимология и происхождение термина

Термин *hook* в экосистеме React семантически связан с идеей «подключения» или «зацепления» функционального компонента за внутренние механизмы ядра React: состояние, контекст, жизненный цикл синхронизации и другие подсистемы. В ранней официальной формулировке концепта подчеркивалось, что хуки позволяют разработчикам «hook into» (подключиться к) состоянию и абстракциям жизненного цикла из функциональных компонентов. Именно эта метафора закрепила номенклатуру, отражая тот факт, что функции не хранят состояние сами в себе, а обращаются к внешнему по отношению к ним персистентному контексту исполнения.

# 3. Историческое развитие и эволюция концепта

Концепция хуков была впервые представлена в релизе React 16.8 (февраль 2019 года) как инновационный подход к использованию состояния и других возможностей фреймворка без написания классов. При внедрении данного механизма React опирался на стратегию постепенной интеграции (gradual adoption): хуки не отменяли классовые компоненты одномоментно и не требовали немедленного рефакторинга существующей кодовой базы, предлагая вместо этого эволюционный путь миграции.

В процессе исторического развития хуки стали основным и рекомендуемым стандартом описания логики в современном React. Актуальная официальная документация (react.dev) фиксирует современную линию развития фреймворка на версии 19.2 и выстраивает всю модель разработки исключительно вокруг функциональных компонентов, хуков и строгих правил React.

Эволюция инструментария не ограничилась базовым набором из `useState`, `useEffect` и `useContext`. В мажорных релизах React 18 и 19 концептуальная модель была существенно расширена хуками для управления конкурентным рендерингом, откладыванием визуальных обновлений, оптимистическими обновлениями UI, а также работой с действиями (actions) и формами. В частности, React 19 обогатил API такими инструментами, как `useOptimistic` и `useActionState`, а пакет `react-dom` пополнился хуком `useFormStatus`, специально разработанным для сценариев обработки серверных и клиентских действий форм (form actions).

# 4. Формальные свойства, структура и ключевые признаки

## Нормативные правила использования (Rules of Hooks)

Вызов хуков регламентируется строгими синтаксическими и топологическими правилами, которые являются не просто стилистическими рекомендациями, а неотъемлемой частью корректной вычислительной модели React. 

<font color="#fb4934">Хуки могут быть вызваны исключительно:</font>
- На верхнем уровне функционального компонента React.
- На верхнем уровне внутри другого пользовательского хука (custom hook).

Официальная спецификация категорически запрещает вызов хуков:
- Внутри циклов (`for`, `while`).
- Внутри условных операторов (`if/else`).
- Внутри вложенных функций и замыканий.
- Внутри обработчиков событий (event handlers).
- Внутри блоков обработки исключений (`try/catch/finally`).
- Внутри обычных (не React) JavaScript-функций.
- Внутри классовых компонентов.

<font color="#d3869b">Причина данных ограничений носит формальный алгоритмический характер:</font> ядро React опирается на детерминированный порядок вызовов хуков для корректного сопоставления фрагментов состояния между последовательными рендерами. На каждой итерации рендеринга React ожидает идентичный набор вызовов хуков в строго неизменной последовательности. Нарушение этого инварианта приводит к разрушению внутреннего связного списка и потере соответствия между конкретным вызовом функции-хука и выделенной для него ячейкой состояния.

## Чистота и идемпотентность

> Современная парадигма React требует соблюдения принципа чистоты для компонентов и хуков. Компонент трактуется как математически чистая функция в том смысле, что для эквивалентных входных данных (`props`, `state`, `context`) он обязан возвращать идентичный результат вычислений (дерево элементов). 

<font color="#d3869b">Ключевые свойства, вытекающие из требования чистоты:</font>
- Побочные эффекты (side effects) не должны инициироваться или выполняться в фазе рендеринга (render phase).
- Входные параметры (`props`, `state`), а также значения, передаваемые в качестве аргументов в хуки, рассматриваются как неизменяемые (immutable) снимки (snapshots) в рамках конкретного цикла рендеринга.
- Хук управляет состоянием и процессом синхронизации исключительно декларативно; сам вызов хука в теле компонента не должен становиться точкой произвольной императивной мутации. 

Императивные операции строго изолируются и допускаются только в специально отведенных абстракциях (escape hatches), к которым относятся эффекты и операции с мутабельными ссылками (refs).

# 5. Классификация базовых встроенных хуков

С архитектурной точки зрения официальная документация React группирует встроенные API-примитивы по их функциональной роли в системе.

- **State Hooks (Хуки состояния)**: `useState` и `useReducer`. Предназначены для сохранения и модификации локального состояния компонента. `useState` применяется для прямого обновления скалярных или простых объектных значений, в то время как `useReducer` оптимизирован для управления сложным состоянием, где логика переходов детерминированно выражается через функцию-редуктор (reducer).
- **Context Hooks (Хуки контекста)**: `useContext`. Обеспечивает прямое чтение и подписку на изменения значений из древовидного контекста (React Context), позволяя получать данные от удаленных узлов-провайдеров без необходимости ручного проброса свойств (prop drilling).
- **Ref Hooks (Хуки ссылок)**: `useRef` и `useImperativeHandle`. 
	- Предназначены для хранения мутабельных данных, изменение которых не должно инициировать цикл повторного рендеринга (например, ссылок на DOM-узлы или идентификаторов таймеров). 
	- Хук `useImperativeHandle` используется для тонкой настройки императивного API, экспортируемого родительскому компоненту через механизм ссылок (зачастую в связке с `forwardRef`).
- **Effect Hooks (Хуки эффектов)**: `useEffect`, а также специализированные `useLayoutEffect`, `useInsertionEffect` и `useEffectEvent`. Служат механизмом синхронизации внутреннего состояния компонента с внешними (по отношению к React) системами: сетевыми запросами, браузерным DOM, анимациями или сторонними библиотеками. Документация акцентирует внимание на том, что эффекты являются «люками спасения» (escape hatches) и не предназначены для оркестрации потока данных внутри чистого React-приложения.
- **Performance Hooks (Хуки производительности)**: `useMemo`, `useCallback`, `useTransition`, `useDeferredValue`. 
	- Хуки `useMemo` и `useCallback` обеспечивают мемоизацию (кэширование) ресурсоемких вычислений и экземпляров функций для сохранения ссылочной идентичности. 
	- Хуки `useTransition` и `useDeferredValue` управляют конкурентным рендерингом, разделяя обновления на высокоприоритетные (срочные) и низкоприоритетные (фоновые).
- **Other Hooks (Специализированные хуки)**: `useDebugValue`, `useId`, `useSyncExternalStore`, `useActionState`. Группа включает инструменты для разработки библиотек, обеспечения доступности (генерация уникальных ID), надежной подписки на внешние (не-React) хранилища данных и управления состоянием асинхронных действий.
- **Environment-specific Hooks (Платформенно-зависимые хуки)**: Инструменты, экспортируемые из пакета `react-dom` (например, `useFormStatus`). Эти хуки жестко связаны с браузерной средой исполнения, опираются на DOM-события веб-форм и недоступны в иных средах (iOS, Android в React Native или desktop-native runtime).

# 6. Архитектурные границы: состояние, эффекты, ссылки и переиспользование логики

Центральной концептуальной задачей хук-модели является строгое разведение различных типов логики по соответствующим абстракциям.

- **Локальное состояние**: Описывает данные, мутация которых должна приводить к повторному рендерингу интерфейса. Для этой задачи предназначены исключительно `useState` и `useReducer`.
- **Ссылки (Refs)**: Если значение должно персистентно сохраняться между циклами рендеринга, но его обновление не требует перерисовки UI, архитектурно корректной абстракцией выступает `ref`, а не `state`. Различие фундаментально: `state` изменяет декларативное представление компонента, `ref` хранит техническую, побочную или императивную информацию.
- **Побочные эффекты (Effects)**: Применяются исключительно для синхронизации с внешней системой (DOM API, таймер, веб-сокет). Эффект не является альтернативным инструментом для реализации внутренней бизнес-логики. Если вычисление можно выполнить в процессе чистого рендеринга на основе существующих пропсов и состояния, использование эффекта является антипаттерном.
- **Повторное использование логики**: Если требуется внедрить не новый фундаментальный тип состояния, а переиспользовать существующую логику с сохранением состояния (stateful logic), применяется паттерн custom hooks. 

Данное разделение ответственности демонстрирует, что React Hooks выступают не просто набором утилитарных функций, а строгим структурным принципом организации логики.

# 7. Паттерн Custom Hooks и инкапсуляция логики

> <font color="#fb4934">Пользовательские хуки (Custom Hooks)</font> — это механизм абстракции, представляющий собой стандартные функции JavaScript, внутри которых вызываются встроенные или другие пользовательские хуки (с соблюдением общих правил). Согласно спецификации, имена таких функций обязаны начинаться с префикса `use`, за которым следует идентификатор в формате PascalCase. Это синтаксическое соглашение обеспечивает возможность статического анализа кода (через линтеры) и визуального распознавания хуков разработчиками.

Принципиальные характеристики паттерна:
- **Изоляция состояния**: Пользовательские хуки инкапсулируют и переиспользуют *структуру логики*, но не делят экземпляр состояния между компонентами. Каждое обращение к custom hook в новом компоненте инициализирует полностью независимый граф состояний и эффектов. Для реализации общего (разделяемого) состояния необходимо поднимать его (lifting state up) или использовать контекст/внешнее хранилище.
- **Чистота исполнения**: Как и компоненты, custom hooks должны оставаться чистыми функциями. Они повторно выполняются при каждом рендере компонента-потребителя. Таким образом, custom hook не является фоновым процессом, демоном или контейнером однократной инициализации — это композиционная форма описания реактивного потока данных.

# 8. Современное значение и интерпретации

В современной экосистеме React хуки трансформировались из альтернативы классовым компонентам в базовую, неделимую единицу описания логики интерфейса. Архитектурная декомпозиция сменила вектор: вместо вынужденной группировки разнородного кода по фазам жизненного цикла компонента (как в `componentDidMount` или `componentDidUpdate`), логика теперь группируется по единому функциональному назначению (подписка на конкретный сокет, управление конкретной формой, анимация элемента).

Современная трактовка тесно интегрирована с развитием парадигм конкурентности и серверных действий. В React 19 хуки выходят за рамки простого локального State Management, предоставляя API для координации переходов (transitions), оптимистического рендеринга (optimistic UI) и комплексной обработки форм. Это знаменует трансформацию хуков в универсальный интерфейс доступа к вычислительному ядру React.

Важной вехой современного этапа является введение хука `use` (в React 19.2), который обладает статусом исключения из классических правил. В отличие от остальных API, вызов `use` разрешен внутри условных операторов и циклов (хотя по-прежнему запрещен внутри блоков `try/catch` и вне контекста React). Данное исключение не отменяет фундаментальные Rules of Hooks для остального множества API, а фиксирует появление узкоспециализированного инструмента для чтения асинхронных ресурсов и контекста в конкурентной среде.

# 9. Математическое и логико-формальное ядро

Формальную модель работы хуков можно представить как детерминированное отображение между позициями вызовов функций внутри лексического тела компонента и внутренними ячейками памяти (state cells) на уровне Fiber-узла React.

Пусть $H = (h_1, h_2, \dots, h_n)$ — упорядоченная последовательность вызовов хуков при выполнении рендера $R_i$. 

Алгоритмическая корректность механизма требует соблюдения строгого инварианта: для любого последующего рендера $R_{i+1}$ длина последовательности $|H|$ и относительный позиционный порядок каждого вызова $h_k$ должны оставаться константой. Любой условный пропуск или динамическое добавление $h_k$ разрушает топологию сопоставления, приводя к тому, что хук $h_{k+1}$ получит доступ к состоянию, выделенному для $h_k$. Именно эта формальная математическая зависимость объясняет категорический запрет на использование хуков в условиях и циклах.

В рамках этой логической модели:
- **State hook** формализуется как функция доступа (getter/setter) к персистентной реактивной ячейке памяти.
- **Effect hook** — как декларация отложенной процедуры синхронизации с внешней средой.
- **Ref hook** — как доступ к стабильному указателю на контейнер, изолированный от графа зависимостей цикла рендеринга.

Такое строгое логическое разделение позволяет анализировать жизненный цикл компонента не как последовательность событий, а как проекцию неизменного графа состояний.

# 10. Практические применения

В прикладной программной инженерии хуки покрывают весь спектр задач по оркестрации интерфейса:

- **Локальное состояние**: Управление интерактивными элементами UI (ввод в формы, переключатели состояния, открытие модальных окон, фильтры данных, пагинация, шаги мастеров настройки, индикаторы загрузки и обработка локальных ошибок).
- **Сквозная передача данных**: Использование `useContext` для внедрения зависимостей, проброса глобальных тем оформления, локалей (i18n), данных авторизованной сессии, конфигурационных объектов или сервисных фасадов на любые уровни дерева компонентов.
- **Синхронизация с окружением**: Применение `useEffect` для инициализации сетевых соединений (WebSocket), интеграции с браузерными API (Intersection Observer, Resize Observer), работы со сторонними не-React виджетами, подписки на события глобального объекта `window`, управления таймерами и сложной CSS-in-JS инфраструктурой.
- **Оптимизация (Performance Tuning)**: Локальное кэширование результатов ресурсоемких математических операций, стабилизация идентичности (referential equality) callback-функций для предотвращения каскадного рендеринга дочерних компонентов-мемоизаторов, а также откладывание визуально второстепенных частей интерфейса для сохранения отзывчивости потока ввода (60 FPS).
- **Стандартизация абстракций**: Использование custom hooks для создания переиспользуемых модулей (например, `useMediaQuery` для адаптивного дизайна, `useDebounce` для задержки ввода, `useForm` для валидации данных, инкапсуляция логики обработки горячих клавиш).

# 11. Типичные ошибки, заблуждения и ограничения

Разработка с использованием хук-модели сопряжена с рядом классических антипаттернов:

- **Нарушение топологии вызовов**: Условный или непоследовательный вызов хуков. Это критическая ошибка, которая разрушает позиционную модель сопоставления состояния и ведет к фатальным несоответствиям между вызываемыми функциями и внутренними ячейками памяти React (state cells).
- **Злоупотребление эффектами (Effect Overuse)**: Ошибочное восприятие `useEffect` как универсального контейнера для выполнения любой логики после рендера или инструмента для вычисления производных данных (derived state). Официальная документация строго ограничивает семантику эффекта задачей синхронизации с внешней системой. В отсутствие внешней системы применение эффекта часто избыточно и вредоносно.
- **Ошибочная трактовка разделения состояния в Custom Hooks**: Популярное заблуждение о том, что вызов пользовательского хука автоматически создает глобальное (shared) состояние между несколькими компонентами. В действительности распределяется исключительно структура логики, в то время как инстанс состояния каждого вызова остается строго изолированным.
- **Преждевременная или избыточная мемоизация**: Использование `useMemo` и `useCallback` в качестве стандартного синтаксического шаблона для всех вычислений и функций. Данные инструменты не являются базовым ритуалом написания кода; их применение должно быть обосновано метриками профилирования, реальной стоимостью вычислений или строгими требованиями к стабильности ссылок (referential stability).
- **Ограничения переносимости (Платформенная зависимость)**: Не все хуки обладают свойством универсальной абстракции. Примитивы, экспортируемые из `react-dom` (например, `useFormStatus`), жестко связаны с браузерными спецификациями и нарушают концепцию платформенной независимости (Learn Once, Write Anywhere), будучи неприменимыми в средах отличных от веба.

# 12. Связанные понятия

- [[React]]: родительская экосистема и библиотека, в рамках которой хуки выступают базовой архитектурной парадигмой.
- `[[Функциональные компоненты (Functional Components)]]`: единственный легитимный контекст (наряду с пользовательскими хуками) для декларации и исполнения хуков.
- `[[Управление состоянием (State Management)]]`: концептуальная область, часть которой покрывается локальными хуками состояния (`useState`, `useReducer`), но которая не исчерпывается ими на уровне архитектуры всего приложения.
- `[[React Context]]`: инфраструктурный механизм внедрения зависимостей, мостом к которому в функциональной парадигме выступает хук `useContext`.
- `[[Побочные эффекты (Effects)]]`: отдельная категория операций, направленная на интеграцию реактивного дерева с императивным внешним миром (DOM, сеть, подписки).
- `[[Custom Hooks]]`: структурный паттерн проектирования, специально созданный для инкапсуляции и композиции stateful logic.
- `[[Refs (Ссылки)]]`: выделенная подсистема escape hatch-механизмов, предназначенная для персистентного хранения мутабельных данных и прямого взаимодействия с императивными API в обход декларативного цикла рендеринга.
- `[[Оптимизация производительности React (React Performance Optimization)]]`: комплекс техник, опирающийся на инструменты мемоизации и приоритизации потоков обновления (`useMemo`, `useTransition` и др.).
