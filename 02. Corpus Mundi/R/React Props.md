# 1. Props in React

> <font color="#fb4934">Props (сокращение от <i>properties</i>)</font> — это иммутабельный объект, представляющий собой формализованный механизм передачи входных данных в компонент со стороны его вызывающего контекста (прежде всего со стороны родительского компонента) в архитектуре React.

* <b>В строгом архитектурном смысле `props` образуют внешний интерфейс компонента:</b> они задают набор входных параметров, через которые компонент получает данные, конфигурацию и точки композиции, необходимые для вычисления своего результата рендеринга. 
	* `Props` описывают внешне переданные значения, параметризуя поведение и визуальное представление, при этом не превращая сам компонент в источник собственного независимого или изменяемого состояния. 
	* Компонент может использовать `props` при вычислении пользовательского интерфейса (UI), но не должен рассматривать их как локально мутируемое хранилище.

# 2. Этимология и происхождение термина

Термин `props` является исторически закрепившимся сокращением от английского слова _properties_ (свойства). В экосистеме React он утвердился как базовое обозначение набора свойств, передаваемых компоненту при его декларативном использовании в синтаксисе JSX или при прямом императивном создании элемента через React API (например, `React.createElement`). Семантически данное название подчеркивает, что компонент получает не произвольный поток команд или инструкций, а строго структурированный набор именованных входных характеристик.

# 3. Историческое развитие и эволюция концепта

Механизм передачи свойств заложен в основу компонентной модели React с момента ее создания. Исторически `props` всегда выступали центральным механизмом конфигурирования компонентов и композиции UI, однако форма их утилизации эволюционировала вместе с парадигмами разработки:

- **Этап фабрик (React.createClass, до ES6):** Свойства передавались в фабрику компонентов при инициализации и были доступны через контекст созданного экземпляра как `this.props`.
- **Этап классовых компонентов (ES6 Classes):** Свойства явно инициализировались в конструкторе класса через вызов `super(props)` и оставались жестко привязаны к экземпляру `this`. Это требовало от инженеров строгого контроля за контекстом выполнения (`this` binding) при передаче методов.
- **Этап функциональных компонентов (React 16.8+):** С переходом к доминированию функциональных компонентов и хуков, концепт `props` редуцировался до своего наиболее чистого выражения. Компонент стал концептуально интерпретироваться как функция от входных параметров, а `props` — как её первый и основной аргумент.

Со временем в инженерной культуре React вокруг `props` сложилась более строгая интерпретация, где они:
- рассматриваются как внешний неизменяемый контракт компонента;
- связываются с парадигмой однонаправленного потока данных сверху вниз;
- выступают фундаментальной основой для композиции, инкапсуляции логики и локализации зоны ответственности между структурными блоками приложения.

# 4. Границы темы и место в модели React

К семантическим границам темы `props` относится форма представления входных данных компонента, их архитектурная роль в композиции, методы конфигурирования и фундаментальная организация потока данных между узлами компонентного дерева.

На границе данной темы находятся концепты, требующие строгого разграничения:
- `state`: внутренние эндогенные данные, которые принадлежат самому компоненту, инициализируются им и могут изменяться через встроенные механизмы React. `Props`, напротив, являются экзогенными (приходят извне).
- `context`: механизм нелокальной (неявной) передачи значений через дерево компонентов, позволяющий обходить промежуточное прокидывание свойств. `Props` остаются строгим, локальным и явным способом передачи данных.

Важно на уровне терминологии различать два уровня абстракции:
- **Props пользовательского компонента:** непосредственно входные данные (аргумент), которые получает функция или класс компонента при вызове.
- **Props элемента React:** объект свойств, связанный с виртуальным узлом (React Element), который формируется на этапе трансляции (например, из JSX-атрибутов). Для пользовательского компонента этот объект впоследствии и становится фактическим входом его вызова в фазе рендеринга.

# 5. Формальные свойства, структура и ключевые признаки

Механизм `props` опирается на ряд строгих инвариантов и структурных характеристик:
- **Внешний характер (Экзогенность):** `Props` всегда принадлежат внешнему интерфейсу компонента. Они определяются и вычисляются не внутри самого компонента, а исключительно в месте его декларативного вызова (в родительском контексте).
- **Однонаправленность передачи (Top-down Data Flow):** В стандартной топологии React данные передаются строго сверху вниз — по направлению от родителя к потомку. Обратная координация возможна через инверсию контроля (передачу callback-функций), но сам поток передачи входных значений остается строго нисходящим.
- **Логическая неизменяемость (Immutability):** В инженерной модели React `props` рассматриваются как _read-only_ вход. Компонент не имеет права изменять собственные `props`. Прямая мутация нарушает детерминированность рендеринга и вызывает побочные эффекты.
- **Интерфейсная роль:** `Props` действуют как публичный API модуля. Они ближе к сигнатуре контракта, описывающего ожидания компонента, чем к его внутренней памяти.
- **Декларативность:** Вызывающий код посредством `props` не инструктирует компонент императивно (пошагово), а декларативно задает срез данных и параметров, опираясь на которые компонент самостоятельно вычисляет финальный UI.
- **Композиционная функция:** `Props` являются клеем для соединения атомарных компонентов в сложные структуры, позволяя родителю передавать данные, а дочернему компоненту — интерпретировать их в пределах своей доменной ответственности.
- **Структурированность и категоризация:** На практике `props` представляют собой сериализуемый (в большинстве случаев) именованный объект. Его свойства типизируются на следующие категории:
    - Данные для отображения (доменные сущности): `title`, `price`, `user`.
    - Параметры поведения и визуального представления: `disabled`, `variant`, `size`.
    - Функции обратного вызова (интерфейсы взаимодействия): `onClick`, `onSubmit`.
    - Точки вложения и пространственной композиции: зарезервированное свойство `children`.
- **Зарезервированные свойства:**
    - `children`: специальное свойство для получения вложенного содержимого (React-элементов), демонстрирующее способность `props` передавать не только скалярные примитивы, но и фрагменты декларативного интерфейса.
    - `key`: служебный идентификатор для алгоритма согласования (Reconciliation), недоступный для прямого чтения внутри дочернего компонента.

# 6. Современное значение и интерпретации

В современной инженерной практике `props` понимаются не просто как пассивный технический контейнер для транспортировки значений, а как мощное средство проектирования компонентных границ. Степень универсальности компонента прямо пропорциональна качеству проектирования его `props`. Чем точнее и абстрактнее определен входной интерфейс, тем проще:

- переиспользовать компонент в различных бизнес-контекстах;
- тестировать его в строгой изоляции (unit-тестирование чистых компонентов);
- встраивать его в композицию более высокого уровня;
- ограничивать и контролировать область ответственности компонента (Single Responsibility Principle).

Современная академическая трактовка также корректирует терминологию. Выражение «изменились props» признается техническим сокращением. Корректная формулировка гласит: _родительский компонент был заново вычислен в ходе фазы рендеринга, сформировал новую конфигурацию (React Element) дочернего узла, вследствие чего дочерний компонент получил следующий детерминированный набор входных значений_. Сам по себе объект `props` не является самостоятельным триггером реактивности вне общего контекста обновления дерева.

# 7. Математическое или логико-формальное ядро

На формальном уровне компонентную абстракцию React можно представить как математическое отображение (функцию), транслирующее входные переменные в структуру пользовательского интерфейса:

$$Component : Props \times State \times Context \rightarrow UI$$

Где:
- $Props$ — внешний экзогенный вход компонента;
- $State$ — внутреннее эндогенное состояние (если компонент является stateful);
- $Context$ — внешний нелокальный вход, проецируемый из вышестоящих провайдеров дерева;
- $UI$ — декларативное описание результата вычисления (структура React Elements).

Для чистого функционального компонента (stateless, без подписки на контекст) модель редуцируется до базового уравнения рендеринга:

$$UI = f(P)$$

Это уравнение выражает фундаментальное архитектурное назначение `props` ($P$): они являются строго заданной областью определения функции на текущем цикле рендеринга. Несмотря на то, что React-компонент не является абсолютно "чистой" математической функцией в универсальном смысле (ввиду возможности сайд-эффектов через хуки), именно данная нотация наиболее точно описывает предсказуемость рендеринга при детерминированном контракте `props`.

# 8. Архитектурные паттерны передачи данных

Вокруг механизма `props` сформировались устойчивые архитектурные паттерны, расширяющие возможности композиции:
- **Инверсия контроля (Callback Props):** Передача исполняемой функции в качестве свойства. Это позволяет дочернему компоненту, не владеющему состоянием, делегировать обработку события логического ветвления родительскому компоненту.
- **Render Props (Function as Child / Props):** Паттерн передачи функции, возвращающей React-элемент, в качестве свойства. Применяется для внедрения динамического отложенного рендеринга и совместного использования сложной инкапсулированной логики поведения (stateful logic) без жесткой привязки к конкретной визуальной разметке.
- **Spread Attributes:** Деструктуризация и сквозная трансляция объекта свойств (`{...props}`) на уровень нижестоящего узла. Критически важный паттерн при разработке компонентов-оберток (Higher-Order Components), полиморфных UI-примитивов и адаптеров.

# 9. Практические применения

Теоретические принципы `props` реализуются в архитектуре приложений через ряд типовых сценариев:

## Конфигурирование переиспользуемых компонентов:

`Props` обеспечивают отвязку компонента от конкретного набора данных. Компонент динамически меняет представление в зависимости от переданных параметров.

```JavaScript
function UserCard({ name, role }) {
  return (
    <section>
      <h2>{name}</h2>
      <p>{role}</p>
    </section>
  );
}
```

Значения `name` и `role` выступают экзогенными факторами, определяющими конкретный экземпляр представления.

## Организация однонаправленного потока данных (Data Flow):

Родитель владеет состоянием, а потомок отвечает исключительно за его презентацию.

```JavaScript
function ProfilePage() {
  const user = { name: "Anna", role: "Editor" };

  return <UserCard name={user.name} role={user.role} />;
}
```

В данном паттерне источник истины (данных) и место их визуального отображения разведены по разным уровням абстракции дерева.

## Передача поведения через callback props:

Реализация интерактивности без нарушения принципа единственной ответственности.

```JavaScript
function SaveButton({ onSave, disabled }) {
  return (
    <button onClick={onSave} disabled={disabled}>
      Save
    </button>
  );
}
```

Компонент предоставляет абстрактный интерфейс вызова (`onSave`), оставаясь агностичным к бизнес-логике самой операции сохранения.

## Композиция через children:

Построение пространственных контейнеров и макетов (Layouts).

```JavaScript
function Panel({ title, children }) {
  return (
    <section>
      <h2>{title}</h2>
      <div>{children}</div>
    </section>
  );
}
```

`Panel` получает конфигурацию (`title`) и готовое поддерево UI (`children`), выступая каркасом абстрактного слота, что является квинтэссенцией декларативной композиции.

# 10. Сравнение с соседними понятиями

Академическое понимание `props` требует четкой демаркации со смежными концептами React:
- **Props и State:** Главный критерий различения — происхождение и принадлежность (ownership). `Props` приходят извне и выражают публичный интерфейс. `State` принадлежит исключительно самому компоненту, инициализируется имманентно и представляет собой его изменяемую внутреннюю память.
- **Props и Context:** Критерий различения — локальность контракта. `Props` передаются строго явно, от родителя к непосредственному потомку, формируя локальный контракт. `Context` реализует пространственное вещание, позволяя проецировать данные глубоко в дерево в обход промежуточных узлов, размывая локальную изоляцию.
- **Props и стандартные параметры функции JavaScript:** В функциональных компонентах `props` физически являются аргументом функции. Однако архитектурно они сложнее: они интегрированы в реактивный жизненный цикл, алгоритм `reconciliation` и декларативную модель построения виртуального DOM, что отличает их от абстрактного входного параметра изолированного скрипта.

# 11. Типичные ошибки, заблуждения, ограничения

В процессе инженерной эксплуатации `props` выявляется ряд критических антипаттернов и заблуждений:
- **Фундаментальное заблуждение (Props = State):** Ошибочная интерпретация `props` как внутреннего состояния компонента. Получение данных на вход не делает компонент владельцем этих данных.
- **Заблуждение об инициативе обновления (Дочерний компонент "меняет" props):** Непонимание однонаправленного потока. Дочерний узел концептуально неспособен изменить `props`. Родительский контекст при новом цикле вычисления предоставляет новый набор константных значений.
- **Заблуждение о реактивности (Props автоматически означают рендер):** Технически некорректная формулировка. Обновление инициируется повторным проходом (re-render) родителя, который генерирует новый `React Element` для потомка. Дальнейшее обновление потомка зависит от фазы `reconciliation` и наличия мемоизации (`React.memo`).
- **Мутация объекта Props (Критическая ошибка):** Прямое императивное переопределение свойств внутри функции (например, `props.value = 10`). Приводит к нарушению целостности данных, игнорируется алгоритмом диффинга React и строго запрещено в React Strict Mode.
- **Антипаттерн "Дублирование источников истины" (Stale State):** Инициализация локального состояния компонента на основе входящих `props` без реализации механизмов синхронизации. При обновлении `props` родительским компонентом, потомок продолжает рендерить устаревшее локальное состояние.
- **Ошибка проектирования (Слишком широкий интерфейс):** Создание компонента с избыточным, слабо структурированным набором `props` (десятками параметров). Это является маркером нарушения архитектурных границ и размытия ответственности компонента.
- **Ограничение масштабируемости (Prop Drilling):** Процесс сквозного прокидывания свойств через глубокую иерархию транзитных компонентов, которым эти данные не требуются для собственного рендеринга. Приводит к избыточной связности (high coupling) и усложняет рефакторинг структурных блоков.

# 12. Связанные понятия

Родительская тема:
[[React Components]] — фундаментальная абстракция, внутри которой `props` реализуют механизм входного контракта.

Связанные заметки:
- `[[Functional Components]]` — парадигма, где концепт `props` тождественен входному аргументу чистой функции.
- `[[Component Composition]]` — архитектурный подход, базирующийся на передаче элементов интерфейса через `props.children`.
- [[React Rendering Model]] — общая система жизненного цикла, где `props` выступают экзогенной переменной в уравнении формирования UI.
- `[[React State]]` — эндогенный антипод экзогенных `props`.
- `[[React Context]]` — паттерн преодоления локальности `props` для распределенных систем.

# 13. Краткий вывод

`Props` в React представляют собой формализованный внешний интерфейс компонента, посредством которого он получает иммутабельные входные данные, параметры конфигурации и элементы пространственной композиции из вызывающего родительского контекста. Будучи математическим аргументом в функции вычисления пользовательского интерфейса, `props` гарантируют поддержание архитектуры однонаправленного потока данных, обеспечивают предсказуемость декларативного рендеринга и строго отделяют экзогенный вход компонента от его эндогенного состояния (`state`). Таким образом, `props` следует рассматривать не как абстрактное хранилище значений внутри узла, а как фундаментальный контракт, регламентирующий интеграцию компонента в общую топологию React-приложения.
