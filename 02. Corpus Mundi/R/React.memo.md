# 1. React.memo

> <font color="#fb4934">React.memo</font> — это API библиотеки React, представляющее собой функцию высшего порядка с сигнатурой `memo(Component, arePropsEqual?)`. Данный интерфейс возвращает мемоизированную версию переданного функционального компонента, что позволяет алгоритму согласования (reconciliation) React пропускать повторный рендер этого компонента в ситуациях, когда родительский узел перерисовывается, но входные свойства (`props`) целевого компонента остаются неизменными согласно установленному правилу сравнения. По умолчанию алгоритм сравнения осуществляет проверку каждого свойства с его предыдущим значением посредством метода `Object.is`.

В инженерном контексте `React.memo` относится исключительно к модели повторных рендеров компонентов. Он отвечает на архитектурный вопрос о том, должен ли уже мемоизированный дочерний узел участвовать в очередном проходе рендеринга при сохранении идентичности входных данных. Следовательно, `React.memo` является оптимизацией рендеринга на границе компонента, а не семантической гарантией отсутствия вызова компонента при любом обновлении дерева, и не выступает механизмом кеширования произвольных вычислений внутри тела функции компонента. Для кеширования вычисленных значений между рендерами применяется `useMemo`, а для сохранения изменяемой ссылки, не участвующей в жизненном цикле рендеринга, — `useRef`.

```typescript
import { memo, NamedExoticComponent, ComponentType } from 'react';

const MemoizedComponent = memo(SomeComponent, arePropsEqual?)

function memo<P extends object>(
  Component: ComponentType<P>,
  arePropsEqual?: (prevProps: Readonly<P>, nextProps: Readonly<P>) => boolean
): NamedExoticComponent<P>;
```

**Аргументы (Входные параметры)**

- **`Component`**: Целевой функциональный компонент, подлежащий мемоизации. В качестве значения принимается валидная функция компонента React (чистая функция отображения). Применение `React.memo` к классовым компонентам недопустимо (для них используется `React.PureComponent`).
- **`arePropsEqual` (опционально)**: Функция-предикат, замещающая стандартный алгоритм поверхностного сравнения (`Object.is`) кастомной эвристикой оценки эквивалентности свойств:
    - `prevProps`: Объект свойств (props), с которыми компонент был отрендерен на предыдущем такте согласования.
    - `nextProps`: Объект свойств, переданный родительским компонентом для текущего такта обновления.
    - Требование к возвращаемому значению: Предикат обязан вернуть `true`, если `nextProps` приводят к идентичному результату рендеринга по сравнению с `prevProps` (сигнал к пропуску рендера), и `false`, если компонент требует перестроения.

**Возвращаемое значение**

- **`MemoizedComponent`**: Новый, мемоизированный React-компонент (внутренне классифицируемый React как `ExoticComponent`). Вызов не мутирует исходную функцию `Component` по месту. Возвращаемый объект выступает в роли прокси-обёртки, которая встраивается в дерево виртуального DOM (VDOM) и инкапсулирует логику условного перехвата фазы рендеринга на основе оценки входящих контрактов (`props`).

# 2. Историческое развитие и эволюция концепта

Концептуально `React.memo` продолжает стратегию оптимизации производительности, которая в архитектуре классовых компонентов (class components) реализовывалась через базовый класс `React.PureComponent`. Фундаментальная идея обоих подходов заключается в предотвращении избыточных циклов рендеринга, если внешние зависимости компонента не претерпели изменений, способных повлиять на итоговое отображение. В классовой парадигме эта логика выражалась через автоматическое поверхностное сравнение (shallow comparison) объектов `props` и `state` в методе жизненного цикла `shouldComponentUpdate`. С переходом к парадигме функциональных компонентов (function components) эта ответственность была вынесена в специализированную HOC-обёртку `memo(...)`.

В современной экосистеме React парадигма ручной мемоизации претерпевает изменения в связи с развитием React Compiler. Официальная спецификация указывает, что при активном компиляторе необходимость в явном использовании `React.memo` существенно снижается, так как компилятор на этапе сборки способен автоматически анализировать зависимости и внедрять эквивалентные инструкции для пропуска лишних рендеров. Ввиду этого, на текущем этапе развития технологии `React.memo` интерпретируется не как универсальная или обязательная архитектурная норма, а как специализированный инструмент ручной и точечной оптимизации, применяемый в узлах, где автоматическая оптимизация недоступна, отключена или признана недостаточной.

# 3. Место в модели рендеринга React

Согласно стандартной модели рендеринга React, обновление состояния родительского компонента инициирует рекурсивный рендер всех его потомков. `React.memo` внедряется непосредственно в этот механизм маршрутизации обновлений: он создаёт перехватывающий слой на границе конкретного дочернего компонента. На этом рубеже выполняется проверка эквивалентности предыдущих и новых `props`. В случае подтверждения равенства, рекурсивный обход для данной ветви прерывается, и повторный вызов функции дочернего компонента пропускается. Таким образом, `React.memo` оперирует не внутри вычислительного контекста компонента, а в пространстве между родительским обновлением и потенциальным вызовом дочернего узла.

Из данной архитектурной позиции вытекает строгая граница применимости. `React.memo` не предназначен для мемоизации производных значений (derived values), результатов фильтрации, сортировки или конструирования объектов-конфигураций. Тем не менее, он находится в сильной синергической связи с хуками `useMemo` и `useCallback`. Мемоизация границы компонента теряет практический смысл, если родительский узел при каждом собственном рендере генерирует новые ссылочные типы данных (объекты, массивы, функции) и передаёт их вниз по дереву, перманентно нарушая условие равенства `props` по ссылке.

# 4. Формальные свойства, структура и ключевые признаки

Поведение и структура `React.memo` строго определяются следующими инвариантами и свойствами:

- **Иммутабельность инициализации**: Вызов `React.memo` возвращает новый, мемоизированный компонент-обёртку, не мутируя исходную функцию компонента по месту. Это архитектурный паттерн декоратора, а не изменение внутренних механизмов исполнения оригинального кода.
- **Изоляция области действия**: Область ответственности мемоизации строго ограничена свойствами (`props`), поступающими от родительского интерфейса. Изменение внутреннего состояния компонента (через `useState` или `useReducer`) гарантированно инициирует повторный рендер, игнорируя мемоизацию обёртки. Аналогично, если компонент является подписчиком контекста через `useContext`, обновление предоставляемого значения контекста спровоцирует рендер узла; `React.memo` не обладает полномочиями блокировать доставку актуального контекста.
- **Поверхностное сравнение по умолчанию (Shallow Equality)**: Встроенный компаратор не осуществляет глубокий рекурсивный обход структур данных (deep compare). Оценка производится путём сопоставления каждого переданного свойства с его предыдущим значением посредством алгоритма `Object.is`. Следовательно, примитивные типы данных с одинаковыми значениями признаются эквивалентными, в то время как заново созданные объекты, массивы или функции всегда отклоняются компаратором, даже при абсолютной структурной идентичности.
- **Эвристический контракт поведения**: `React.memo` классифицируется как эвристика оптимизации, а не строгий семантический контракт. Спецификация React оставляет за планировщиком право в исключительных случаях инициировать рендер мемоизированного компонента даже при неизменных `props` (например, для освобождения памяти). Разработка логики компонента, опирающаяся на предположение о 100% блокировке рендеров, является антипаттерном.
- **Пользовательская предикация (Custom Comparator)**: API допускает передачу функции-компаратора вторым аргументом: `arePropsEqual(oldProps, newProps)`. Данный предикат обязан возвращать `true` исключительно в том случае, когда новый набор свойств детерминированно приводит к идентичному визуальному выводу и сохраняет прежнее наблюдаемое поведение компонента. Компаратор должен выражать не просто подобие структур, а функциональную эквивалентность компонента на текущем шаге согласования.

# 5. Математическое или логико-формальное ядро

Логику принятия решения о допуске к рендерингу можно формализовать в виде предиката перехвата обновления, инициированного изменением `props`:

$$render(C, prevProps, nextProps) = \begin{cases} true, & \text{если } eq(prevProps, nextProps) = false \\ skip, & \text{если } eq(prevProps, nextProps) = true \end{cases}$$

В данной модели $C$ — целевой компонент, а функция $eq$ по умолчанию представляет собой конъюнкцию проверок `Object.is(prevProps[k], nextProps[k])` для каждого ключа $k \in props$. В случае использования пользовательского компаратора, функция $eq$ полностью замещается реализацией `arePropsEqual`. Данная формальная модель применима исключительно к векторам обновления, направленным сверху вниз (через `props`); векторы обновления изнутри (локальный `state`) или сбоку (внедрённый `context`) не маршрутизируются через предикат $eq$ и не отменяются им.

Целесообразность применения (Utility) `React.memo` выражается как функция двух переменных: частоты инициируемых родительских рендеров при неизменных `props` ($F$) и вычислительной стоимости самого рендера дочернего компонента ($C_r$), сопоставленной со стоимостью выполнения функции сравнения ($C_{eq}$). Оптимизация имеет математический смысл только при выполнении условия $C_{eq} \ll C_r \times F$. Если компонент рендерится редко или обладает тривиальной вычислительной сложностью, добавление проверки равенства ($C_{eq}$) может привести к чистой деградации производительности.

# 6. Современное значение и интерпретации

В современной инженерной практике React мемоизация через `React.memo` интерпретируется как локальная декларация чистоты рендеринга относительно входящих свойств. Разработчик эксплицитно сигнализирует планировщику React, что при эквивалентных входных векторах функция компонента гарантированно вернёт изоморфное представление и, следовательно, фаза её выполнения может быть безопасно пропущена. Инструмент не рассматривается как «ускоритель по умолчанию», а служит механизмом явного управления границами каскадных обновлений в топологии дерева.

Практическая интерпретация сводится к тому, что внедрение `React.memo` наиболее оправдано на топологически стабильных и ресурсоёмких поддеревьях: объемных списках данных, комплексных графиках, формах с глубокой вложенностью элементов управления, а также на узлах, которые регулярно подвергаются воздействию шумовых обновлений родителя (например, из-за анимаций или изменения несвязанного локального состояния). Важно подчеркнуть, что мемоизация не устраняет корневую причину генерации нестабильных ссылок; она лишь утилизирует уже существующую стабильность входных контрактов.

С учетом внедрения React Compiler, интерпретация инструмента смещается от стандарта архитектуры к средству профилированной отладки. В кодовых базах с функционирующим компилятором ручное применение `React.memo` становится маргинальным и требует обоснования через метрики профилирования.

# 7. Практические применения

- **Изоляция от локального шума родителя**: Наиболее частый и эффективный паттерн. Родительский узел содержит высокочастотное состояние (например, координаты курсора или ввод текста), обновление которого не влияет на данные, требуемые части дочерних компонентов. Мемоизация таких потомков прерывает каскад абстрактных вычислений и перестроения виртуального DOM для нерелевантных ветвей.
- **Стабилизация комплексных графов передачи данных**: Если ресурсоёмкий дочерний компонент принимает массивы, сложные конфигурационные объекты или функции обратного вызова, применение `React.memo` является лишь первой фазой. Для достижения фактического пропуска рендеров требуется интеграция с паттернами стабилизации на стороне родителя: обёртывание структур данных в `useMemo` и замыканий в `useCallback`, что обеспечивает сохранение идентичности ссылок между циклами.
- **Пользовательская редукция сложности сравнения**: Применение пользовательского предиката `arePropsEqual` целесообразно в сценариях, где компонент принимает избыточно сложные, глубоко вложенные структуры данных, из которых фактически используется лишь малая часть скалярных значений. Разработчик конструирует компаратор, проверяющий исключительно релевантные конечные узлы данных, что обходится значительно дешевле, чем генерация и согласование целого VDOM-дерева. Однако эффективность такого подхода всегда должна верифицироваться профилированием.

# 8. Примеры, случаи и следствия

## Пример 1. Полезная мемоизация дочернего компонента

В данном сценарии родительский компонент обновляет локальное состояние `theme`, что вызывает его собственный рендер. Однако дочерний узел `HeavyList` получает статичный массив строк. Применение `React.memo` гарантирует, что переключение темы не вызовет перестроение ресурсоёмкого списка.

```javascript
import { memo, useState } from 'react';

const HeavyList = memo(function HeavyList({ items }: { items: string[] }) {
  console.log('HeavyList render');
  return (
    <ul>
      {items.map((item) => <li key={item}>{item}</li>)}
    </ul>
  );
});

export default function Page() {
  const [theme, setTheme] = useState('light');
  const items = ['A', 'B', 'C']; // Ссылка стабильна, так как массив не пересоздается внутри рендера (или должен быть вынесен вне компонента/стабилизирован useMemo)

  return (
    <div className={theme}>
      <button onClick={() => setTheme(t => t === 'light' ? 'dark' : 'light')}>
        toggle theme
      </button>
      <HeavyList items={items} />
    </div>
  );
}
```

_Следствие:_ Оптимизация работает корректно при условии, что `items` не генерируется динамически на каждом рендере родителя. Документация неразрывно связывает эффективность мемоизации с обеспечением стабильности ссылочных свойств.

## Пример 2. Случай, где React.memo не даёт пользы (Пессимизация)

Родительский компонент при каждом обновлении счётчика конструирует новый объект `options` inline. Для встроенного компаратора `Object.is` это означает поступление новой ссылки на каждом такте, что немедленно инвалидирует кеш мемоизации.

```javascript
import { memo, useState } from 'react';

const Panel = memo(function Panel({ options }: { options: { dense: boolean } }) {
  console.log('Panel render');
  return <section>{options.dense ? 'dense' : 'normal'}</section>;
});

export default function Page() {
  const [count, setCount] = useState(0);

  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>{count}</button>
      <Panel options={{ dense: true }} /> 
      {/* Объект { dense: true } создается заново при каждом клике */}
    </>
  );
}
```

_Следствие:_ Разработчик формирует ложное ожидание оптимизации, опираясь на семантическую идентичность данных, игнорируя ссылочную природу JavaScript. В результате система выполняет бесполезную работу по сравнению объектов и всё равно инициирует рендер дочернего компонента.

## Пример 3. Случай, где мемоизация не изолирует от изменений context

Компонент `Greeting` обернут в `React.memo` и получает неизменное свойство `name`. Тем не менее, он является прямым подписчиком `ThemeContext`.

```javascript
import { createContext, memo, useContext, useState } from 'react';

const ThemeContext = createContext('light');

const Greeting = memo(function Greeting({ name }: { name: string }) {
  const theme = useContext(ThemeContext);
  console.log('Greeting render');
  return <h3 className={theme}>Hello, {name}</h3>;
});

export default function App() {
  const [theme, setTheme] = useState('light');

  return (
    <ThemeContext.Provider value={theme}>
      <button onClick={() => setTheme(t => t === 'light' ? 'dark' : 'light')}>
        toggle
      </button>
      <Greeting name="Ada" />
    </ThemeContext.Provider>
  );
}
```

_Следствие:_ Обновление значения провайдера приведет к рендеру `Greeting`, несмотря на стабильность входных `props`. Это демонстрирует принципиальное ограничение: мемоизация выступает фильтром исключительно для нисходящих свойств потока данных, но не обладает властью над механизмом доставки контекста.

# 9. Типичные ошибки, заблуждения и ограничения

- **Миф об «ускорении по умолчанию»**: Распространенное заблуждение о том, что обертывание всех компонентов в `memo` автоматически улучшает производительность приложения. В реальности `React.memo` эффективен исключительно при высокой частоте коллизий одинаковых `props` и значительной стоимости рендера самого узла. Бессистемное применение добавляет издержки на выполнение компаратора и усложняет чтение кода без профилируемой пользы.
- **Категориальное смешение с useMemo**: Заблуждение о функциональной идентичности интерфейсов. `useMemo` оперирует на микроуровне, кешируя конкретные результаты ресурсоёмких вычислений _между_ проходами рендера одного компонента. `React.memo` оперирует на макроуровне модели, управляя _фактом допуска_ дочернего компонента к циклу обновления, инициированного родителем.
- **Передача нестабильных ссылочных зависимостей**: Использование inline-объектов, литералов массивов или стрелочных функций в JSX родителя при передаче свойств в мемоизированный компонент. Данная конфигурация полностью нивелирует работу `React.memo`, так как новые ссылки заставляют алгоритм считать `props` изменившимися. Создается опасная иллюзия архитектурной оптимизации.
- **Некорректная реализация arePropsEqual (Stale Closures)**: Разработка кастомного компаратора, игнорирующего сравнение передаваемых функций-обработчиков. Если предикат возвращает `true` (допуская пропуск рендера), в то время как ссылка на обработчик обновилась, компонент продолжит оперировать устаревшей функцией. При вызове этот обработчик обратится к замыканию предыдущего цикла рендеринга родителя, работая с устаревшими (stale) `props` или `state`, что приводит к неконсистентному поведению и трудноуловимым багам.
- **Накладные расходы компаратора (Comparator Cost)**: Ограничение, связанное с тем, что сама операция сравнения требует процессорного времени. Для легковесных компонентов (UI-примитивы, текстовые блоки) затраты на итерацию по ключам объекта `props` и их сопоставление могут превосходить стоимость генерации тривиального VDOM-узла. Решение о применении мемоизации корректно принимать исключительно на основе данных React DevTools Profiler, а не интуитивных предположений.

# 10. Связанные понятия

- `[[React Rendering Model]]`: фундаментальная архитектурная рамка, определяющая правила обхода дерева; `React.memo` выступает средством её локальной модификации.
- `[[Functional Components]]`: базовая структурная единица декларативного UI, к которой применяется оптимизация через обёртку HOC.
- `[[Props in React]]`: механизм нисходящей передачи данных, стабильность ссылок и структура которых являются критическим фактором для работоспособности мемоизации.
- `[[React Component Re-renders]]`: смежный концепт, описывающий полный процесс перестроения узла; `React.memo` формирует фильтр, допускающий пропуск данного процесса.
- `[[useMemo]]`: семантически иной инструмент, применяемый для сохранения ссылочной идентичности вычисленных значений, часто используемый в качестве подготовительного этапа для корректной передачи `props` в мемоизированный дочерний компонент.

---

> **Концептуальное резюме:**
> 
> `React.memo` представляет собой специализированный механизм мемоизации архитектурной границы компонента, не являясь средством кеширования внутренних вычислений или гарантом иммутабельности рендера в условиях контекстных обновлений. Внедрение инструмента оправдано в строго определенных сценариях: при необходимости блокировки каскадных рендеров ресурсоёмких дочерних узлов в ответ на высокочастотные локальные изменения родительского состояния, при условии обеспечения абсолютной ссылочной стабильности входящих `props`. В ситуациях деградации входных ссылок, тривиальности стоимости рендера целевого узла или превосходства вычислительной сложности компаратора над генерацией узла VDOM, применение `React.memo` является нецелесообразным, либо приводит к прямой архитектурной пессимизации.
