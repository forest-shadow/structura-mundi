# 1. useMemo

> `useMemo` — это встроенный hook библиотеки React, предназначенный для локальной мемоизации результата детерминированного вычисления функции (выполнение тяжелых вычислений) в пределах конкретного экземпляра функционального компонента. 
> * Его базовая механика заключается в повторном использовании ранее полученного значения между циклами рендеринга до тех пор, пока не изменятся задекларированные входные зависимости.

В строгом инженерном смысле `useMemo` представляет собой механизм локального кэширования производных значений (derived values) на фазе рендера. Назначение данного инструмента состоит не в автоматическом ускорении компонента как такового, а в контролируемом предотвращении избыточных перерасчётов и сохранении стабильности производных ссылочных значений в тех узлах, где это критически влияет на вычислительную стоимость рендера или на поведение нисходящих (downstream) оптимизаций.

Хук не является средством выполнения побочных эффектов (side effects), не заменяет архитектурного упрощения кодовой базы и категорически не должен рассматриваться как универсальный или обязательный инструмент оптимизации по умолчанию.

# 2. Этимология и происхождение термина

Термин «memoization» (мемоизация) восходит к латинскому корню, родственному английскому слову «memo» (запись для запоминания). В Computer Science этот термин обозначает технику оптимизации программ, при которой сохраняются результаты выполнения функции для их переиспользования при поступлении аналогичных входных данных.

В контексте экосистемы React номенклатура `useMemo` буквально означает «использовать мемоизированное значение». Семантический акцент здесь ставится на результирующем значении, а не на императивном действии: хук сохраняет и предоставляет уже вычисленный результат, абстрагируя компонент от необходимости повторного инициирования самого процесса его вычисления.

# 3. Историческое развитие и эволюция концепта

Появление `useMemo` (в составе релиза React 16.8) неразрывно связано с переходом архитектуры библиотеки от классовой модели к парадигме функциональных компонентов.

В классической объектно-ориентированной модели React-компонентов задачи оптимизации производительности решались преимущественно на уровне экземпляра класса. Разработчики контролировали повторные рендеры посредством методов жизненного цикла, таких как `shouldComponentUpdate`, наследования от `React.PureComponent` и ручного кэширования вычислений в свойствах класса (instance properties).

С массовым распространением хуков возникла объективная необходимость выразить аналогичные оптимизационные намерения в чисто функциональной парадигме. В данной модели компонент переисполняется как математическая функция при каждом рендере. Как следствие, любые локальные вычисления внутри тела компонента по умолчанию инициализируются заново. `useMemo` был спроектирован как средство локального контроля над вычислительной стоимостью этих переисполнений.

По мере развития архитектурных паттернов концептуальное восприятие хука эволюционировало. В ранней практике сообщества его нередко применяли избыточно, используя как универсальное средство защиты от лишних операций. Современное академическое и инженерное понимание значительно строже: применение `useMemo` оправдано исключительно в сценариях, где наличествует либо математически дорогая операция, либо критическая необходимость стабилизировать производное значение ради корректной работы других механизмов мемоизации.

# 4. Формальные свойства, структура и ключевые признаки

Сигнатура хука концептуально выражается следующим образом:

```typescript
const memoizedValue = useMemo(calculateValue, dependencies)
```

Где:
- `calculateValue`: чистая функция (pure function), не содержащая побочных эффектов, инкапсулирующая логику вычисления производного значения.
- `dependencies`: массив зависимостей (входных переменных), на основании поверхностного сравнения которых React принимает решение об инвалидации кэша.
- `memoizedValue`: закэшированный результат последнего релевантного исполнения функции.

## Ключевые инварианты и свойства хука:

- <b>Локальность области действия:</b> Кэш существует исключительно внутри конкретного экземпляра компонента (привязан к узлу Fiber). Он не обладает глобальной областью видимости, не разделяется между параллельными экземплярами одного компонента и перманентно уничтожается при размонтировании (unmounting) компонента.
- <b>Привязка к фазе рендеринга:</b> Выполнение функции `calculateValue` происходит строго синхронно в процессе рендера (render phase). Это означает, что вычисление обязано быть детерминированным и полностью свободным от мутаций внешнего состояния.
- <b>Детерминированная зависимость от массива:</b> Пересчёт инициируется только тогда, когда алгоритм React (`Object.is`) регистрирует изменение хотя бы одной зависимости. Массив `dependencies` является не «перечнем желаемых триггеров», а строгой декларацией всех реактивных значений, от которых логически зависит итоговый результат.
- <b>Кэширование значения, а не подавление рендера:</b> `useMemo` не обладает способностью предотвращать рендеринг самого компонента. Хук предоставляет ранее вычисленный результат в ходе переисполнения компонента, но не отменяет сам факт переисполнения.
- <b>Статус производного состояния (Derived State):</b> Хук хранит не первичное состояние предметной области (source of truth), а производные данные, синтезированные из доступных входов (`props`, `state`). Архитектурно он принадлежит слою derived data, а не state management.
- <b>Отсутствие семантической гарантии персистентности:</b> Архитектурная корректность программы не должна опираться на то, сохранит ли внутренний планировщик React закэшированное значение в любой мыслимой ситуации. Спецификация допускает сброс кэша для высвобождения памяти (например, для скрытых элементов экрана). Бизнес-логика обязана оставаться функциональной и без этой оптимизации.

# 5. Современное значение и интерпретации

В современной практике разработки пользовательских интерфейсов `useMemo` классифицируется как инструмент избирательной оптимизации. Его ценность определяется соотношением вычислительной стоимости операции, частотой ререндеров и выгодой от сохранения стабильного результата.

<font color="#d3869b">Инженерно значимы две фундаментальные интерпретации:</font>
- <b>Контроль стоимости вычислений:</b> Применение целесообразно, если компонент на каждом такте рендеринга выполняет нетривиальные трансформации (агрегация, сортировка, нормализация структур данных, построение инвертированных индексов), накладные расходы на которые способны снизить кадровую частоту (FPS).
- <b>Стабилизация производных ссылочных значений (Referential Equality):</b> В JavaScript литеральное создание объектов или массивов в теле функции генерирует новую область памяти (идентичность ссылки) при каждом вызове. Это разрушает оптимизации в дочерних узлах (например, обёрнутых в `React.memo`), провоцируя каскадный рендеринг всего поддерева. В таких случаях хук используется ради обеспечения стабильности ссылки на производное значение (derived value) для нисходящих узлов.

<font color="#d3869b">Практические критерии уместности (оптимизация оправдана при наличии одного из факторов):</font>
- Вычисление объективно нетривиально по времени или объёму памяти.
- Результатом является структура ссылочного типа, чья нестабильность ломает механизмы мемоизации или вызывает побочные эффекты (`useEffect`) ниже по дереву.
- Инструментальное профилирование рендеринга подтверждает наличие измеримого узкого места (bottleneck) именно на данном участке.

# 6. Математическое или логико-формальное ядро

Концептуальную модель работы можно выразить через формальное частичное кэширование чистой функции.

Пусть имеется детерминированное вычисление производного значения $v$, зависящее от набора параметров:

$$v = f(d_1, d_2, \dots, d_n)$$

где $d_1, d_2, \dots, d_n$ — компоненты вектора зависимостей, а $f$ — чистая функция преобразования.

В контексте жизненного цикла экземпляра компонента, на дискретном шаге рендеринга $t$, `useMemo` реализует следующую функцию:

$$v_t = \begin{cases} v_{t-1}, & \text{если } \forall i \in [1, n] : Object.is(d_i^{(t)}, d_i^{(t-1)}) \equiv true \\ f(d_1^{(t)}, d_2^{(t)}, \dots, d_n^{(t)}), & \text{иначе} \end{cases}$$

Данная модель формализует, что планировщик React осуществляет возврат сохраненного значения $v_{t-1}$ только при полном соблюдении предиката строгой эквивалентности для всех зависимостей вектора. В противном случае происходит повторная аппликация функции $f$.

# 7. Архитектура и сравнение с соседними понятиями

Для системного понимания `useMemo` необходимо дифференцировать его от смежных механизмов.

- Отличие от немемоизированного вычисления (в теле компонента): Обычная декларация переменных является режимом по умолчанию. Она лишена накладных расходов на инициализацию кэша и сверку массива зависимостей. `useMemo` добавляет управленческую сложность, поэтому он предпочтителен только тогда, когда цена потери ссылочной идентичности или стоимость пересчёта многократно превышает стоимость работы механизма хука.
- Отличие от `useEffect`: Данные хуки обслуживают разные фазы жизненного цикла. `useMemo` инкапсулирует чистые вычисления во время рендеринга. `useEffect` запускает синхронизацию с внешним миром (сеть, DOM API, подписки) асинхронно после коммита изменений. Подмена концептов недопустима: логика получения derived data не должна выноситься в `useEffect`, а эффекты не должны инкапсулироваться в `useMemo`.
- Отличие от `useRef`: `useRef` предоставляет персистентный, мутабельный контейнер (ссылку), сохраняемый между рендерами без алгоритма автоматического пересчета. Если задача состоит в сохранении стабильного императивного дескриптора, используется `useRef`. Если требуется реактивное обновление вычисленного значения при изменении входов, применяется `useMemo`.
- Отличие от `useCallback`: Архитектурно `useCallback` представляет собой синтаксическую абстракцию над `useMemo`, специализирующуюся на ссылках на функции. В то время как `useMemo` обычно кэширует производные данные (результат), `useCallback` мемоизирует саму функцию (поведение) для стабилизации callback-обработчиков.

# 8. Практические применения

Прикладное применение `useMemo` в архитектуре ПО концентрируется в трех основных паттернах.

## 8.1. Дорогостоящая фильтрация и сортировка данных

В сценариях обработки объемных коллекций, когда многократное применение алгоритмов сортировки ($O(n \log n)$) и фильтрации деградирует производительность основного потока.

```javascript
function ProductList({ products, query, sortBy }) {
  const visibleProducts = useMemo(() => {
    return products
      .filter(product => product.title.includes(query))
      .sort((a, b) => {
        if (sortBy === "price") return a.price - b.price;
        return a.title.localeCompare(b.title);
      });
  }, [products, query, sortBy]);

  return <List items={visibleProducts} />
}
```

Оптимизация релевантна из-за высокой вычислительной стоимости итераций по массиву.

## 8.2. Стабилизация объекта конфигурации для дочернего компонента

Случаи, когда родитель на каждом цикле конструирует объект пропсов, разрушая оптимизацию дочернего узла (`React.memo`), зависящего от ссылочной идентичности (referential equality).

```javascript
function Dashboard({ theme, locale, data }) {
  const chartOptions = useMemo(() => {
    return {
      theme,
      locale,
      showLegend: true,
      precision: 2,
    };
  }, [theme, locale]);

  return <Chart data={data} options={chartOptions} />
}
```

Хук гарантирует стабильность ссылки на структуру `chartOptions`, защищая компонент `<Chart />` от каскадного перерендеринга.

## 8.3. Нормализация или индексирование данных

Трансформация линейных списков в структуры с константным временем доступа ($O(1)$) для оптимизации нисходящей выборки элементов.

```javascript
function UserTable({ users }) {
  const usersById = useMemo(() => {
    const map = new Map();
    for (const user of users) {
      map.set(user.id, user);
    }
    return map;
  }, [users]);

  return <Table users={users} usersById={usersById} />
}
```

Вычисление индекса `usersById` мемоизируется во избежание тяжеловесного пересчета на каждом кадре.

# 9. Типичные ошибки, заблуждения, ограничения

- Использование хука «на всякий случай» (Just-in-case memoization): Механическое оборачивание тривиальных операций, что приводит к деградации производительности за счет аллокации памяти под замыкания и алгоритма сравнения зависимостей. Если вычисление элементарно, оптимизация наносит больший урон, чем её отсутствие.
- Помещение побочных эффектов внутрь хука: Мутация DOM, вызов сетевых запросов, использование таймеров или запись в хранилища внутри функции обратного вызова. Это является критическим нарушением инварианта чистоты функции рендера.
- Неполный или некорректный массив зависимостей: Исключение из массива `dependencies` реактивных переменных (props, state), фактически используемых в расчетах. Это порождает логическую ошибку «устаревших замыканий» (stale closures), при которой декларативная модель React нарушается.
- Подмена архитектурной проблемы микрооптимизацией: Использование хука в попытках устранить просадки FPS, вызванные неудачной декомпозицией, тяжелой иерархией дерева или ошибочным подъемом состояния. Инструмент маскирует симптомы, но не решает фундаментальных проблем архитектуры компонента.
- Попытка использовать хук как хранилище состояния: Применение примитива для хранения персистентных мутабельных значений с ожиданием, что кэш не будет очищен. Для долговременного стейт-менеджмента применяются хуки `useState` или `useReducer`.
- Ожидание строгой семантической гарантии: Проектирование систем в расчете на то, что кэш `useMemo` будет существовать неизменно при всех режимах рендеринга. Механизм является сугубо оптимизационной эвристикой, при отключении которой программа обязана продолжать корректную работу.
