# 1. useReducer

> `useReducer` — это встроенный hook библиотеки React, предназначенный для управления локальным состоянием компонента через явную модель переходов, задаваемую reducer-функцией.

В формальном смысле `useReducer` организует состояние как кортеж: текущее значение состояния и механизм диспетчеризации (`dispatch`), принимающий действие (`action`) и инициирующий вычисление следующего состояния на основе чистой функции перехода вида `(state, action) => newState`.

<font color="#d3869b">В инженерной практике данный паттерн применяется тогда, когда состояние компонента не сводится к нескольким независимым скалярным значениям, а образует связанную структуру с нетривиальными правилами изменения. Ключевая концепция заключается в архитектурном разделении: описание переходов состояния выносится в изолированную функцию, тогда как место вызова обновления оперирует не новым значением напрямую, а семантически осмысленным действием (намерением).</font>

# 2. Этимология и происхождение термина

Термин _reducer_ алгоритмически восходит к фундаментальной математической и программистской концепции _reduction_ (свертка) — сведению последовательности преобразований к накоплению нового результата из предыдущего состояния и текущего входного воздействия.

В экосистеме JavaScript семантика слова _reduce_ первично связана с функцией высшего порядка `Array.prototype.reduce`, где последовательность элементов итеративно сводится к единому накопленному значению. <font color="#d3869b">В архитектурных паттернах управления состоянием (в частности, в Flux-подобных архитектурах и Redux) термин закрепился как строгое обозначение чистой функции, получающей на вход предыдущее состояние и объект действия, и возвращающей следующее состояние.</font>

В контексте `useReducer` это первоначальное значение полностью сохраняется: редьюсер не исполняет побочных эффектов и не подает императивных команд системе, а исключительно вычисляет следующее детерминированное состояние из формально заданного входа.

### 3. Историческое развитие и эволюция концепта

Появление `useReducer` неразрывно связано с парадигмальным сдвигом в React — переходом от объектно-ориентированной модели классовых компонентов к функциональной парадигме, где состояние и хуки жизненного цикла стали выражаться через алгебраические эффекты (Hooks).

Исторически эволюция подхода прошла следующие этапы:
- **Локальная мутация**: Изначально React опирался на локальное состояние классовых компонентов и асинхронный метод `this.setState`.
- **Глобальная формализация**: По мере усложнения клиентских приложений получил широкое распространение подход с reducer-функциями, концептуализированный архитектурой Flux и популяризованный библиотекой Redux. Данный подход продемонстрировал критические преимущества явного описания переходов состояния: высокую предсказуемость, тестируемость, строгую декомпозицию логики и устойчивость архитектуры при масштабировании числа пользовательских сценариев.
- **Локализация паттерна**: С релизом React 16.8 (Hooks) ядро перенесло этот принцип непосредственно внутрь функционального компонента. `useReducer` стал нативным, локальным вариантом reducer-модели, устраняющим необходимость подключения внешнего глобального хранилища (store). Тем самым разработчики получили возможность применять единую концептуальную схему конечных автоматов как на макроуровне (архитектура приложения), так и на микроуровне (отдельный компонент).

# 4. Формальные свойства, структура и ключевые признаки

## Базовая сигнатура

Каноническая форма вызова хука в TypeScript/JavaScript имеет вид:

```TypeScript
const [state, dispatch] = useReducer(reducer, initialArg, init?)
```

Структура аргументов и возвращаемых значений:
- `state`: текущее представление локальной модели данных.
- `dispatch`: функция отправки (инициирования) действия.
- `reducer`: функция вычисления следующего состояния.
- `initialArg`: исходный аргумент инициализации.
- `init`: опциональная функция ленивой инициализации состояния.
    

## Ключевые признаки и инварианты

Корректная математическая модель `useReducer` требует соблюдения строгих инвариантов:
- **Косвенное обновление**: Изменение состояния выражается исключительно через диспетчеризацию действия, а не через прямую запись нового значения.
- **Централизация**: Переходы централизованы в единой функции, а не распределены по множеству разрозненных обработчиков событий (event handlers).
- **Изоляция**: Логика вычисления состояния полностью отделена от JSX-разметки, событийных колбэков и побочных эффектов.
- **Согласованность**: Один редьюсер атомарно и транзакционно обновляет несколько взаимосвязанных полей состояния.
- **Чистота функции**: Редьюсер обязан быть чистой функцией (pure function). Одинаковый вход всегда генерирует идентичный выход без обращения к внешнему контексту.
- **Иммутабельность**: Редьюсер не должен мутировать существующее состояние по ссылке.
- **Семантика намерений**: Действие должно описывать свершившееся событие или намерение, а не содержать императивную скрытую логику изменения вне редьюсера.
- **Детерминированность переходов**: Возвращаемое состояние должно на 100% определяться парой `(state, action)`.

## Инициализация

Механизм поддерживает два режима аллокации начального состояния:
- **Прямая инициализация**: состояние конструируется непосредственно из `initialArg` при первом рендере.
- **Ленивая инициализация**: функция `init(initialArg)` вычисляет начальное состояние ровно один раз при монтировании компонента. Данный паттерн критически важен, когда построение начального состояния сопряжено с тяжелыми вычислениями, парсингом, нормализацией входных DTO или восстановлением сложной локальной структуры из `localStorage`.

# 5. Современное значение и интерпретации

В современной инженерной практике `useReducer` перестал трактоваться как простая «альтернатива `useState` для больших объектов». Он выступает фундаментальным инструментом формализации предметной области на уровне UI. Выделяются три доминирующие интерпретации:

- **Первая интерпретация (Локальный state machine)**: Инструмент позволяет рассматривать React-компонент как систему дискретных состояний и переходов. Пользовательские взаимодействия, результаты I/O операций и внутренние таймеры выступают сигналами, которые через `dispatch` переводят систему по строго заданному графу переходов.
- **Вторая интерпретация (Архитектурная дисциплина)**: Редьюсер вводит жесткую структурную границу между бизнес-логикой переходов и слоем представления (View). Компонент лишается возможности хаотично мутировать разрозненные переменные; вместо этого он вынужден использовать унифицированный, типизированный словарь действий (Action Dictionary).
- **Третья интерпретация (Повышение тестируемости)**: Поскольку редьюсер является изолированной чистой функцией, его верификация не требует рендеринга компонента, мокирования DOM или среды React. Бизнес-логика проверяется стандартными unit-тестами как детерминированное математическое отображение входа в выход.

# 6. Математическое и логико-формальное ядро

Функционирование `useReducer` формализуется через строгую алгебраическую модель системы переходов (переходной системы).

Пусть заданы следующие множества:
- $S$ — множество всех допустимых состояний компонента.
- $A$ — множество всех допустимых действий (сигналов).
- $R$ — функция редьюсера, выступающая отображением декартова произведения множества состояний и действий на множество состояний:
    $$R: S \times A \rightarrow S$$

Для любого текущего состояния $s \in S$ применение действия $a \in A$ порождает следующее состояние $s'$:
$$s' = R(s, a)$$

При поступлении последовательности действий $a_1, a_2, \dots, a_n$, эволюция (траектория) состояния системы выражается как рекурсивная композиция функции переходов:

$$s_n = R( \dots R(R(s_0, a_1), a_2) \dots, a_n)$$

## Академическая ценность модели:

- Вычисление состояния строго рекуррентно — оно всегда дедуцируется из предыдущего снимка $s_{t-1}$, блокируя возможность произвольной внешней инъекции недопустимого состояния.
- В типизированных средах (TypeScript) множество действий $A$ моделируется через дискриминируемое объединение (Discriminated Union), а множество $S$ — как структурный тип. Это позволяет статическому анализатору доказывать полноту покрытия всех ветвей графа переходов (Exhaustive checking) на этапе компиляции.

# 7. Сравнение абстракций (useReducer vs useState)

Оба хука решают задачу управления локальным состоянием, однако оперируют на принципиально разных уровнях абстракции. Выбор детерминируется топологией данных, а не их объемом.

|**Критерий**|**useState**|**useReducer**|
|---|---|---|
|**Оптимальный контекст**|Тривиальные, локальные обновления. Независимые значения, простые флаги, строки ввода.|Связанные структуры, синхронные изменения множества полей, формализация графа переходов.|
|**Механизм обновления**|Прямая мутация указателя состояния или примитивное вычисление от предыдущего.|Семантически осмысленная отправка типизированного действия (Action).|
|**Архитектурная роль**|Непосредственное хранение атомарного значения в памяти дерева React.|Моделирование локального конечного автомата, изоляция бизнес-правил от UI-слоя.|

# 8. Практические применения (Сценарии)

# Сценарий 1: Форма с зависимыми полями и валидацией

При наличии статусов ошибок, флагов отправки и каскадных зависимостей, разрозненные вызовы `useState` приводят к фрагментации компонента и состоянию гонки (race conditions). Редьюсер централизует транзакцию.



```TypeScript
type State = {
  values: { email: string; password: string }
  errors: { email?: string; password?: string }
  isSubmitting: boolean
  isSubmitted: boolean
}

type Action =
  | { type: 'fieldChanged'; field: 'email' | 'password'; value: string }
  | { type: 'submitStarted' }
  | { type: 'submitSucceeded' }
  | { type: 'submitFailed'; errors: State['errors'] }
  | { type: 'reset' }

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'fieldChanged':
      return {
        ...state,
        values: { ...state.values, [action.field]: action.value },
        errors: { ...state.errors, [action.field]: undefined },
      }
    case 'submitStarted':
      return { ...state, isSubmitting: true }
    case 'submitSucceeded':
      return { ...state, isSubmitting: false, isSubmitted: true }
    case 'submitFailed':
      return { ...state, isSubmitting: false, errors: action.errors }
    case 'reset':
      return initialState
    default:
      return state
  }
}
```

## Сценарий 2: Локальный асинхронный запрос как конечный автомат

Сетевой запрос представляет собой микро-автомат с фазами: `idle`, `loading`, `success`, `error`. Использование редьюсера исключает возникновение невозможных состояний (например, флага `isLoading === true` при одновременном наличии `data` и `error`).


```TypeScript
type State =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: string[] }
  | { status: 'error'; message: string }

type Action =
  | { type: 'requestStarted' }
  | { type: 'requestSucceeded'; data: string[] }
  | { type: 'requestFailed'; message: string }
  | { type: 'reset' }

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'requestStarted':
      return { status: 'loading' }
    case 'requestSucceeded':
      return { status: 'success', data: action.data }
    case 'requestFailed':
      return { status: 'error', message: action.message }
    case 'reset':
      return { status: 'idle' }
    default:
      return state
  }
}
```

## Сценарий 3: Многошаговый интерфейс (Wizard)

Для пошаговых модальных окон или редакторов с фазами (черновик $\rightarrow$ подтверждение $\rightarrow$ сохранение) редьюсер контролирует строгий направленный граф переходов. Действия `nextStep`, `prevStep`, `draftChanged`, `confirmed` исключают пропуск этапов валидации и логически противоречивые состояния интерфейса.

# 9. Типичные ошибки, заблуждения и ограничения

## Типичные антипаттерны и ошибки:

- **Смешение логики и эффектов**: Внедрение сетевых запросов, логирования или таймеров в тело редьюсера. Редьюсер — это чистая математическая функция; сайд-эффекты делегируются в `useEffect` или обработчики событий.
- **Избыточная гранулярность действий**: Описание микро-изменений, из-за чего редьюсер теряет роль смыслового узла и становится механической `set`-прослойкой.
- **Мутация по ссылке**: Изменение вложенных объектов «по месту» (`state.user.name = 'John'`) ломает механизм согласования React (Reconciliation) и блокирует рендеринг компонента.
- **Неоправданное усложнение (Over-engineering)**: Применение `useReducer` к атомарным булевым флагам, где синтаксический оверхед не компенсируется архитектурной пользой.
- **Нестрогая модель (God Action)**: Использование универсальных антипаттернов вида `{ type: 'setEverything', payload: any }`, которые уничтожают выразительность и размывают границы инвариантов.

## Распространенные заблуждения:

- _«Используется только для больших объектов»_: На практике решающим фактором является алгебраическая сложность переходов, а не объем данных в байтах.
- _«Делает код Redux-подобным и автоматически чистым»_: Редьюсер полезен исключительно при наличии естественной дискретной логики и осмысленных действий. Искусственное внедрение паттерна только ухудшает читаемость.

## Системные ограничения:

- **Не заменяет серверное состояние**: Не обладает механизмами кэширования, дедупликации, фоновой синхронизации и инвалидации (задачи для React Query, SWR, Apollo).
- **Не является полноценной FSM-библиотекой**: Позволяет моделировать автоматы, но лишен встроенного декларативного синтаксиса для entry/exit actions, параллельных состояний и guards (в отличие от XState).
- **Локальность**: Не решает проблему Prop Drilling автоматически. Требует комбинации с Context API для распределения состояния по дереву компонентов.

### 12. Архитектурный вывод

В парадигме React хук `useReducer` следует классифицировать не как хранилище «сложных структур», а как строгий **механизм формализации локального состояния через явную систему переходов**.

Его фундаментальное архитектурное достоинство заключается в способности инвертировать контроль: отделение неизменяемых математических правил перехода состояния от императивных UI-обработчиков, инициирующих эти переходы. Следовательно, применение `useReducer` алгоритмически оправдано именно в тот момент, когда внутри компонента зарождается собственная связная бизнес-логика, требующая явности, строгой статической типизации, изолированного unit-тестирования и архитектурной устойчивости к масштабированию.
