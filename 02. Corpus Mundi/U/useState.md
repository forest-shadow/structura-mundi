# 1. useState

> `useState` — это встроенный хук (hook) библиотеки React, предназначенный для объявления, инициализации и обновления локального состояния внутри функционального компонента. На структурном уровне он предоставляет компоненту кортеж, состоящий из текущего значения состояния и функции-мутатора (сеттера) для его обновления. Вызов функции обновления детерминированно инициирует процесс повторного рендеринга (re-render) компонента.

В строгом инженерном смысле `useState` представляет собой базовый механизм описания изменяемой, но строго изолированной в рамках экземпляра компонента памяти, которая непосредственно участвует в вычислении пользовательского интерфейса. React трактует данное состояние исключительно как неотъемлемую часть декларативной модели рендера, а не как произвольное внешнее хранилище данных или глобальный контекст.

# 2. Этимология и происхождение термина

Номенклатура `useState` является составной и семантически обусловленной:
- Префикс `use`: выступает в роли стандартизированного маркера. Он сигнализирует компилятору и разработчику, что данная функция является хуком — специализированной функцией React, которая строго подчиняется правилам вызова хуков (Rules of Hooks) и интегрирована во внутренний цикл рендеринга.
- Компонент `State`: указывает на функциональное назначение. Он определяет сущность как данные (состояние), которые конкретный экземпляр компонента обязан сохранять («помнить») между последовательными циклами рендеринга и использовать в качестве базиса при формировании графа пользовательского интерфейса (UI).

# 3. Историческое развитие и эволюция концепта

Внедрение `useState` произошло в рамках глобального архитектурного обновления React 16.8 (релиз Hooks). Исторически это ознаменовало фундаментальный переход от объектно-ориентированной модели, в которой локальное состояние было эксклюзивно ассоциировано с классовыми компонентами (class components) через инстанс `this.state`, к функциональной парадигме. Новая модель позволила инкапсулировать состояние и жизненные циклы непосредственно внутри чистых функциональных компонентов без необходимости создания классов и управления контекстом выполнения (`this`).

В ходе дальнейшего развития и стабилизации экосистемы React, статус `useState` эволюционировал от «инновационной возможности» до стандартного, де-факто ультимативного паттерна для управления простым локальным состоянием. В современной парадигме документации React он классифицируется как базовый хук состояния (basic state hook). При этом сформировалось четкое разделение ответственности: `useState` применяется для изолированных локальных данных, более сложная графовая логика обновлений делегируется хуку `useReducer`, а императивная синхронизация с внешними системами — хуку `useEffect`.

# 4. Формальные свойства, структура и ключевые признаки

## 4.1. Сигнатура и контракт

Каноническая форма декларации:

```JavaScript
const [state, setState] = useState(initialState)
```

Контракт возвращаемого значения: хук возвращает массив, состоящий ровно из двух элементов.
1. `state` — первый элемент, репрезентует текущее состояние. 
2. `setState` — второй элемент, является функцией-сеттером, вызов которой регистрирует запрос на вычисление следующего состояния, тем самым ставя компонент в очередь на повторный рендер. При первичном рендере значение первого элемента строго эквивалентно аргументу `initialState`.

## 4.2. Семантика инициализации

>Аргумент `initialState` учитывается ядром React исключительно в момент первичного монтирования компонента. При всех последующих циклах рендеринга данный аргумент игнорируется.

> <font color="#fb4934">Ленивая инициализация (Lazy Initialization):</font> в случае передачи функции в качестве аргумента `useState`, React классифицирует её как функцию-инициализатор (initializer function) и гарантирует её выполнение только единожды — на этапе конструирования начального состояния. Данный паттерн критически важен для предотвращения избыточных ресурсоемких вычислений при каждом рендере.


> [!important] **Важное поведение среды разработки:**
> В режиме `Strict Mode` React намеренно вызывает функции-инициализаторы дважды. Это диагностическая мера, направленная на выявление нечистых (impure) вычислений и скрытых побочных эффектов. На поведение в production-сборке это не влияет.

## 4.3. Семантика обновления и батчинг

<font color="#fb4934">Функция `setState` полиморфна и поддерживает два типа аргументов:</font>
- Абсолютное значение: готовое к применению следующее состояние.
- Функция обновления (updater function): коллбэк вида `prev => next`.

<font color="#d3869b">При использовании updater function React помещает её во внутреннюю очередь абстрактного узла (Fiber). На этапе следующего рендеринга ядро последовательно (коммутативно) применяет все накопленные в очереди функции к значению предыдущего состояния. Это делает функциональное обновление единственно корректным выбором в сценариях, где новое значение вычисляется на основе предыдущего.</font>

<font color="#fb4934">Оптимизация производительности (Batching)</font>: React группирует множественные вызовы обновлений состояния. Вычисление нового дерева и обновление экрана происходят только после полного завершения всех синхронных обработчиков событий (а в React 18+ и асинхронных), а не после каждой индивидуальной инструкции `setState`. Следовательно, n-ное количество обновлений в рамках одного макро- или микротаска сводится к единому циклу ререндера.

## 4.4. Состояние как снимок (Snapshot-семантика)

Фундаментальное свойство `useState` заключается в том, что переменная состояния ведет себя как иммутабельный снимок (snapshot) для текущего цикла выполнения функции компонента. Вызов `setState` не мутирует значение локальной переменной состояния в рамках уже выполняющегося лексического окружения обработчика; он исключительно формирует запрос к ядру React на инициацию следующего рендера с новым значением. Как следствие, любая операция чтения переменной состояния непосредственно после вызова `setState` всегда вернет прежнее значение, принадлежащее текущему замыканию.

## 4.5. Критерий эквивалентности и эвристика пропуска рендера

Если вычисленное новое значение признается эквивалентным текущему состоянию на основе алгоритма жесткого сравнения `Object.is`, React активирует оптимизационный механизм и прерывает цепочку обновлений, пропуская рендер текущего компонента и всего его поддерева.

Следствие: императивная мутация существующего объекта или массива (например, `array.push()`) с последующей передачей той же ссылки в `setState` не приведет к перерасчету интерфейса, так как ссылочная идентичность останется неизменной.

## 4.6. Топологическая привязка состояния

<font color="#d3869b">Память, выделенная через `useState`, не привязана к функции компонента как к объекту JavaScript. React ассоциирует каждую единицу состояния со строгой позицией компонента в топологии дерева рендера (Fiber tree). Состояние сохраняется до тех пор, пока компонент идентичного типа находится в той же структурной позиции.</font>

<font color="#fb4934">Условия сброса состояния:</font>
- Удаление узла из дерева (unmounting).
- Замена типа компонента в текущей позиции.
- Изменение пропса `key`, который выступает явным идентификатором идентичности узла.

## 4.7. Ограничения вызова (Rules of Hooks)

Как архитектурная константа, `useState` может быть вызван исключительно на верхнем уровне функционального компонента или внутри пользовательского (custom) хука.


> [!important] Запрещено:
> размещение хука внутри условных операторов (`if`), циклов (`for`/`while`), вложенных локальных функций, асинхронных обработчиков событий или после оператора раннего возврата (`return`). Это строгое ограничение продиктовано внутренним устройством React, которое опирается на константный и предсказуемый индекс (порядок) вызова хуков при каждом рендере для корректной маршрутизации данных состояния.

# 5. Современное значение и интерпретации

В современном инженерном дискурсе React `useState` утвердился как безальтернативный стандарт для декларативного описания простого локального состояния, непосредственно детерминирующего интерактивность конкретного узла UI. 

Типичные домены применения включают: 
* текстовые значения полей ввода, 
* булевы переключатели, 
* индексы активных элементов (табы), 
* локальные стейт-машины режимов отображения, 
* шаги многостраничных форм (wizards), а также 
* дискретные статусы локальных сетевых транзакций.

<b>Современная интерпретация делает сильный акцент на строгом разделении границ ответственности:</b> `useState` легитимен исключительно для данных, прямо участвующих в формуле рендера `UI = f(state)`. В отличие от него, `useEffect` позиционируется как механизм побочных эффектов для синхронизации с внешними API (DOM, сеть, подписки), а не как инструмент конструирования реактивных потоков данных внутри компонента. Если внешней системы не существует, применение `useEffect` для трансформации одного состояния в другое считается антипаттерном.

# 6. Математическое или логико-формальное ядро

Концептуальную архитектуру `useState` можно формализовать как математическое отображение, описывающее пользовательский контракт хука.

Пусть $S$ — множество всех допустимых значений состояния компонента.

Тогда сигнатуру возвращаемого значения `useState` можно выразить как:

$$useState : S \to (S, U)$$

Где $U$ — полиморфная функция обновления состояния, принимающая либо абсолютное значение из множества $S$, либо функцию преобразования $S \to S$.

Динамику изменения состояния во времени (в разрезе последовательности циклов рендеринга) можно описать рекуррентным соотношением. Пусть $n$ — индекс рендера, $s_n$ — значение состояния на $n$-ом рендере, а $q_n$ — упорядоченная очередь обновлений (updater functions), зарегистрированных между рендерами $n$ и $n+1$.

Инициализация:

$$s_0 = initialState$$

Функция перехода к следующему рендеру:

$$s_{n+1} = applyQueue(s_n, q_n)$$

Где $applyQueue$ последовательно применяет функции из $q_n$ к базовому значению $s_n$. Именно эта формализация математически обосновывает семантическую разницу между параллельными вызовами абсолютного обновления `setCount(count + 1)` (где базовое значение константно) и функционального обновления $setCount(prev \to prev + 1)$ (где каждая функция в очереди оперирует результатом предыдущей вычисления).

# 7. Практические применения

Ниже представлены академические примеры реализации фундаментальных паттернов использования локального состояния.

## 7.1. Управление значением поля ввода (Controlled Input)

Классический паттерн «управляемого компонента», в котором значение элемента ввода формы полностью делегировано и синхронизировано с локальным состоянием React. Компонент выступает единым источником истины (single source of truth).

```TypeScript
import { useState } from 'react';

export function SearchBox() {
  const [query, setQuery] = useState('');

  return (
    <input
      value={query}
      onChange={(e) => setQuery(e.target.value)}
      placeholder="Search..."
    />
  );
}
```

Данный сценарий репрезентует базовую онтологическую роль `useState`: компонент инкапсулирует скалярное локальное значение, мутация которого линейно и предсказуемо обновляет отображение.

## 7.2. Переключение локального режима интерфейса (UI Toggles)

Реализация бинарных состояний пользовательского интерфейса: контроль видимости модальных окон, аккордеонов, выпадающих списков.

```TypeScript
import { useState } from 'react';

export function Details() {
  const [expanded, setExpanded] = useState(false);

  return (
    <section>
      <button onClick={() => setExpanded(v => !v)}>
        {expanded ? 'Hide details' : 'Show details'}
      </button>

      {expanded && <p>Additional information</p>}
    </section>
  );
}
```

Состояние в данном случае изолировано, логика переходов тривиальна (инверсия булева значения) и максимально естественно выражается через встроенный сеттер с применением функционального обновления для гарантии корректности переключения.

## 7.3. Представление статуса локальной операции (State Machine)

Моделирование конечного автомата (finite state machine) для UI-компонента, отражающего фазы асинхронной транзакции (idle, submitting, success, error).

```TypeScript
import { useState } from 'react';

type Status = 'idle' | 'submitting' | 'success' | 'error';

export function LocalForm() {
  const [status, setStatus] = useState<Status>('idle');

  async function handleSubmit() {
    try {
      setStatus('submitting');
      await Promise.resolve(); // Эмуляция сетевого запроса
      setStatus('success');
    } catch {
      setStatus('error');
    }
  }

  return (
    <div>
      <button onClick={handleSubmit} disabled={status === 'submitting'}>
        Submit
      </button>
      <p>Status: {status}</p>
    </div>
  );
}
```

Этот паттерн демонстрирует масштабируемость хука: `useState` способен эффективно управлять не только примитивами, но и категориальными дискретными значениями, жестко детерминирующими режим работы интерфейса и блокирующими невалидные пользовательские действия (например, повторную отправку формы).

# 8. Границы применимости: useState против useReducer

Инженерная граница между `useState` и `useReducer` пролегает не в плоскости объема хранимых данных, а в плоскости цикломатической сложности переходов и необходимости централизованного управления бизнес-логикой (state logic).

- Прерогатива `useState`: состояние атомарно, графы переходов тривиальны, обновление выражается прямой безусловной установкой нового значения.
- Прерогатива `useReducer`:
    - Состояние представляет собой комплексный объект с множеством взаимосвязанных узлов, где изменение одного поля требует пересчета других.
    - Происходит перегрузка обработчиков событий (event handlers) множественными, императивными вызовами `setState`.
    - Возникает архитектурная потребность в формализации транзакций в парадигме `action -> newState`.
    - Требуется строгая локализация и инкапсуляция логики мутации данных за пределами JSX-ориентированной разметки компонента для повышения тестируемости.

# 9. Типичные ошибки, заблуждения, ограничения

Функциональная модель `useState` порождает ряд специфических когнитивных искажений и инженерных ошибок:
- **Иллюзия синхронной мутации (Ошибка немедленного чтения):** Попытка чтения переменной состояния в следующей строке кода после вызова `setState` с ожиданием получения обновленного значения. Данное действие игнорирует snapshot-семантику: обработчик продолжает оперировать значением из замыкания текущего рендера.
- **Игнорирование ссылочной идентичности (Прямая мутация):** Конструкции вида `form.name = 'x'; setForm(form)`. React отбрасывает подобное обновление интерфейса, поскольку эвристика `Object.is` определяет идентичность старой и "новой" ссылки на объект/массив. Требуется явное конструирование нового объекта (например, через spread-оператор `...`).
- **Потеря обновлений в асинхронных замыканиях (Stale Closure):** Многократный последовательный вызов `setCount(count + 1)` внутри единого обработчика опирается на устаревшее значение `count`. Корректное накопление транзакций требует использования updater function: `setCount(prev => prev + 1)`.
- **Десинхронизация графа вызовов (Нарушение правил хуков):** Условный вызов `useState` (внутри `if` или циклов) фатально разрушает индексный порядок хуков, что приводит к критической рассинхронизации памяти React и неконтролируемым багам в UI.
- **Петля рендеринга (Too many re-renders):** Безусловный вызов функции `setState` непосредственно в теле функционального компонента (вне хука эффекта или обработчика событий). Это мгновенно инициирует новый рендер, который снова вызывает `setState`, создавая бесконечный цикл, приводящий к крашу приложения.
- **Ошибочная интерпретация функций как значений:** При необходимости сохранить саму функцию в состоянии, прямая передача `useState(myFunc)` приведет к ее выполнению ядром React в качестве инициализатора. Корректный паттерн требует двойного функционального оборачивания: `useState(() => myFunc)` и `setFunc(() => newFunc)`.

## 10. Связанные понятия

Родительская тема:
[[React Hooks]] — фундаментальная архитектурная надстройка, базисом которой является `useState`.

Связанные заметки:
- `[[useReducer]]` — комплементарный хук состояния для инкапсуляции графовой логики переходов.
- `[[useEffect]]` — механизм императивной синхронизации побочных эффектов с внешними системами.
- `key (React)` — специальный пропс, манипулирующий идентичностью узла виртуального дерева и управляющий жизненным циклом (сбросом/сохранением) локального состояния.
