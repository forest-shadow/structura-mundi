# 1. Virtual DOM

> <font color="#fb4934">Virtual DOM (Виртуальный DOM)</font> — промежуточное, находящееся в оперативной памяти представление пользовательского интерфейса (tree data structure), с которым библиотека (например, React) работает до применения вычисленных изменений к реальному объектному модулю документа (DOM) браузера.

* **Объясняющая абстракция**: Virtual DOM следует понимать как концептуальную модель, описывающую тот факт, что React не изменяет браузерный DOM непосредственно в момент вызова компонента. Вместо этого система сначала формирует и сопоставляет внутреннее представление дерева UI до и после изменения, после чего строго вычисленные изменения применяются на фазе фиксации (commit).
* **Архитектурная граница**: В контексте React термин Virtual DOM не является исчерпывающим названием всей внутренней архитектуры рендеринга. Этот концепт критически полезен для объяснения различия между декларативным описанием интерфейса, промежуточным представлением и фактическими императивными DOM-операциями, однако он не покрывает в полном объёме современные механизмы планирования (scheduling), приоритизации задач обновления и внутренние структуры React (такие как Fiber).

# 2. Этимология и происхождение термина

**Происхождение компонентов термина**: Конструкт образован по аналогии с понятием _virtual_ (виртуальный) в значении «не совпадающий напрямую с физическим носителем, но представляющий его функциональность в другой форме» и _DOM_ (Document Object Model) — браузерной объектной моделью документа.

**Семантическое ядро**: Смысл термина состоит в том, что речь идёт не о настоящем физическом DOM-дереве, управляемом движком браузера, а о его концептуальном или программном аналоге. Это представление существует исключительно в памяти приложения и используется как вычислительный полигон для определения необходимых мутаций. Термин был популяризован инженерами компании Facebook (Meta) при публичном релизе библиотеки React в 2013 году для объяснения нового подхода к оптимизации ресурсоемких операций перерасчета макета (Reflow) и перерисовки (Repaint).

# 3. Историческое развитие и эволюция концепта

* **До появления концепта**: На ранних этапах веб-разработки обновление интерфейсов требовало ручных, точечных манипуляций с DOM (что приводило к сильной связности и запутанному императивному коду) или полной замены крупных HTML-узлов через шаблонизаторы (что вызывало деградацию производительности).
* **Популяризация через React**: Идея промежуточного представления интерфейса стала особенно известной благодаря React, который популяризировал декларативный подход к UI. Библиотека объясняла эффективность обновлений именно через сопоставление виртуального дерева с его предыдущим состоянием. На ранних этапах термин Virtual DOM использовался почти как центральное объяснение всей модели обновления интерфейса. Это было полезно педагогически: разработчику становилось ясно, что компонент не «лезет вручную в DOM», а возвращает статичное описание интерфейса.
* **Современный этап эволюции**: По мере развития экосистемы React данное объяснение стало недостаточным как полное инженерное описание. Внутренние механизмы библиотеки эволюционировали, включив более сложные структуры данных и этапы работы, в частности строгое разделение на _render phase_ (фаза вычисления) и _commit phase_ (фаза применения), а также переход на архитектуру _Fiber_. В современной инженерной трактовке Virtual DOM сохраняет свою ценность исключительно как вводная и объясняющая абстракция, а не как исчерпывающее название физической модели движка.

# 4. Формальные свойства, структура и ключевые признаки

* **Промежуточность**: Структура существует строго между декларативным описанием интерфейса и реальными мутациями браузерного DOM. Это не конечная среда отображения графики и не источник пользовательского ввода, а рабочий вычислительный слой, на котором алгоритмы React выполняют поиск различий.
* **Древовидная топология**: Поскольку пользовательский интерфейс естественно моделируется как направленный ациклический граф вложенных узлов, промежуточное представление также имеет иерархическую структуру. Узлы такого дерева соответствуют элементам интерфейса, компонентам или иным единицам представления.
* **Легковесность**: Виртуальные узлы (VNode) представляют собой простые структуры данных (Plain Old JavaScript Objects), содержащие только семантическую информацию (имя тега, атрибуты, ссылки на дочерние элементы). Они лишены тяжеловесных методов и свойств реальных DOM-узлов.
* **Декларативное происхождение**: Virtual DOM не пишется разработчиком вручную как набор инструкций модификации. Он детерминированно выводится из декларативного описания UI (JSX, вызовов `React.createElement`), инкапсулированной логики компонента и текущих значений свойств (`props`) и состояния (`state`).
* **Сопоставимость во времени**: Ключевая алгоритмическая ценность концепта состоит в том, что система может сохранять и сравнивать предыдущее и новое представления дерева. Это делает возможным _reconciliation_ — процесс вычисления дельт (различий) между состояниями интерфейса.
* **Независимость от непосредственных DOM-операций**: Изменение виртуального представления не означает автоматического или синхронного изменения браузерного DOM. Между вычислением нового дерева и реальным применением мутаций заложен барьер принятия решений о том, какие операции действительно необходимо передать в API браузера.
* **Неравенство реальному DOM**: Промежуточное представление не является точной побайтовой или структурной копией браузерного DOM. Во-первых, его узлы не обязаны совпадать «один к одному» с объектами среды (существуют концептуальные компоненты, фрагменты). Во-вторых, абстракция не привязана к вебу: React работает с различными целевыми средами (React Native, React 3D), поэтому Virtual DOM относится к фундаментальному уровню абстракции UI, а не к спецификации W3C.

# 5. Уровни абстракции и место в модели React

Для корректного системного понимания темы необходимо строго различать четыре уровня абстракции:

* **Уровень 1. Декларативное описание интерфейса**: Компонент (как чистая функция) возвращает описание того, каким интерфейс должен быть при заданных входных условиях. На этом этапе нет ни DOM-операций, ни инстанцирования узлов документа.
* **Уровень 2. Внутреннее представление дерева**: Ядро React преобразует полученное декларативное описание во внутреннюю графовую структуру, пригодную для сохранения в памяти, обхода и сопоставления. Именно на этом уровне локализуется абстракция, традиционно называемая Virtual DOM.
* **Уровень 3. Согласование (Reconciliation)**: Процесс алгоритмического сопоставления нового и предыдущего внутренних представлений. Система определяет, какие топологические части дерева нужно сохранить, обновить, удалить или создать заново. Virtual DOM не тождественен процессу согласования: он является _структурой данных_, над которой выполняется алгоритм согласования.
* **Уровень 4. Фиксация (Commit) и побочные эффекты**: После вычисления минимального графа различий React применяет мутации к целевой среде (браузерному DOM). Исключительно на этом этапе происходят императивные вызовы: изменение атрибутов, вставка элементов, удаление узлов дерева или замена текстовых констант. Virtual DOM выступает гарантом того, что этот этап будет максимально компактным и быстрым.

# 6. Современное значение и интерпретации

В современной академической и инженерной практике парадигма Virtual DOM интерпретируется в трех основных плоскостях:

* **Педагогическое значение**: Концепт служит ментальной моделью, помогающей объяснить, почему архитектура позволяет описывать интерфейс декларативно (перерисовывая всё приложение концептуально), избегая при этом ручного императивного управления каждым узлом.
* **Архитектурное значение**: Термин фиксирует фундаментальный паттерн проектирования: наличие промежуточного вычислительного буфера (прослойки) между пользовательской бизнес-логикой компонента и ресурсоемкими вызовами API целевой платформы рендеринга.
* **Ограниченно-историческое значение**: Отражает ранний (до React 16) способ описания внутренних механизмов библиотеки. Сегодня термин признается недостаточным для глубокого понимания современной rendering-модели, поскольку он семантически не охватывает асинхронное планирование задач (scheduling), прерываемый рендеринг (concurrent rendering) и структуру связанных списков Fiber.

# 7. Математическое и логико-формальное ядро

На формально-логическом уровне механику применения Virtual DOM можно представить как направленную конвейерную последовательность функциональных отображений:

$State \times Props \rightarrow UI\_Description \rightarrow Internal\_Tree \rightarrow Diff \rightarrow Host\_Operations$

**Декомпозиция формальной модели**:
* **Вычисление описания** ($State \times Props \rightarrow UI\_Description$): Компонент (как математическая функция) на основе текущего состояния и входных параметров детерминированно вычисляет декларативное описание желаемого интерфейса.
* **Трансляция структуры** ($UI\_Description \rightarrow Internal\_Tree$): Описание переводится движком во внутреннее графовое представление (Virtual DOM), пригодное для обхода по узлам.
* **Вычисление разницы** ($Internal\_Tree_{old}, Internal\_Tree_{new} \rightarrow Diff$): Эвристический алгоритм сравнивает старое и новое состояния представления. Классические алгоритмы сравнения деревьев имеют вычислительную сложность $O(n^3)$, однако React применяет эвристику (базирующуюся на типах узлов и ключах), снижающую асимптотическую сложность до линейной $O(n)$, выдавая набор необходимых различий (Diff).
* **Применение мутаций** ($Diff \rightarrow Host\_Operations$): Набор вычисленных различий компилируется в конечные императивные операции над целевой средой отображения.

# 8. Практические применения и архитектурные следствия

**Объяснение декларативного UI через JSX**: Понимание промежуточного представления критически важно для корректной интерпретации синтаксиса JSX. Конструкция JSX не инициирует немедленное создание DOM-элемента. Она означает команду «описать элемент интерфейса, который будет включён во внутреннее дерево представления».

```javascript
function Greeting({ name }: { name: string }) {
  return <h1>Hello, {name}</h1>;
}
```

В данном случае компонент не исполняет прямой вызов API наподобие `document.createElement("h1")`. Он возвращает легковесный объект-описание, который ядро встроит в Virtual DOM, самостоятельно приняв решение о моменте и способе физической материализации узла.

**Различение фаз вычисления (Render) и реального обновления**: Ментальная модель промежуточного слоя предотвращает ошибочное отождествление повторного вызова JavaScript-функции компонента с физической перерисовкой окна браузера.

```javascript
function Counter() {
  const [count, setCount] = React.useState(0);

  return (
    <div>
      <span>{count}</span>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}
```

При инициации изменения через `setCount`, система повторно вычисляет описание интерфейса с новым значением `count`. Затем это новое виртуальное представление сопоставляется с предыдущим. В результате реальное изменение в браузере сведется к обновлению исключительно одного текстового значения внутри тега `span`. Тег `div` и кнопка не будут пересозданы или затронуты императивными мутациями.

**Обеспечение минимальности DOM-операций**: Концепция объясняет, почему инструмент способен генерировать предельно узкий набор физических изменений, даже если визуально компонент «перерисовался» полностью.

```javascript
function UserCard({ name, city }: { name: string; city: string }) {
  return (
    <section>
      <h2>{name}</h2>
      <p>{city}</p>
    </section>
  );
}
```

Если в компонент поступит обновленный проп `city` (при неизменном `name`), выполнение компонента породит новое описание всего фрагмента целиком. Однако на этапе _reconciliation_ движок установит, что топология дерева (узлы `section` и `h2`) полностью сохранена, и сформирует точечную инструкцию обновления только для содержимого текстового узла внутри `p`.

# 9. Типичные ошибки, заблуждения и ограничения

* **Заблуждение о точной копии (Изоморфизм 1:1)**: Утверждение о том, что Virtual DOM является абсолютной и точной копией браузерного DOM, ошибочно. Это специализированное представление, агрегирующее только данные, необходимые для сравнения и рендеринга. Оно не дублирует внутренние свойства браузерного API и включает элементы, не имеющие физического воплощения в DOM (например, Context Providers, Fragments).
* **Заблуждение о масштабах сравнения**: Мнение, что React «постоянно сканирует и сравнивает весь реальный DOM», в корне неверно. Архитектура опирается исключительно на сравнение двух версий своего _внутреннего_ виртуального представления, изолируя вычислительную нагрузку от медленных вызовов браузера.
* **Заблуждение о «бесплатных» рендерах**: Промежуточное представление радикально снижает стоимость ручного управления макетом и минимизирует дорогие операции перекомпоновки браузера (Reflow). Однако сами процессы генерации новых объектов описания интерфейса (VNodes), обхода деревьев памяти и выполнения алгоритмов согласования потребляют процессорное время. Избыточные рендеры на уровне Virtual DOM способны вызвать деградацию производительности приложения.
* **Заблуждение об абсолютном превосходстве в скорости**: Популярный тезис «Virtual DOM всегда быстрее реального DOM» некорректен. Создание абстракций в памяти является накладным расходом (overhead). Эта парадигма работает быстрее _неоптимизированного_ и избыточного императивного кода, но при прочих равных всегда уступает идеально спроектированному точечному обновлению элементов напрямую (vanilla JS).
* **Ограниченность объясняющей модели**: Концепт объясняет базовую синхронную связь между состоянием и DOM, но пасует перед необходимостью описать конкурентный рендеринг (Concurrent Rendering), приоритизацию очередей задач и возможность приостановки вычислений.
* **Путаница со спецификацией Shadow DOM**: Shadow DOM представляет собой стандарт веб-компонентов W3C для инкапсуляции стилей и разметки на уровне браузера. Virtual DOM является программным паттерном на уровне JavaScript-библиотеки. Эти технологии не пересекаются в своем назначении.

# 10. Связанные понятия

* [[DOM]]: Реальная объектная модель документа среды исполнения браузера, являющаяся конечным приемником (target) вычисленных изменений.
* **[[React Render]]**: Строго детерминированный этап вычисления декларативного описания интерфейса на основе текущего состояния и входных свойств.
* **[[React Reconciliation]]**: Механизм алгоритмического сопоставления предыдущего и нового графов виртуального дерева для генерации оптимального набора обновлений.
* **[[01. Inbox/To Classify/React/React Rendering Model]]**: Макроархитектурная рамка библиотеки, в которую концепция Virtual DOM входит лишь как один из исторических и объясняющих компонентов.
* **[[React Elements]]**: Базовые единицы, представляющие собой иммутабельные объекты декларативного описания UI-узлов, из множества которых и формируется промежуточное представление.
* [[React Key Prop]]: Критически важный атрибут-идентификатор, управляющий эвристикой сопоставления элементов между фазами рендеринга и радикально влияющий на вычислительную сложность результата _reconciliation_.
* **[[Fiber]]**: Глубокая внутренняя архитектурная структура React (представляющая узлы как односвязные списки единиц работы), используемая в современных версиях движка вместо классического монолитного дерева Virtual DOM для поддержки прерываемого рендеринга.
* **[[Incremental DOM]]**: Конкурирующий архитектурный подход (применяемый, например, в Angular), избегающий создания промежуточного дерева в памяти путем компиляции шаблонов в инструкции для обновления DOM «по месту» (in-place).

# 11. Краткое итоговое положение

Virtual DOM — это не магический механизм «ускорения интерфейса» и не полное имя современной внутренней архитектуры фреймворков. Это концептуально полезное промежуточное представление пользовательского интерфейса в оперативной памяти, позволяющее разработчику понять, что ядро библиотеки сначала вычисляет и сопоставляет внутреннее дерево, а уже затем применяет минимально необходимые, алгоритмически выверенные изменения к реальному DOM. Именно в этом качестве, несмотря на значительную эволюцию внутренних структур библиотек, данный термин остается канонически важным для системного понимания процессов вычисления (render) и согласования (reconciliation) в современной UI-инженерии.
